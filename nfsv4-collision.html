NFSv4 · Protocol State · ONTAP

NFSv4 Collision
in NetApp Filers

NFSv4 introduced a stateful protocol model — clients hold leases, delegations, and open/lock state on the server. When that state conflicts between clients, or between client and server expectations, a collision occurs: stale filehandles, permission denials, hung mounts, or corrupted lock state. This tutorial covers every collision type, how to detect each one with ONTAP CLI, and exactly how to resolve them.

Section 01 · Foundation

NFSv4 State Model — Why Collisions Exist

NFSv4 is stateful — NFSv3 was not

NFSv3 was a stateless protocol. Every RPC carried all the information needed to service it independently. The server held no per-client state — if the server rebooted, clients simply retried their operations and continued as if nothing happened. This made NFSv3 very resilient but also limited: no true locking semantics, no caching guarantees, no delegation.

NFSv4 is fundamentally stateful. The server tracks per-client state including open files, byte-range locks, and delegations — all tied to a time-bounded lease. This enables richer semantics: true POSIX locking, client-side caching with consistency guarantees, and cross-server referrals. But it introduces a new class of problem: state collisions — situations where two clients disagree about who owns a lock, where a client's state has expired on the server, or where identity translation produces conflicting ownership.

Lease, Grace period, and why they matter for collisions

Every client that connects via NFSv4 is assigned a Client ID (via the SETCLIENTID / EXCHANGE_ID operation). All state (open files, locks, delegations) is associated with this Client ID and protected by a lease timer — 30 seconds by default in ONTAP.

The client must renew its lease before it expires, either via an explicit RENEW call or by any other NFSv4 operation. If the lease expires:

  • Server perspective: the client's state is considered abandoned. The server reclaims all locks and delegations held by that Client ID.
  • Client perspective: if it reconnects after lease expiry, all its stateids are stale. Any operation using an old stateid returns NFS4ERR_EXPIRED or NFS4ERR_BAD_STATEID.

The grace period (45 seconds by default) is a window after a server restart during which clients can reclaim their previous state before new conflicting state is granted. If a client misses the grace period window, it cannot reclaim its locks and must start fresh — potentially colliding with another client that was granted new state on the same file during that window.

🔑
Core principle Every NFSv4 collision — whether a domain mismatch, a delegation conflict, or a lock state error — ultimately comes down to one of two things: stale or conflicting state on the server, or identity translation failure between the client's numeric UID and the server's user string. Understanding these two root causes explains every symptom described in this tutorial.
Section 02 · Collision Types

The Six NFSv4 Collision Types

Identity

ID Domain Mismatch

Client and server use different NFSv4 ID mapping domains. Ownership appears as nobody or 65534. Files created on the client appear owned by an unknown user on the server — and vice versa.

Caching

Delegation Conflict

Client A holds a write delegation (exclusive cache rights) while Client B attempts to open the same file for writing. Server must recall Client A's delegation — if recall fails or times out, Client B is blocked or gets NFS4ERR_DELAY.

Locking

Lock State Collision

Two clients attempt incompatible byte-range locks on the same file region. One receives NFS4ERR_LOCK_CONFLICT. Also occurs after a server restart if a client tries to reclaim a lock that another client already acquired during the grace period.

Open State

Open State Conflict

A client holds an open stateid for a file that was deleted or renamed on the server by another client. The server returns NFS4ERR_STALE or NFS4ERR_NOENT on subsequent operations using the old stateid.

Sequence

Sequence ID (seqid) Error

NFSv4.0 uses per-owner sequence IDs to order state-changing operations. A retransmitted or out-of-order request with the wrong seqid causes NFS4ERR_BAD_SEQID. Common after network glitches or client-side RPC retransmission storms.

Filehandle

Stale Filehandle (NFS4ERR_STALE)

A client holds a cached filehandle for a file or directory that was deleted, moved, or the volume was unmounted and remounted. The server no longer recognises the filehandle. Results in Stale file handle errors on the client.

1 — ID Domain Mismatch (the most common collision)

NFSv4 does not transmit numeric UIDs in protocol messages the way NFSv3 does. Instead, it uses a string identity format: username@domain. When a client creates a file, it sends the owner as ramesh@example.com. The server translates this to a local UID using its ID mapping service (idmapd or sssd). If the domain strings do not match between client and server, translation fails and ONTAP maps the owner to the anonymous UID (default: 65534 / nobody).

❌ Mismatch — collision occurs
Client /etc/idmapd.conf:
Domain = localdomain

ONTAP SVM setting:
NFSv4 ID Domain = defaultv4iddomain.com

Result: owner string sent as
ramesh@localdomain
Server cannot map → nobody (65534)
✓ Match — no collision
Client /etc/idmapd.conf:
Domain = example.com

ONTAP SVM setting:
NFSv4 ID Domain = example.com

Result: owner string sent as
ramesh@example.com
Server maps → UID 1001 (ramesh)
⚠️
Symptom you will see All files created by NFS clients appear owned by nobody or 65534 when viewed from other clients or from the ONTAP CLI with ls -la. Existing files the client previously owned now show as owned by nobody. Write operations on files you own return EPERM even though the export policy grants rwrule.

2 — Delegation Conflict

A delegation is a grant from the server allowing a client to cache file data and metadata locally without checking back with the server on every operation. A write delegation grants exclusive access — the client can read and write the file and cache changes, and the server guarantees no other client will open the file for writing until the delegation is recalled.

A delegation collision becomes a hard problem when the recall fails — typically because Client A crashed, lost network connectivity, or is overloaded and not processing callbacks. ONTAP waits up to the lease timeout before forcibly revoking the delegation. During this window, Client B is stuck with repeated NFS4ERR_DELAY responses.

3 — Lock State Collision

NFSv4 supports POSIX-style byte-range locking. A lock state collision occurs when two clients attempt to hold incompatible locks on overlapping byte ranges of the same file. Common scenarios:

  • Two writers: Client A holds an exclusive write lock on bytes 0–4095. Client B requests an exclusive lock on bytes 0–8191. ONTAP returns NFS4ERR_LOCK_CONFLICT to Client B.
  • Post-restart reclaim conflict: After a server restart, Client A tries to reclaim a lock it held before the restart. But Client B was granted the same lock during the grace period. Client A gets NFS4ERR_RECLAIM_CONFLICT.
  • Expired lease lock reclaim: Client A's lease expired while it was disconnected. It reconnects and tries to reclaim its locks, but the server already reclaimed and reassigned them. The client gets NFS4ERR_EXPIRED.

4 — Open State Conflict

When a client opens a file via NFSv4, the server assigns a stateid — a 128-bit identifier binding the open to a specific client and sequence counter. Subsequent operations (reads, writes, locks) must reference a valid stateid. A collision occurs when:

  • File deleted while open: Client A has the file open. Client B (or a local process on the ONTAP node) deletes the file. Client A's stateid now references a file that no longer exists — ONTAP returns NFS4ERR_STALE on the next operation.
  • Volume remount / junction change: Admin changes a volume junction path while clients have open stateids against files in that volume. All existing stateids become stale.
  • SVM failover / LIF migration: In NFSv4.0, a client's session is tied to a specific IP address. If the LIF migrates to a different IP during an HA failover, the client's existing stateids are unknown to the new network path, causing NFS4ERR_STALE_CLIENTID.

5 — Sequence ID (seqid) Error

In NFSv4.0, every state-changing operation (OPEN, CLOSE, LOCK, LOCKU, RENAME, REMOVE) carries a sequence ID (seqid) that must increment by exactly 1 from the previous operation for the same open-owner. The server tracks the last seen seqid and rejects operations where the seqid is unexpected.

  • NFS4ERR_BAD_SEQID — the seqid in the request does not match what the server expects. Causes the operation to be dropped entirely.
  • NFS4ERR_OLD_STATEID — a stateid's embedded sequence counter is behind the server's current value. Happens when a client retransmits a stale request after the state was updated.

NFSv4.1 eliminates per-owner seqids entirely, replacing them with per-session slot sequence numbers — one of the key reasons NFSv4.1 is preferred for enterprise ONTAP deployments.

6 — Stale Filehandle

A filehandle is a server-assigned opaque identifier for a file or directory. NFSv4 clients cache filehandles. A stale filehandle occurs when the server no longer recognises a cached filehandle — the file was deleted, the volume was dismounted, or the SVM was restarted. Symptoms: applications receive ESTALE, mount points appear frozen, ls hangs or returns Stale file handle.

Section 03 · CLI Detection

ONTAP CLI — Detect Every Collision Type

Check ID domain configuration

ONTAP CLI — Show current NFSv4 ID domain on SVM 
# Show NFSv4 ID mapping domain configured on the SVM
vserver nfs show \
  -vserver svm1 \
  -fields  v4-id-domain

# Output
vserver  v4-id-domain
-------  -------------------
svm1     defaultv4iddomain.com   ← must match client idmapd.conf Domain=

# Show full NFS config including ID domain, lease, grace timeouts
vserver nfs show -vserver svm1 -instance | grep -E "domain|lease|grace"
ONTAP CLI — Check files owned by nobody (domain collision symptom) 
# Check file ownership — nobody UID 65534 indicates domain mismatch
set -privilege advanced
vserver security file-directory show \
  -vserver  svm1 \
  -path     /vol-app01

# Show actual UID stored on a file from ONTAP CLI
volume file show-inode \
  -vserver svm1 \
  -volume  vol-app01 \
  -path    /testfile.txt

set -privilege admin

Section 05 · Fix Domain Mismatch

Fix 1 — Resolve ID Domain Mismatch

1

Decide on your domain string

Choose one consistent domain string — typically your DNS domain (e.g. example.com). It does not need to be resolvable via DNS — it just needs to be identical on every NFSv4 client and on the ONTAP SVM. Using localdomain on Linux and anything else on ONTAP is the most common misconfiguration.

2

Update ONTAP SVM domain

ONTAP CLI — Set the NFSv4 ID domain 
# Change the NFSv4 ID mapping domain on the SVM
vserver nfs modify \
  -vserver      svm1 \
  -v4-id-domain example.com

# Verify the change
vserver nfs show -vserver svm1 -fields v4-id-domain

# No restart required — takes effect immediately for new operations
3

Update all Linux clients

Linux Client — Update idmapd.conf and restart service 
# Edit /etc/idmapd.conf — set Domain to match ONTAP
vi /etc/idmapd.conf

# Change or add this line in the [General] section:
Domain = example.com

# Restart the idmapd service
systemctl restart rpc-idmapd

# Flush the ID mapping cache
nfsidmap -c

# Remount the NFS filesystem to pick up the new mapping
umount  /mnt/vol-app01
mount   /mnt/vol-app01

# Verify file ownership now shows correct user
ls -la /mnt/vol-app01/
ℹ️
RHEL / Rocky Linux note On RHEL 8+ and Rocky Linux, the idmapd service may be replaced by sssd for ID mapping when the system is joined to Active Directory or an LDAP realm. In that case, set the NFSv4 domain in /etc/sssd/sssd.conf under [domain/...] as ldap_idmap_range_size and ensure the SVM domain matches the AD/LDAP domain.
Section 06 · Clear Lock State

Fix 2 — Clear Stuck Lock State & Delegations

ONTAP CLI — Break a stuck byte-range lock 
# Find the lock holding up other clients
vserver locks show -vserver svm1 -path /vol-app01/dbfile.dat

# Break the lock by client IP (use only when client is confirmed dead/unreachable)
set -privilege advanced

vserver locks break \
  -vserver    svm1 \
  -volume     vol-app01 \
  -path       /vol-app01/dbfile.dat \
  -client-ip  10.10.10.55

set -privilege admin

# Verify lock is gone
vserver locks show -vserver svm1 -path /vol-app01/dbfile.dat
🚨
Break locks with caution Breaking an active lock on a live client can cause data corruption if the client is mid-write. Only break locks when you have confirmed the holding client is crashed, network-isolated, or has already given up on the file. Always check with the application owner before breaking a database file lock.
ONTAP CLI — Revoke a stuck delegation 
set -privilege advanced

# List all delegations
vserver nfs delegation show -vserver svm1

# Revoke delegation from a specific client that is not responding to recall
vserver nfs delegation revoke \
  -vserver   svm1 \
  -client-ip 10.10.10.55

# Revoke all delegations on a volume (e.g. before maintenance)
vserver nfs delegation revoke \
  -vserver svm1 \
  -volume  vol-app01

set -privilege admin
ONTAP CLI — Clear all NFSv4 state for a crashed client 
# When a client VM crashed and left orphan state — clear it completely
set -privilege advanced

# Find the client ID for the crashed client
vserver nfs clients show \
  -vserver   svm1 \
  -client-ip 10.10.10.55

# Delete the client state (locks + delegations + open stateids)
vserver nfs clients delete \
  -vserver   svm1 \
  -client-ip 10.10.10.55

set -privilege admin
Section 07 · Delegation Tuning

Fix 3 — Delegation Tuning to Prevent Conflicts

If delegation conflicts are recurring — particularly in environments with many concurrent writers — you can tune or disable delegations on specific SVMs or volumes.

ONTAP CLI — Disable delegations on an SVM 
# Disable read AND write delegations on the SVM
vserver nfs modify \
  -vserver                    svm1 \
  -v4.0-read-delegation       disabled \
  -v4.0-write-delegation      disabled \
  -v4.1-read-delegation       disabled \
  -v4.1-write-delegation      disabled

# Disable only write delegations (keep read for performance)
vserver nfs modify \
  -vserver               svm1 \
  -v4.1-write-delegation disabled

# Verify
vserver nfs show -vserver svm1 \
  -fields v4.0-read-delegation,v4.0-write-delegation,v4.1-read-delegation,v4.1-write-delegation
ONTAP CLI — Tune lease and grace timeout 
# Increase lease timeout — gives slow clients more time to renew
# before their state is considered expired (default: 30s)
vserver nfs modify \
  -vserver        svm1 \
  -v4-lease-seconds 60

# Increase grace period — gives clients more time to reclaim
# state after a server restart (default: 45s, must be > lease)
vserver nfs modify \
  -vserver         svm1 \
  -v4-grace-seconds 90

# Show current values
vserver nfs show -vserver svm1 \
  -fields v4-lease-seconds,v4-grace-seconds
⚠️
Lease vs Grace — important relationship Grace timeout must always be greater than lease timeout. If grace < lease, clients whose leases have not yet expired may miss the grace window and be unable to reclaim their state after a server restart — generating NFS4ERR_NO_GRACE errors. ONTAP will reject the modification if you try to set grace < lease.
Section 08 · Prevention

Prevention Best Practices

Best PracticeWhy it prevents collisionsHow to implement
Use NFSv4.1 instead of NFSv4.0 Eliminates per-owner seqid entirely. Sessions are more resilient to network interruption. Trunking support on ONTAP reduces LIF migration collision risk. Mount with nfsvers=4.1. Verify with cat /proc/mounts.
Set identical ID domain on all systems Prevents the most common collision — nobody ownership from domain mismatch. Set Domain= in /etc/idmapd.conf and match -v4-id-domain on ONTAP.
Disable write delegations on shared write volumes Eliminates delegation recall collisions on volumes with multiple concurrent writers. vserver nfs modify -v4.1-write-delegation disabled
Set grace > lease timeout Ensures all clients have time to reclaim state after a server restart before new conflicting state is granted. Set lease=60, grace=90 on SVMs with sensitive workloads.
Monitor EMS for nfs.* events Early detection of domain mismatches, delegation recall failures, and expired leases before they cause application impact. Configure event notification in ONTAP to alert on nfs.* severity ERROR/WARNING.
Avoid live LIF migration with NFSv4.0 clients NFSv4.0 ties session state to a specific IP. LIF migration causes the client to see a different server and lose its stateids. Upgrade clients to NFSv4.1 (session-based, not IP-tied) before performing LIF moves.
Clean up stale client state proactively Crashed VMs and decommissioned hosts leave orphaned locks and delegations that block new clients. After decommissioning a host, run vserver nfs clients delete -client-ip <ip>.
Section 09 · Interview Q&A

Interview & Exam Questions

  • Q01 What is an NFSv4 collision and why does it not occur in NFSv3? ▾
    An NFSv4 collision is a conflict that arises because NFSv4 is a stateful protocol — the server holds per-client state including open file records (stateids), byte-range locks, and delegations tied to a lease. When two clients disagree about who owns a lock, when a client's state expires, or when identity translation fails between client UID and server user string, a collision occurs. NFSv3 is stateless — every RPC is self-contained and the server holds no per-client state. There is no lease, no delegation, and no stateid to expire or conflict. NFSv3's locking was handled by the separate NLM (Network Lock Manager) protocol which was advisory and much simpler in scope.
  • Q02 All files created by Linux NFSv4 clients appear owned by "nobody". What is the cause and fix? ▾
    This is an ID domain mismatch. NFSv4 transmits file ownership as a string in the format username@domain. If the domain configured in the client's /etc/idmapd.conf (e.g. localdomain) does not match the domain configured on the ONTAP SVM's v4-id-domain parameter (e.g. defaultv4iddomain.com), translation fails and ONTAP maps the owner to the anonymous UID 65534 (nobody). Fix: choose one consistent domain string, set Domain=example.com in /etc/idmapd.conf on all clients, restart rpc-idmapd, run nfsidmap -c to flush the cache, then set vserver nfs modify -vserver svm1 -v4-id-domain example.com on ONTAP.
  • Q03 What is an NFSv4 delegation and how can it cause a collision? ▾
    A delegation is a grant from the NFSv4 server giving a client the right to cache file data and metadata locally without polling the server. A write delegation grants exclusive cache rights — the server promises no other client will open the file for writing. A collision occurs when a second client tries to open the same file for writing while the first client holds a write delegation. ONTAP must first recall the delegation from Client A (via a callback to the client). If Client A is slow to respond or unreachable, Client B receives repeated NFS4ERR_DELAY responses and is blocked until the recall completes or the lease expires and ONTAP forcibly revokes the delegation.
  • Q04 What is the NFSv4 grace period and what happens if a client misses it? ▾
    The grace period is a window after a server restart (default 45 seconds on ONTAP) during which clients can reclaim their previous state — locks, opens, and delegations — before new conflicting state is granted to other clients. During the grace period, the server defers granting new state that could conflict with reclaims. If a client misses the grace period (e.g. was itself restarting or network-isolated during the grace window), it cannot reclaim its previous locks and receives NFS4ERR_RECLAIM_CONFLICT or NFS4ERR_NO_GRACE. It must then start fresh — re-opening files and re-acquiring locks from scratch — potentially finding that another client was granted conflicting state in the meantime.
  • Q05 A crashed VM left orphan NFSv4 locks blocking other clients. What do you do on ONTAP? ▾
    First confirm the client is truly dead and not just slow — check connectivity to the VM's IP. Then use the ONTAP CLI in advanced privilege mode: run vserver nfs clients show -vserver svm1 -client-ip <crashed-vm-ip> to confirm it still has active state. Then run vserver nfs clients delete -vserver svm1 -client-ip <crashed-vm-ip> to clear all orphan state (locks, delegations, open stateids) for that client. Alternatively, break individual locks with vserver locks break. Once the state is cleared, the blocking clients will be able to acquire their locks on the next retry.
  • Q06 What is the difference between NFS4ERR_EXPIRED, NFS4ERR_BAD_STATEID, and NFS4ERR_STALE? ▾
    NFS4ERR_EXPIRED means the client's lease expired before it could be renewed — all state associated with that Client ID has been reclaimed by the server. The client must re-establish its Client ID and re-acquire all open/lock state from scratch. NFS4ERR_BAD_STATEID means the stateid in the request is syntactically valid but does not match any known current stateid on the server — typically because the state was reclaimed, the stateid's sequence counter is wrong, or the wrong stateid was used in a compound operation. NFS4ERR_STALE means the filehandle referenced in the request is no longer valid on the server — the file or directory was deleted, the volume was unmounted, or the SVM was restarted. The client must invalidate its cached filehandle and re-lookup the file from the root.
  • Q07 Why is NFSv4.1 preferred over NFSv4.0 for enterprise ONTAP deployments? ▾
    NFSv4.1 fixes several collision-prone weaknesses of NFSv4.0. First, it replaces per-owner sequence IDs (seqids) with per-session slot sequences — eliminating NFS4ERR_BAD_SEQID entirely and making retransmissions safe. Second, sessions in NFSv4.1 are identified by a session ID, not a client IP address — so LIF migration during an HA failover does not invalidate the client's state the way it does in NFSv4.0. Third, NFSv4.1 introduces session trunking, allowing a client to use multiple network paths in a single session for both load balancing and redundancy. Fourth, NFSv4.1 is the foundation for pNFS and for NFSv4.2 features like extended attributes and sparse file operations.
  • Q08 How do you confirm which NFSv4 operation is causing a collision without a network capture? ▾
    Use debug nfs trace start -vserver svm1 -client-ip <ip> -trace-ops open,close,lock,locku,delegreturn,renew in advanced privilege mode on ONTAP. This captures a record of every specified NFSv4 operation exchanged between the server and that client. After reproducing the error on the client, stop the trace with debug nfs trace stop and view the results with debug nfs trace show. The trace output shows the operation name, the stateid involved, and the error code returned — identifying exactly which operation collided and with what. Also check EMS with event log show -message-name nfs.* which logs delegation recall failures, domain mismatch events, and expired lease warnings automatically.

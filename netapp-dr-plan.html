ARCH

How does SnapMirror async replication work at the data level?

▾

SnapMirror tracks changed blocks using ONTAP's snapshot engine. After an initial baseline transfer, each replication cycle ships only the changed blocks between the last common snapshot and the current one.

  Primary (Source)                           Secondary (Destination)
  ─────────────────                          ───────────────────────
  vol1  (RW)  ──── Snapshot A ────────────►  vol1_dr  (DP)
                                                      Snapshot A
              ──── Δ blocks (A→B) ────────►          Snapshot B
              ──── Δ blocks (B→C) ────────►          Snapshot C
                      ↑
           Only changed blocks travel the wire

The destination volume is in Data Protection (DP) mode — read-only until a failover or test break. The relationship schedule is defined by a SnapMirror policy and a job schedule on the destination cluster.

What is SVM-DR and when should you use it instead of volume-level SnapMirror?

▾

SVM-DR replicates an entire Storage Virtual Machine — including all its volumes, network configuration (LIFs, export policies, CIFS shares, NFS exports), and identity — to a destination cluster. This means that at failover time, the SVM comes up at the DR site without manual reconfiguration of shares, exports, or network settings.

AspectVolume SnapMirrorSVM-DR
ScopeIndividual volumesWhole SVM (volumes + config)
Network config at DRMust rebuild manuallyReplicated automatically
Failover complexityHigher — more manual stepsLower — SVM activates as-is
Best forSelective volume DRFull application DR with consistent identity

DR Plan — What to Document Before an Incident

The runbook nobody wants to write at 2am
PLAN

What must a NetApp DR plan document before any incident occurs?

▾

A DR plan must be executable by someone under pressure at 3am. Every ambiguity you leave in the plan will cost minutes or hours during an actual incident. Mandatory sections:

🗂️
Inventory — source cluster, destination cluster, SVM names, volume names, SnapMirror relationship names, and the SnapMirror policy applied to each. Export from ONTAP and keep a copy outside the array.
🎯
RTO / RPO targets per application — agree these with the business owner, not the storage team. Different workloads (ERP vs. dev/test) have different tolerance.
🌐
Network cutover plan — IP address scheme at DR site, DNS changes required, load balancer configuration, and who owns each of those changes (usually not the storage team).
📞
Escalation contacts — NetApp Support contract number, primary/backup administrators, application owners, and network team contacts. With the site down, you cannot look these up in an internal wiki.
⚠️
Known gaps — document any workloads NOT covered by DR replication. "Oracle DB on vol_prod03 is not replicated — business accepted this risk." No surprises during an incident.

How do you audit the health of SnapMirror relationships as part of DR readiness?

▾ 
# Show all SnapMirror relationships and their lag/status
snapmirror show -fields state,status,lag-time,healthy

# Filter for unhealthy or lagged relationships
snapmirror show -fields state,lag-time,healthy -healthy false

# Show transfer status (active transfers)
snapmirror show -fields transfer-state

# Get transfer history for a specific relationship
snapmirror history show -source-path svm1:vol1

# Force an immediate update (don't wait for schedule)
snapmirror update -source-path svm1:vol1 -destination-path svm2_dr:vol1_dr
📊
Key fields to monitor in DR readiness checks: lag-time (should be within your RPO window), healthy (must be true), and state (should be Snapmirrored for async, InSync for sync).

Failover Execution — Step by Step

Run these phases in order. Do not skip verification steps.
1.1
Declare the incident and notify stakeholders
Confirm the primary site or cluster is genuinely unreachable — not a management network issue. Page the storage team lead, application owners, and network team before starting any failover actions.
1.2
Check SnapMirror lag at the DR cluster
SSH to the destination cluster and check lag time to understand your RPO exposure.
# Run on the DESTINATION cluster
snapmirror show -fields lag-time,state,transfer-state
# Note the lag-time for every critical volume — this is data at risk
1.3
Determine if a planned or unplanned failover is needed
Planned (primary site is still accessible) allows a clean quiesce-and-break. Unplanned (primary is down) skips quiesce — you accept whatever the last successful transfer captured.
⚠️
Planned failover only. If the primary site is down, skip to Phase 3 (Break). Quiesce requires the source cluster to be reachable.
2.1
Stop application I/O to source volumes
Work with application owners to cleanly shut down databases, unmount NFS mounts, and disconnect iSCSI sessions on the application side. Do not skip this — dirty failover can corrupt data.
2.2
Quiesce the SnapMirror relationship
Quiesce pauses scheduled transfers so you can run a final manual update cleanly.
# Run on the DESTINATION cluster
snapmirror quiesce -destination-path svm2_dr:vol1_dr

# Verify quiesced state
snapmirror show -destination-path svm2_dr:vol1_dr -fields state
# Expected: Quiesced
2.3
Run a final SnapMirror update to minimize RPO
Push any remaining dirty blocks to the destination before breaking the relationship.
snapmirror update -destination-path svm2_dr:vol1_dr

# Wait for transfer to complete
snapmirror show -destination-path svm2_dr:vol1_dr -fields transfer-state
# Expected: Idle (no active transfer)
🔴
This step is irreversible until you re-establish the relationship. Breaking the relationship makes the destination volume writable. You cannot re-sync from the source until the relationship is explicitly re-initialized or resync'd.
3.1
Break the SnapMirror relationship
This makes the destination volume read-write. Run on the destination cluster.
# Run on the DESTINATION cluster
snapmirror break -destination-path svm2_dr:vol1_dr

# For SVM-DR — break at the SVM level
snapmirror break -destination-path svm2_dr:

# Verify broken state
snapmirror show -destination-path svm2_dr:vol1_dr -fields state
# Expected: Broken-off
3.2
Verify destination volume is read-write
The volume should now show type RW instead of DP.
volume show -vserver svm2_dr -volume vol1_dr -fields type,state
# Expected: type=RW, state=online
4.1
Bring up network interfaces (LIFs) at DR site
For SVM-DR, network LIFs are replicated but initially down. Bring them up manually or configure them to auto-start based on the DR network configuration documented in your plan.
# Check LIF status on DR SVM
network interface show -vserver svm2_dr

# Bring up a specific LIF
network interface modify -vserver svm2_dr -lif data_lif_01 -status-admin up

# Migrate LIF to a specific home port if needed
network interface modify -vserver svm2_dr -lif data_lif_01 -home-node dr-node-01 -home-port e0d
4.2
Verify NFS exports and CIFS shares are accessible
For SVM-DR, export policies replicate automatically. For volume SnapMirror, you must manually recreate or pre-stage export policies and CIFS shares on the destination SVM.
# Check export policy on the volume
volume show -vserver svm2_dr -fields export-policy

# List export policy rules
export-policy rule show -vserver svm2_dr -policyname default

# List CIFS shares
cifs share show -vserver svm2_dr
4.3
Test data accessibility before handing to application team
Mount a share or NFS export from a test host and verify data is visible and accessible. Run a read of a known file. Check iSCSI connectivity if block volumes are in scope.
# NFS test mount from a Linux host
mount -t nfs 10.20.1.50:/vol1_dr /mnt/test
ls -la /mnt/test
cat /mnt/test/known-file.txt

# iSCSI — discover and log in to target
iscsiadm -m discovery -t sendtargets -p 10.20.1.51
iscsiadm -m node --login
5.1
Update DNS to point to DR site IPs
Work with the network team to update DNS A records and any load balancer VIPs. Set a short TTL (60–300 seconds) before any planned failover. For unplanned, accept the TTL you have.
# Example — verify new DNS resolution from application host
nslookup nas-prod.example.com
# Should resolve to DR site IP now
5.2
Hand off to application team and validate application layer
Start the application (database, application server) and confirm end-to-end functionality. This is the true RTO milestone — when the application is serving users, not when the volume comes online.
5.3
Document the failover — timestamp, RPO achieved, data loss (if any)
Record the time of the last successful SnapMirror transfer versus time of incident to calculate actual RPO. This feeds the post-incident review and potential improvements to replication schedules.

DR Testing — Without Breaking Production

Test regularly. Untested DR plans are just hope documents.
TEST

How do you test a SnapMirror failover without impacting the live replication relationship?

▾

Use a FlexClone of the SnapMirror destination volume. This creates a writable, space-efficient clone from a DP snapshot — you can test fully without touching the live replication relationship.

# On the DESTINATION cluster — list available snapshots on the DP volume
snapshot show -vserver svm2_dr -volume vol1_dr

# Clone from the latest SnapMirror snapshot
volume clone create -vserver svm2_dr -flexclone vol1_dr_test \
  -parent-volume vol1_dr -parent-snapshot snapmirror.snap01 \
  -type RW -junction-path /vol1_dr_test

# Mount and test — SnapMirror relationship continues unaffected
volume show -vserver svm2_dr -volume vol1_dr_test

# Clean up after test
volume unmount -vserver svm2_dr -volume vol1_dr_test
volume clone delete -vserver svm2_dr -flexclone vol1_dr_test
✅
This is the recommended approach for DR testing. The SnapMirror relationship remains intact and continues replicating throughout the test. No risk to production or replication schedule.

Failback — Returning to Primary

After the primary site is restored, resync and reverse the relationship
FAILBACK

Walk through the steps to resync and fail back from DR to primary after the primary site is restored.

▾

Failback is a reverse SnapMirror — the DR site becomes the temporary source, replicating changed data back to the (now repaired) primary before you cut over again.

# Step 1 — Resync (run on the DESTINATION / DR cluster)
# This re-establishes the SnapMirror relationship using the common snapshot
# Direction is still DR→Primary at this point (reverse resync)
snapmirror resync -source-path svm2_dr:vol1_dr -destination-path svm1:vol1

# Step 2 — Monitor until InSync / Snapmirrored
snapmirror show -destination-path svm1:vol1 -fields state,lag-time

# Step 3 — When lag-time is acceptable, quiesce application at DR site
# Then run a final update
snapmirror update -destination-path svm1:vol1

# Step 4 — Break the reverse relationship (makes primary RW again)
snapmirror break -destination-path svm1:vol1

# Step 5 — Re-establish the original relationship (primary → DR)
# This restores your normal DR protection
snapmirror resync -source-path svm1:vol1 -destination-path svm2_dr:vol1_dr
⚠️
Do not run the final resync (Step 5) until the primary site is confirmed fully stable. A second failure immediately after failback with no DR protection in place would be catastrophic.

How do you verify MetroCluster health as part of ongoing DR readiness?

▾ 
# Full MetroCluster health check — run this daily/weekly
metrocluster check run
metrocluster check show

# Verify NVRAM mirroring (sync health)
metrocluster interconnect mirror show

# Check aggregate mirror state
storage aggregate show -fields mirror-status,state

# Expected: mirror-status=mirrored for all data aggregates
🔔
Automate metrocluster check run output into your monitoring platform (Harvest, AIQUM). A degraded mirror or split-brain condition must be detected before an actual site failure — not during one.

Core NFS Access

General NFS Access true

The master on/off switch for NFS on this SVM. When true, the SVM can serve NFS to clients. Setting it to false immediately blocks all new NFS connections without removing any configuration — useful for emergency maintenance windows.

If clients suddenly lose access, this is the first parameter to check. A mis-set false here overrides every other NFS setting.

NFS Mount Root Only enabled

Restricts the NFS MOUNT RPC call (used during mount) so it can only originate from a privileged source port (<1024) on the client. Most Unix/Linux NFS clients naturally use privileged ports for mount, but some embedded or non-standard clients may not.

Keep enabled in production. Disable only when specific clients (e.g., older Windows NFS or certain NAS gateways) cannot use privileged ports.

NFS Root Only disabled

When enabled, restricts all NFS requests (not just MOUNT) to originate from privileged source ports. This is a stricter version of Mount Root Only and may break clients that use unprivileged ports for data I/O.

Generally left disabled unless your security policy explicitly mandates it, as it can cause compatibility issues with modern NFS clients.

Showmount Enabled enabled

Controls whether the SVM responds to showmount -e queries, which list the NFS exports available on the server. Useful for administrators and clients discovering available shares.

Can be disabled in high-security environments to prevent clients from enumerating exports. Export access is still enforced by export policies regardless of this setting.

Provide Root Path as Showmount State disabled

When enabled, the showmount response always reports the root path ("/") as the exported path, regardless of actual export policy paths. Some older clients or NFS automounters expect this format.

Qtree Exports Enabled disabled

Enables or disables the ability to export individual qtrees as NFS mount points. When disabled, only volumes (not their qtrees) can be independently mounted by NFS clients.

Enable if workloads require per-qtree mount points with separate export policies. Keep disabled if qtree-level exports are not needed — it simplifies administration.

Protocol Versions & Transport

NFS v3 enabled

NFSv3 is the most widely supported NFS version. It is stateless (no persistent client–server state), uses separate MOUNT, NLM (locking), and NSM (status monitor) protocols, and supports both UDP and TCP. It remains common in legacy environments and is often the fallback for clients that do not support NFSv4.

If your workloads are exclusively NFSv4/4.1, consider disabling v3 to reduce attack surface and eliminate NLM/NSM port exposure.

NFS v4.0 enabled

NFSv4.0 is a stateful protocol that runs exclusively over TCP. It eliminates the need for separate MOUNT, NLM, and NSM side-protocols, folding everything into a single port (2049). Introduces file-level locking, compound operations (multiple ops in one RPC), and ACL support.

NFSv4.0 requires correct ID mapping (see NFSv4 ID Mapping Domain). Mismatched domains between client and server cause all files to show as nobody.

UDP Protocol enabled

Allows NFS traffic to use UDP as its transport. UDP has lower per-packet overhead than TCP but provides no reliability, ordering guarantees, or flow control. NFSv4 never uses UDP — this setting only affects NFSv3 (and NFSv2 if enabled).

In most modern environments, UDP NFS is unnecessary. Disable it to force all NFS over TCP, which is more reliable over congested or lossy networks.

TCP Protocol enabled

Allows NFS traffic over TCP. TCP is required for NFSv4/4.1/4.2 and strongly recommended for NFSv3 in production environments. TCP provides error detection, retransmission, and flow control that UDP lacks.

This should almost never be disabled. Disabling TCP while keeping UDP means all NFSv4 clients will fail to connect.

UDP Maximum Transfer Size (bytes) 32768

The largest single NFS read or write I/O size allowed over UDP. 32768 bytes (32 KB) is the traditional maximum for UDP NFS, as larger UDP datagrams risk fragmentation. Clients negotiate down to this value if they request larger transfers.

TCP Maximum Transfer Size (bytes) 65536

The largest single NFS read or write I/O block over TCP. 65536 bytes (64 KB) is the default. Larger values (e.g., 1 MB = 1048576) can improve throughput for sequential large-file workloads on high-bandwidth networks by reducing the number of individual RPC calls needed.

For big data or HPC workloads on 10GbE+ networks, increasing this to 262144 (256 KB) or 1048576 (1 MB) often improves sequential throughput significantly.

NFSv4 Core Settings

NFSv4 ID Mapping Domain defaultv4iddomain.com

In NFSv4, user and group identities are transmitted as user@domain strings (e.g., alice@example.com) rather than numeric UIDs/GIDs. The domain here must exactly match the Domain = setting in /etc/idmapd.conf on every client. A mismatch causes ownership to display as nobody or 4294967294.

Always set this to your actual DNS domain (e.g., company.com). The default defaultv4iddomain.com is a placeholder — update it before putting NFSv4 clients into production.

NFSv4 Lease Timeout Value (in secs) 30

The period (in seconds) for which a client's NFSv4 lease is valid. Clients must renew their lease within this window or the server may reclaim their locks and open file state. 30 seconds is the ONTAP default and is appropriate for most LAN environments.

In high-latency WAN environments, consider increasing this to 60–90 seconds to avoid spurious lease expirations. Decreasing it speeds up state recovery after a client crash but increases renewal traffic.

NFSv4 Grace Timeout Value (in secs) 45

After a server restart or failover, NFSv4 enters a "grace period" during which it accepts only reclaim requests (clients re-establishing their previous lock/open state). New lock requests are queued until the grace period expires. The 45-second value is slightly longer than the 30-second lease, giving all clients time to reclaim.

Set the grace timeout to at least the lease timeout. Clients that do not reclaim within the grace period lose their locks and may receive NFS4ERR_NO_GRACE errors.

NFSv4.0 ACL Support disabled

Controls whether NFSv4.0 GETATTR and SETATTR calls can read and write NFSv4 Access Control Lists on files. When disabled, ACL requests over NFSv4.0 are ignored or return errors. This does not affect POSIX permission mode bits.

Enable if applications running over NFSv4.0 require fine-grained ACL control (e.g., Solaris or AIX clients using NFSv4 ACLs).

NFSv4.0 Read Delegation Support disabled

Read delegations allow a client to cache a file read-locally, knowing the server will notify (recall) the delegation if another client modifies the file. This reduces round-trips for read-heavy workloads but adds complexity to recall handling.

Enable for read-intensive workloads where files are rarely modified by multiple clients simultaneously. Avoid on highly shared, frequently modified files.

NFSv4.0 Write Delegation Support disabled

Write delegations grant a single client exclusive write access to a file, allowing it to buffer writes locally. Other clients' access to that file is blocked until the delegation is recalled. Can significantly improve write throughput for single-writer patterns.

Use with caution. Write delegations can delay I/O for other clients waiting for a recall. Best suited for workloads with clear single-writer patterns.

NFSv4 Support for Numeric Owner IDs enabled

When enabled, the server accepts and returns numeric UIDs/GIDs in NFSv4 owner strings (e.g., 1234@domain) in addition to named users. This is useful when name-service lookups fail or when clients and server have different user databases.

NFSv4 Validate UTF-8 Encoding of Symbolic Link Data disabled

When enabled, the server validates that symbolic link targets contain valid UTF-8 characters, per the NFSv4 specification. Some legacy systems create symlinks with non-UTF-8 byte sequences.

Enable in strict standards-compliance environments. Disable if existing symlinks were created with non-UTF-8 names (e.g., Latin-1 encoded paths).

Maximum Number of ACEs per ACL 400

Sets the upper limit on how many Access Control Entries (ACEs) a single ACL can contain. The default of 400 is sufficient for virtually all workloads. Excessively large ACLs can impact performance during permission evaluation.

Show Change in FSID as NFSv4 Clients Traverse Filesystems enabled

Controls whether NFSv4 clients see a different Filesystem ID (FSID) when they cross a volume or qtree boundary (a filesystem junction). When enabled, FSID changes at junctions — which is spec-compliant behavior and helps clients detect they have crossed a mount point boundary.

NFSv4.1 & pNFS

NFSv4.1 Minor Version Support enabled

Enables the NFSv4.1 minor version, which introduces sessions (eliminating the need for separate connection-level state), improved security, and the foundation for pNFS (parallel NFS). NFSv4.1 is required for modern Linux clients (kernel 3.9+) using the default mount options.

NFSv4.1 Parallel NFS Support (pNFS) disabled

pNFS allows clients to access storage nodes directly in parallel, bypassing the metadata server for data I/O. ONTAP supports the "files" layout type, where pNFS clients can be directed to individual nodes in the cluster for data. Can dramatically improve throughput in large-scale parallel workloads.

Enable for HPC or large-scale analytics workloads where multiple clients access different files simultaneously. Requires clients that support the NFSv4.1 pNFS files layout.

Number of Slots in NFSv4.x Session Slot Tables 180

NFSv4.1 sessions use a slot table for request tracking and exactly-once semantics. Each slot can hold one in-flight RPC at a time. This value sets how many concurrent outstanding RPCs a single session can have. Higher values allow more parallelism per client session.

For workloads with high concurrency (e.g., databases with many threads), increasing this value can improve throughput. Reduce it on memory-constrained systems with many clients.

Size of Reply Cached in Each NFSv4.x Session Slot (bytes) 640

Each slot in the session table can cache the server's reply to the most recent RPC in that slot, enabling exactly-once semantics (idempotent replay detection). This sets how many bytes of each reply are cached. 640 bytes is sufficient for most metadata operations but not for large data reads.

NFSv4.1 ACL Support disabled

Enables ACL attribute support for NFSv4.1 sessions. Functionally equivalent to the NFSv4.0 ACL setting but applies specifically to NFSv4.1 connections. When disabled, ACL get/set operations over 4.1 return NFS4ERR_ATTRNOTSUPP.

NFSv4.1 Read Delegation Support disabled

Same semantics as NFSv4.0 read delegations, but applies to clients connected via NFSv4.1. NFSv4.1 delegation handling is improved over 4.0 due to session-based state management.

NFSv4.1 Write Delegation Support disabled

Same semantics as NFSv4.0 write delegations for NFSv4.1 clients. Session-based error recovery in NFSv4.1 makes delegation recall more reliable than in NFSv4.0.

NFSv4.1 Implementation ID Domain netapp.com

The domain portion of the server's NFSv4.1 implementation identifier, sent in the EXCHANGE_ID operation. Clients can use this to identify the vendor of the server implementation. This is informational and does not affect functionality.

NFSv4.1 Implementation ID Name NetApp Release 9.18.1

The human-readable name string in the NFSv4.1 implementation ID, identifying the server software version. Purely informational — returned to clients during session creation to identify the server.

NFSv4.1 Trunking Support disabled

Session trunking allows a single NFSv4.1 session to use multiple network connections (trunks) simultaneously, increasing aggregate bandwidth and providing path redundancy. The client must also support trunking (RFC 5661).

Enable on systems with multiple network interfaces per node when clients support trunking. Particularly valuable for high-throughput workloads on multi-NIC configurations.

NFSv4.2 Features

NFSv4.2 Extended Attributes Support enabled

NFSv4.2 (RFC 8276) adds support for user-defined extended attributes (xattrs) on files and directories — the same mechanism used by Linux getfattr/setfattr. These are separate from ACLs and allow applications to store arbitrary metadata alongside files.

Required for modern Linux workloads that use xattrs (e.g., SELinux labels, container image layers, some databases). Ensure clients mount with vers=4.2 to use this feature.

NFSV4.2 Sparse Files Operations Support disabled

Enables NFSv4.2 SEEK and ALLOCATE operations, which allow clients to create and manipulate sparse files (files with "holes" that do not consume physical storage). Applications like virtual machine disk images and databases commonly use sparse files.

Enable for VM workloads (e.g., QCOW2/VMDK images) or database applications that benefit from sparse file support to reduce storage consumption.

NFSV4.2 Security Label Support disabled

Enables Labeled NFS (RFC 7204), allowing MAC (Mandatory Access Control) security labels (e.g., SELinux or Smack labels) to be stored and retrieved on files via NFSv4.2. Required for environments where SELinux-enforcing hosts need consistent labels on NFS-mounted files.

Enable in SELinux-enforcing environments where label consistency across NFS is required. Clients must also be running in enforcing mode with the labeled NFS feature enabled in the kernel.

Security & Authentication

Permitted Kerberos Encryption Types aes-128, aes-256

Specifies which Kerberos encryption algorithms are accepted for krb5, krb5i (integrity), and krb5p (privacy) NFS mounts. Only AES-128 and AES-256 are permitted here — older, weaker algorithms (DES, RC4/ARCFOUR) are not listed, reflecting a modern security posture.

AES-256 (aes256-cts-hmac-sha1-96) is the preferred type. Both entries here mean the server accepts either, negotiating the strongest the client offers. Ensure your KDC and clients are configured to match.

AUTH_SYS Extended Groups Enabled disabled

NFS AUTH_SYS credentials carry the user's UID, primary GID, and up to 16 supplementary groups in the RPC credential itself. When disabled, ONTAP only evaluates those credentials. When enabled, ONTAP performs an additional name-service lookup to expand group membership beyond 16 groups.

Enable if users belong to more than 16 groups and those group memberships need to be honored for export policy or permission checks. Requires a correctly configured LDAP/NIS name service.

AUTH_SYS and RPCSEC_GSS Auxiliary Groups Limit 32

Sets the maximum number of auxiliary (supplementary) groups evaluated per user when Extended Groups is enabled. The default of 32 is above the NFS wire limit of 16 but below the Linux kernel maximum of 65536.

Vserver NTFS Unix Security Options use_export_policy

Controls how UNIX-style permission operations (chmod, chown) are handled on volumes with NTFS security style. use_export_policy means the behavior is determined per-export by the export policy rules. Other options include ignore (silently ignore UNIX operations) or fail (return an error).

Vserver Change Ownership Mode use_export_policy

Determines whether NFS clients can perform chown operations on files. use_export_policy defers the decision to the active export policy. Can also be set to restricted (only root may chown) or unrestricted (any owner may chown, POSIX default).

Ignore the NT ACL Check for NFS User 'root' disabled

On unified security style volumes, NFS root is normally subject to NT ACL permission checks. When enabled, NFS root bypasses NT ACL checks and has unrestricted access. This can be a significant security risk in mixed-protocol environments.

Keep disabled unless a specific application requires root-level NFS bypass of ACL checks. Audit access carefully if you enable this.

Skip Permission Check for NFS Write from Root/Owner disabled

When enabled, NFS write calls from the file owner or root skip the normal permission check. This can allow writes to read-only files by their owner. Not recommended for most environments.

Map Unknown UID to Default Windows User enable

When an NFS client presents a UID that cannot be mapped to a Windows user account (in mixed or unified security style), this setting controls whether the access falls back to a configured default Windows user account. Prevents access failures for unmapped users in mixed-protocol environments.

Enable or Disable Checksum for Replay-Cache enabled

The NFS replay cache (duplicate request cache for NFSv3) stores checksums of recent requests to detect retransmitted duplicate RPCs. When enabled, checksums are verified before serving a cached reply, preventing stale cache entries from being incorrectly matched.

Performance & I/O

Time To Live – Positive Cached Credential (ms) 3,600,000

How long (in milliseconds) a successfully resolved credential (user/group mapping) is kept in the NFS credential cache. At 3,600,000 ms = 1 hour, group membership changes in LDAP/NIS will not be visible to NFS for up to an hour without a cache flush.

Reduce to 300,000 ms (5 min) in environments where user/group changes need to propagate quickly. Increase for read-heavy, stable environments to reduce name-service load.

Time To Live – Negative Cached Credential (ms) 3,600,000

How long a failed credential lookup (user not found) is cached before being retried. Caching negative results prevents repeatedly hammering a name service for non-existent users. However, if a user is added, they cannot connect for up to 1 hour.

Set lower than the positive TTL (e.g., 60,000 ms = 1 minute) so newly added users are recognized quickly after provisioning.

Time To Live – Cached Entry for Transient Error (ms) 30,000

How long a credential lookup that failed due to a transient error (e.g., name service temporarily unreachable) is cached before being retried. The 30-second value means NFS will retry the name service every 30 seconds during an outage.

Harvest Timeout for Cached Credential (ms) 86,400,000

The maximum time (24 hours) a cached credential can live in the cache before it is unconditionally evicted and must be re-resolved. This is an absolute upper bound regardless of the positive TTL.

I/O Count to Be Grouped as a Session 5,000

The number of I/O operations that are grouped together for session-based analytics and monitoring. After 5,000 I/Os, the session is considered complete for reporting purposes. This parameter affects analytics aggregation, not data path behavior.

Duration for I/O to Be Grouped as a Session (secs) 120

The time window (2 minutes) after which an I/O session is closed for analytics purposes, even if the I/O count threshold has not been reached. Works alongside the I/O count threshold — whichever is hit first closes the session.

Idle Connection Timeout Value (seconds) 360

The number of seconds an NFS TCP connection can remain idle (no RPC traffic) before the server considers closing it. A value of 360 seconds (6 minutes) means idle connections are cleaned up after 6 minutes, freeing server resources.

Are Idle NFS Connections Supported enabled

When enabled, the server actively manages and can close idle NFS TCP connections after the configured timeout. When disabled, connections persist indefinitely until the client closes them. Enabling this frees resources from dormant clients.

Enable Dropping a Connection When an NFSv3 Request is Dropped enabled

When the server must drop an NFSv3 request (e.g., under extreme load), this setting causes the server to also drop the TCP connection, forcing the client to reconnect and reissue the request. This signals the overload condition clearly to the client rather than leaving it waiting indefinitely.

Use 64 Bits for NFSv3 FSIDs and File IDs enabled

Enables 64-bit filesystem IDs and inode numbers for NFSv3. Required for large filesystems or those with more than 2^32 inodes. Modern NFS clients support 64-bit file IDs; very old clients (pre-2.6 Linux kernels, older Solaris) may not.

Use 64 Bits for NFSv4.x FSIDs and File IDs enabled

Same as above but for NFSv4.x clients. All modern NFSv4 clients support 64-bit file IDs; this should always be enabled for production volumes.

Ports & Service Timers

Note: The following ports are fixed by ONTAP and used by the SVM's NFS services. Ensure firewall rules between clients and the SVM's LIF addresses allow these ports. NFSv4 exclusively uses port 2049 and does not require the auxiliary ports below.
NFS Mount Daemon Port 635

The port on which the MOUNT daemon listens. Clients use MOUNT to resolve an export path to a filehandle before performing NFS I/O (NFSv3 only). ONTAP uses port 635 rather than the traditional portmapper-assigned dynamic port, simplifying firewall rules.

Network Lock Manager Port 4045

The port for NLM (Network Lock Manager), which provides advisory file locking for NFSv3 clients (the fcntl/flock equivalent over NFS). NFSv4 and later have locking built into the protocol and do not use NLM.

Network Status Monitor Port 4046

The port for NSM (Network Status Monitor), which works alongside NLM to notify clients of server reboots so they can reclaim their locks. Used only with NFSv3.

NFS Quota Daemon Port 4049

The port for the RQUOTAD service, which allows NFS clients to query disk quota information from the server using the quota command. This requires Rquota Enable to be turned on.

Rquota Enable disabled

Enables the RQUOTAD service that exposes disk quota information to NFS clients. When disabled, quota commands on NFS clients will fail or return no data. Quotas on the volume still function — clients just cannot query them remotely.

Enable if NFS clients need to run quota or repquota commands against this server. Most environments do not require this.

NFS Response Trace Enabled false

When enabled, ONTAP logs details about NFS operations that exceed the response trigger threshold. Useful for diagnosing latency or hung NFS operations. Should normally be disabled in production to avoid log overhead.

NFS Response Trigger (in secs) 60

The threshold in seconds at which an NFS operation is considered "slow" and logged when response tracing is enabled. Operations that take longer than 60 seconds will be traced.

Miscellaneous & Monitoring

Require All NFSv3 Reads to Return Read Attributes false

When enabled, every NFSv3 READ response includes updated file attributes (size, mtime, etc.) in the reply. This guarantees attribute consistency at the cost of extra processing per read. Disabled by default because most clients issue separate GETATTR calls when needed.

Show Change in FSID as NFSv3 Clients Traverse Filesystems enabled

Same behavior as the NFSv4 version but for NFSv3 clients. When traversing a FlexVol volume junction, NFSv3 clients see a change in the FSID, indicating a filesystem boundary. Required for correct df output on mounted sub-trees.

Enable NFSv3 EJUKEBOX Error true

Controls whether ONTAP returns the EJUKEBOX (NFS3ERR_JUKEBOX) error to NFSv3 clients when a request must be retried (e.g., waiting for a migrate or tiering operation). Clients that receive EJUKEBOX are expected to retry after a short delay. Some older clients may not handle this error gracefully.

Hide Snapshot Directory under NFSv3 Mount Point disabled

When disabled, the .snapshot directory is visible at the root of each NFS-mounted volume, allowing clients to browse and restore previous snapshots directly from the filesystem. Enabling this hides the directory from NFS listings (though it can still be accessed if the path is known).

Keep disabled to allow self-service snapshot restores. Enable in regulated environments where snapshot visibility must be restricted.

Force Usage of SpinNp Readdir Requests false

A legacy compatibility option for certain NFS clients that require a specific READDIR implementation. Disabled by default and rarely needed in modern deployments.

RDMA Protocol enabled

Enables NFS over RDMA (Remote Direct Memory Access), also known as NFS/RDMA or RoCE. RDMA allows NFS data to be transferred directly between client and server memory, bypassing the CPU and OS networking stack for dramatically lower latency and CPU overhead.

Requires RDMA-capable NICs (RoCE or iWARP) on both client and server, and an RDMA-aware switch fabric. Delivers significant benefits for latency-sensitive HPC or analytics workloads.

Validation of Qtree IDs for Qtree File Operations enabled

When enabled, ONTAP validates that NFS operations on files within a qtree use a filehandle whose embedded qtree ID matches the actual qtree containing the file. This prevents a class of filehandle-guessing attacks that could allow access across qtree boundaries.

Set Protocol Used for Name Services Lookups for Exports udp

Controls whether ONTAP uses UDP or TCP when communicating with NIS/LDAP name services during export policy client matching. UDP is the traditional NIS transport; TCP is preferred for LDAP and in environments where UDP traffic may be filtered.

Change to tcp if your name service infrastructure uses LDAP over TCP or if UDP is unreliable in your network.

DNS Domain Search Enabled During Netgroup Lookup enabled

When enabled, DNS domain search suffixes are appended to short hostnames during netgroup lookups. This allows netgroup entries with unqualified hostnames (e.g., webserver1) to match even if clients connect using fully qualified names.

Trust No-Match Result from Any Name Service During Netgroup Lookup disabled

When enabled, a definitive "not found" response from any name service (NIS, LDAP, files) during netgroup lookup is treated as authoritative, stopping the lookup chain. When disabled (default), all name service sources are consulted before a final no-match is declared.

Display Maximum NT ACL Permissions to NFS Client disabled

On NTFS or unified security style volumes, NFS clients see a POSIX permission mode derived from the NT ACL. When disabled, ONTAP shows the most conservative interpretation of the NT ACL as the mode bits. When enabled, it shows the most permissive interpretation.

Leave disabled. Showing maximum permissions can mislead NFS applications into attempting access that will ultimately be denied by the NT ACL.

NFSv3 MS-DOS Client Support disabled

Enables compatibility features for very old MS-DOS based NFS clients. Not relevant to any modern deployment.

Allow Non-root User MOUNT Operations from Hadoop Connector enabled

Allows the Hadoop/HDFS connector to initiate NFS MOUNT operations from a non-privileged (non-root) user. The standard NFS Mount Root Only rule requires the MOUNT request to come from a privileged port, which the Hadoop connector may not use.

Leave enabled if using ONTAP with a Hadoop or Spark cluster via the NFS connector. Disable if no Hadoop integration is present.

Qtree QoS Enabled disabled

Enables Quality of Service (QoS) policy enforcement at the qtree level, in addition to the volume level. When enabled, QoS policy groups can be applied to individual qtrees for more granular I/O throttling.

Modebits and ACL Setting is Restricted unrestricted

Controls whether changes to mode bits (chmod) or ACLs are restricted on this SVM. unrestricted means any client with appropriate permissions can change mode bits or ACLs, following standard POSIX rules. restricted would limit these operations.

Is Qtree Extended Performance Monitoring Enabled disabled

When enabled, ONTAP collects detailed per-qtree I/O performance statistics. Useful for chargeback reporting or identifying hot qtrees. Carries a small overhead due to per-qtree counter tracking.

NFSv4 Subnet Filter Support disabled

When enabled, allows export policies to match clients by subnet in NFSv4, in addition to the standard IP/hostname matching already available. Provides finer-grained network-based access control for NFSv4 clients.

Preserves and Modifies NFSv4 ACL (and NTFS File Permissions in Unified Security Style) enabled

When enabled, ONTAP preserves existing NFSv4 ACLs and NTFS ACLs when mode bit changes (chmod) are applied. The ACL is updated to reflect the new mode bits rather than being discarded entirely. When disabled, a chmod replaces the ACL with a minimal POSIX-equivalent ACL.

Keep enabled in mixed-protocol environments where SMB ACLs must not be wiped by NFS chmod operations.

Section 01

Deduplication

Deduplication eliminates redundant data blocks, storing only unique 4KB blocks and replacing duplicates with lightweight metadata pointers. It operates at three distinct levels in ONTAP.

🔍 Zero-block Dedup

Detects and eliminates data blocks filled entirely with zeros. Enabled automatically on all deduplicated volumes. Saves 100% of zero-block space — immediately effective when VMs are first provisioned (empty pages).

⚡ Inline Deduplication

Detects and eliminates duplicate blocks before they are written to disk. No extra I/O for scanning — savings are immediate and the storage never gets wasted in the first place. Default on AFF.

🕐 Background (Post-process) Dedup

Runs as a scheduled job on existing data already written to disk. Scans fingerprints of written blocks and consolidates duplicates. Can be scheduled or triggered manually. Useful for HDD-based systems.

🌐 Cross-Volume Inline Dedup

Finds duplicate blocks across multiple volumes on the same aggregate. Particularly effective in multi-VM environments where OS images share identical OS-level blocks. Default on AFF ONTAP 9.2+.

📦 Cross-Volume Background Dedup

Post-process scan across volumes in an aggregate. Finds duplicates that inline missed. Runs during low-activity windows. Effective for large datasets with high duplicate rates like backup volumes and NFS datastores.

🔁 How Inline Deduplication Works — Block Write Path
1
Client Write
4KB block arrives at ONTAP controller from client
2
Fingerprint
SHA-256 hash computed for the incoming block in memory
3
Lookup
Hash compared against dedup fingerprint database (NVRAM + on-disk)
4
Match → Pointer
Duplicate found: write a metadata pointer, no new block stored

Section 02

Compression

Compression reduces the physical storage required for a volume by encoding data blocks in a mathematically smaller form. ONTAP supports three distinct compression modes, each optimized for different performance and savings trade-offs.

⚡ Inline Compression

Compresses data in memory before writing to disk. Uses 8KB compression groups. Provides immediate savings with minimal latency impact. Default on AFF platforms. If compression ratio would be less than 50%, data is written uncompressed to avoid wasted CPU.

🕐 Post-process Compression

Data is written uncompressed first, then a background job compresses it using larger 32KB groups. Uses more CPU during the scan, but achieves better compression ratios than inline for sequential data. Suitable for FAS/HDD systems where write latency matters more.

🌡️ Adaptive / Secondary Compression

Used by TSSE (Temperature-Sensitive Storage Efficiency). Cold data is recompressed with larger 32KB blocks (vs 8KB for inline) for deeper savings. Hot data uses 8KB for faster decompression. Enabled automatically on AFF C-Series.

📏
8KB vs 32KB Compression Groups
8KB groups (inline): Faster read/write performance because only a small block needs decompression on read. Lower compression ratio but negligible latency impact. 32KB groups (secondary): Better compression ratio (2-3x vs 8KB), but requires reading and decompressing a larger unit. Used for cold/archival data where compression ratio matters more than read speed.

Data Compaction

Data compaction addresses a specific inefficiency: data that is smaller than a full 4KB ONTAP block wastes the remainder of that block. Compaction packs multiple sub-4KB pieces into a single 4KB physical block, reclaiming that wasted space.

🗑️ Without Compaction — Wasted Space
A
1KB Chunk
Stored in a full 4KB block. 3KB wasted.
B
800B Chunk
Stored in a full 4KB block. 3.2KB wasted.
C
512B Chunk
Stored in a full 4KB block. 3.5KB wasted.
✅ With Compaction — Packed Efficiently
1
Pack Together
A (1KB) + B (800B) + C (512B) = 2.3KB → fits in 1 block
2
One 4KB Block
All 3 chunks stored in 1 physical block vs 3 blocks previously
3
66% Savings
2 physical blocks freed. Metadata tracks sub-block locations.
📋
Prerequisites for Compaction
Compaction requires: space-guarantee=none (thin-provisioned volume) AND storage efficiency must be enabled on the volume. It works at the WAFL block level after deduplication and compression, so the savings stack on top of the other two.
# Prerequisite: ensure volume is thin-provisioned
volume modify -vserver svm_prod -volume vol_data \
  -space-guarantee none

# Enable efficiency (if not already)
volume efficiency on -vserver svm_prod -volume vol_data

# Enable data compaction
volume efficiency modify -vserver svm_prod -volume vol_data \
  -data-compaction true

# Enable ALL three (dedup + compression + compaction) together
volume efficiency modify -vserver svm_prod -volume vol_data \
  -dedupe true \
  -inline-dedupe true \
  -compression true \
  -inline-compression true \
  -data-compaction true
# Show compaction savings
volume show -vserver svm_prod -volume vol_data \

Section 04

Thin Provisioning

Thin provisioning allows you to present more logical storage to hosts than you have physically allocated. Space is consumed from the aggregate only when data is actually written — not when the volume or LUN is created.

📦 Thick Provisioning (Traditional)

  • All space reserved upfront at creation
  • 100TB volume = 100TB immediately consumed from aggregate
  • Safe but wasteful — most space sits empty
  • Required for certain latency-sensitive workloads
  • ONTAP: -space-guarantee volume

📏 Thin Provisioning (Recommended)

  • Space consumed only as data is written
  • 100TB volume may use only 20TB of physical space initially
  • Enables over-provisioning: 500TB logical in 100TB physical pool
  • Prerequisite for compression and compaction
  • ONTAP: -space-guarantee none
# Create a thin-provisioned volume
volume create -vserver svm_prod -volume vol_thin \
  -aggregate aggr1 \
  -size 10t \
  -space-guarantee none       # thin — no space reserved

# Convert existing thick volume to thin
volume modify -vserver svm_prod -volume vol_old \
  -space-guarantee none

# Check volume guarantee type
volume show -vserver svm_prod \
  -fields space-guarantee,size,available
# Create a thin-provisioned LUN (space-reservation=disabled)
lun create -vserver svm_san \
  -path /vol/vol_san/lun_db \
  -size 2t \
  -ostype linux \
  -space-reserve disabled   # thin LUN

# Enable T10 SCSI thin provisioning (host-visible reclamation)
lun modify -vserver svm_san \
  -path /vol/vol_san/lun_db \
  -space-allocation enabled

# Show LUN space reservation status
lun show -vserver svm_san \
  -fields size,space-reserve,space-allocation
# Monitor space usage — critical with thin provisioning
volume show-space -vserver svm_prod -volume vol_thin

# Show aggregate utilization (thin provisioning can over-commit)
storage aggregate show \
  -fields name,availsize,usedsize,percent-used

# Show logical space vs physical space (over-commitment ratio)
volume show -vserver svm_prod \
  -fields size,used,logical-used,percent-used

# Set volume space alerts (warn at 80%, autodelete snapshots at 90%)
volume modify -vserver svm_prod -volume vol_thin \
  -space-mgmt-try-first snap_delete \
  -percent-snapshot-space 20
# Enable autogrow — volume expands automatically when space is low
volume autosize -vserver svm_prod -volume vol_thin \
  -mode grow_shrink \
  -minimum-size 500g \
  -maximum-size 10t \
  -grow-threshold-percent 85 \
  -shrink-threshold-percent 50 \
  -increment-size 100g

# Check autosize configuration
volume autosize show -vserver svm_prod -volume vol_thin
Section 05

Snapshot Copies — Space-Efficient Protection

NetApp Snapshot copies are read-only, point-in-time images of volumes. Unlike traditional backup copies, Snapshots use a Redirect-on-Write (ROW) mechanism — they consume near-zero space at creation and only grow as the original data changes.

📸 Redirect-on-Write — How Snapshots Save Space
1
Snapshot Taken
ONTAP creates a pointer map to all current blocks. No data copied.
2
New Write
Active file system data changes. Old block must be preserved.
3
Block Redirect
New data written to a new block. Snapshot pointer still holds old block.
4
Shared Blocks
Unchanged blocks are shared between Snapshot and active. Zero extra space.
5
Efficient Result
Snapshot only grows by the blocks that actually changed. 255 Snaps max.
# Create a manual snapshot
volume snapshot create -vserver svm_prod -volume vol_data \
  -snapshot snap_pre_change_20250613

# List all snapshots on a volume
volume snapshot show -vserver svm_prod -volume vol_data

# Show snapshot sizes and cumulative percentages
volume snapshot show -vserver svm_prod -volume vol_data \
  -fields size,cumulative-total,cumulative-percentage-of-total-size

# Delete a specific snapshot
volume snapshot delete -vserver svm_prod -volume vol_data \
  -snapshot snap_pre_change_20250613
# Show existing snapshot policies
volume snapshot policy show

# Create a custom snapshot policy
volume snapshot policy create \
  -vserver svm_prod \
  -policy prod_policy \
  -enabled true \
  -schedules hourly,daily,weekly \
  -count 6,2,1

# Assign a snapshot policy to a volume
volume modify -vserver svm_prod -volume vol_data \
  -snapshot-policy prod_policy

# View snapshot reserve (space set aside for snapshots)
volume show -vserver svm_prod -volume vol_data \
  -fields percent-snapshot-space,snapshot-space-used
# Reduce snapshot reserve (default 5% — often unnecessary with ROW)
volume modify -vserver svm_prod -volume vol_data \
  -percent-snapshot-space 1

# Enable automatic snapshot deletion when volume fills
volume snapshot autodelete modify -vserver svm_prod \
  -volume vol_data \
  -enabled true \
  -commitment destroy \
  -trigger volume \
  -target-free-space 20

# Show snapshot efficiency savings across all SVMs
storage aggregate show-efficiency -details
  >> Snapshot Volume Storage Efficiency: 8.81:1
# Restore entire volume to a snapshot (revert)
volume snapshot restore -vserver svm_prod -volume vol_data \
  -snapshot snap_pre_change_20250613

# Restore a SINGLE FILE from snapshot (via .snapshot directory)
# From a client: navigate to /vol_data/.snapshot/snap_name/path/to/file
# OR use ONTAP single-file restore:
volume snapshot restore-file \
  -vserver svm_prod -volume vol_data \
  -snapshot snap_pre_change_20250613 \
  -path /reports/q2_report.xlsx \
  -restore-path /reports/q2_report_recovered.xlsx

FlexClone — Instant Zero-Space Clones

FlexClone creates an instantaneous, writable clone of a volume, LUN, or file from a Snapshot. The clone shares all existing blocks with the parent — consuming near-zero space at creation time, using space only as it diverges.

🧬
FlexClone Volume
A full volume clone. Perfect for dev/test environments, parallel builds, or staging. A 50TB volume cloned instantly — the clone uses 0 extra space at creation. Developers can work on it independently, writing only deltas.
📄
FlexClone LUN / File
Clone individual LUNs or files within a volume. Used heavily by Virtual Desktop Infrastructure (VDI) to provision 1000s of desktop VMs from a golden image — each desktop shares the golden image blocks until they diverge.
# Create a FlexClone volume from a snapshot (instant — seconds)
volume clone create \
  -vserver svm_prod \
  -flexclone vol_dev_clone \
  -parent-volume vol_prod \
  -parent-snapshot snap_daily_20250613 \
  -type RW

# Create FlexClone from current state (auto-creates a snap)
volume clone create \
  -vserver svm_prod \
  -flexclone vol_test_clone \
  -parent-volume vol_prod

# List all FlexClone volumes and their parent
volume clone show

# Show space used by clone vs parent (blocks shared)
volume show -vserver svm_prod -volume vol_dev_clone \
  -fields size,used,is-flexclone,flexclone-parent-volume
# Clone a LUN from a snapshot (e.g. golden VDI image)
lun copy start \
  -vserver svm_vdi \
  -source-path /vol/golden_vol/.snapshot/golden_snap/golden.vmdk \
  -destination-path /vol/desktop_vol/desktop001.vmdk

# Create FlexClone LUN directly (faster, uses block sharing)
lun copy start \
  -vserver svm_san \

volume file clone create \
  -vserver svm_prod \
  -volume vol_data \

  -destination-path /vms/vm001.vmdk

# Clone file from specific snapshot

FabricPool — Intelligent Data Tiering

FabricPool automatically moves cold (infrequently accessed) data blocks from expensive performance-tier SSDs to low-cost object storage — on-premises (StorageGRID, ONTAP S3) or cloud (AWS S3, Azure Blob, GCP). Hot data stays on flash. Metadata always remains on the local tier.

Section 08

Temperature-Sensitive Storage Efficiency (TSSE)

TSSE is an intelligent, adaptive compression system introduced in ONTAP 9.8. It monitors how frequently each data block is accessed (its "temperature") and applies different compression algorithms based on heat — small fast-decompress blocks for hot data, larger higher-ratio compression for cold data.

🔥 Hot Data — 8KB Compression

  • Frequently accessed blocks (last 14 days by default)
  • Compressed in 8KB groups — fast decompression
  • Lower compression ratio but minimal read latency impact
  • Primary goal: performance maintained

❄️ Cold Data — 32KB Compression

  • Infrequently accessed blocks (older than cooling threshold)
  • Recompressed in 32KB groups — 2–3x better ratio than 8KB
  • Slightly higher read latency acceptable for cold data
  • Primary goal: maximum space savings
🌡️
Two TSSE Storage Efficiency Modes
default mode: Standard ONTAP efficiency. Inline dedup, 8K compression, compaction. TSSE temperature scanning enabled but secondary (32K) compression only applied to colder blocks.  |  efficient mode: Full TSSE with adaptive compression. Both 8K (hot) and 32K (cold) compression applied based on block temperature. Maximum space savings. Recommended for AFF C-Series and general storage efficiency optimization.
# Check current storage efficiency mode on a volume
volume show -vserver svm_prod -volume vol_data \
  -fields storage-efficiency-mode

# Enable TSSE 'efficient' mode on a volume
volume modify -vserver svm_prod -volume vol_data \
  -storage-efficiency-mode efficient

# Create new volume with TSSE efficient mode from the start
volume create -vserver svm_prod -volume vol_tsse \
  -aggregate aggr_aff_01 \
  -size 10t \
  -storage-efficiency-mode efficient \
  -space-guarantee none

# Note: Volumes created BEFORE ONTAP 9.8 do not get TSSE automatically.
# Migrate old volumes to a new volume to gain TSSE benefits:
volume move start \
  -vserver svm_prod -volume vol_legacy \
  -destination-aggregate aggr_aff_01

# AFF A-Series (e.g. A400, A800) — ONTAP 9.8+

Best Practices & Workload Guide

Use this reference to quickly determine which efficiency features to enable for each workload type, and follow the operational best practices to maintain healthy, efficient storage.

1
Verify volume space-guarantee before enabling compression / compaction
Both features require space-guarantee=none
▶
Why it matters

Compression and compaction store data in variable-size physical blocks. This is incompatible with thick (space-guaranteed) volumes where block sizes are fixed at creation. Attempting to enable compression on a thick volume will fail or have no effect.

Fix
volume modify -vserver svm_prod -volume vol_data -space-guarantee none
# Then enable compression:
volume efficiency modify -vserver svm_prod -volume vol_data -compression true
2
Monitor aggregate free space when thin provisioning heavily
Over-commitment is a feature — managed over-commitment is a risk
▶
Why it matters

Thin provisioning lets you present 500TB of volume space from a 100TB aggregate. If all those volumes fill up simultaneously, the aggregate runs out of space and writes fail. Set threshold alerts at 75–80%.

Setup
# Set an EMS alert when aggregate hits 80%
event notification destination create -name capacity_alert -email admin@company.com
storage aggregate show -fields name,percent-used,availsize
3
Run efficiency jobs on existing data after enabling features
Inline features only apply to new data — scan-old-data catches everything
▶
Why it matters

When you enable inline deduplication or compression on an existing volume, only data written after that point benefits inline. Existing data on disk is not retroactively processed. Use -scan-old-data true to run a full pass.

volume efficiency start -vserver svm_prod -volume vol_data \
  -scan-old-data true -compression true -dedupe true
4
Use NAE (Aggregate Encryption) instead of NVE when dedup savings matter
Cross-volume dedup cannot work across NVE volumes with different keys
▶
Why it matters

With NVE, each volume is encrypted with a unique key. Two identical blocks in different volumes encrypt to different ciphertext — deduplication cannot find a match. NAE encrypts at the aggregate level with one shared key, so deduplicated blocks remain identical after encryption.

# Enable NAE on an aggregate
storage aggregate modify -aggregate aggr_data -encrypt-with-aggr-key true
5
Don't enable post-process compression on high-IOPS random-write volumes
Background compression scanner can contend with production I/O
▶
Why it matters

Post-process compression scans existing data and re-writes it compressed. On busy OLTP volumes with 50K+ IOPS, this background work can increase latency by 5–15% during the scan window. Either use inline-only, or schedule the job during a low-activity maintenance window.

# Schedule compression to run only at low-traffic times (2am–6am)
volume efficiency modify -vserver svm_prod -volume vol_oltp \
  -schedule "sun-sat@2"
# OR use inline only, disable post-process:
volume efficiency modify -vserver svm_prod -volume vol_oltp \
  -inline-compression true -compression false
6
Use FabricPool 'all' policy on SnapVault destinations, not primary volumes
The 'all' policy marks data cold immediately — not suitable for active data
▶
Why it matters

The all tiering policy tiers data to the cloud tier as fast as possible. On a production volume, this means read performance will suffer because data must be recalled from object storage. On backup destinations, it's ideal — backup data is rarely re-read and should be cheap to store.

# SnapVault/backup destination: use 'all'
volume modify -vserver svm_backup -volume vol_vault_dst -tiering-policy all
# Production volumes: use 'auto' with appropriate cooling days
volume modify -vserver svm_prod -volume vol_data -tiering-policy auto
✅
Golden Rule: Start Efficient, Stay Efficient
Enable all efficiency features at volume creation time, especially on AFF platforms. Retrofitting efficiency to existing volumes requires scanning all data — expensive and time-consuming. On AFF in ONTAP 9.15.1+, all features are already on by default. Always verify with volume efficiency show -instance and storage aggregate show-efficiency -details to confirm savings are accumulating as expected.

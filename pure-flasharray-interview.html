Architecture & Concepts

RAID-HA, DirectFlash, Pods, and storage constructs
Q01

What is a DirectFlash Module (DFM), and why does Pure manage flash directly instead of using off-the-shelf SSDs?

▾

DFMs are Pure's custom NAND modules where Purity's software directly controls flash translation, wear-leveling, and garbage collection — bypassing any onboard SSD controller firmware.

Standard SSDs have their own Flash Translation Layer (FTL) that manages these tasks internally, opaquely, and only for that one drive. By removing that layer and letting Purity manage the full pool globally, Pure gains:

⚡
Global wear-leveling — Purity can balance writes across all DFMs in the array, not just within a single module, extending media lifespan.
🔍
Media health visibility — Purity knows the exact health state of every flash cell, enabling proactive failure prediction.
📦
Space efficiency — No duplicate metadata. One layer handles what would otherwise require both an SSD's FTL metadata and Purity's own data-placement metadata.

Q02

What is RAID-HA and why doesn't an administrator configure RAID group sizes on FlashArray?

▾

RAID-HA is Pure's proprietary erasure-coding scheme designed specifically for flash failure characteristics. Unlike traditional RAID, there is only one global storage pool — all DFMs participate in a single RAID-HA domain.

  Traditional RAID                      FlashArray RAID-HA
  ─────────────────                     ──────────────────────────
  [ RAID Group 1  ] ← admin-defined     [ Global Pool              ]
  [ RAID Group 2  ] ← admin-defined       All DFMs — one domain    
  [ RAID Group 3  ] ← admin-defined       Parity: Purity-managed   
  Group size = admin risk/capacity tradeoff    No group to configure

Purity automatically determines parity layout, group sizing, and rebuild behavior. The administrator's job is to monitor capacity and health — not to design RAID topology.

Q03

Explain the difference between a Volume and a Pod on FlashArray.

▾
Construct What it is Primary use
Volume A single block storage object presented to a host — the LUN equivalent Host connectivity, I/O
Pod A consistency-group wrapper that bundles one or more volumes (and/or file systems) and replicates them as a crash-consistent unit ActiveCluster & ActiveDR replication

Think of it this way: a Volume is the storage object a host actually writes to; a Pod is the grouping construct that ensures those writes replicate together with ordering guarantees across arrays.

Host connectivity, protection groups, SafeMode
Q04

How do you connect a new host to a FlashArray and present a volume?

▾

The process follows three steps regardless of protocol (iSCSI or FC):

# 1. Register the host's initiator
purehost create myhost --iqnlist iqn.2023-01.com.example:host01
# or for FC:
purehost create myhost --wwnlist 21:00:00:24:ff:ab:cd:ef

# 2. Create the volume
purevol create myvolume --size 1T

# 3. Connect volume to host
purehost connect myhost --vol myvolume
💡
For an ESXi cluster or any group sharing the same volumes, create a Host Group first (purehgroup create) and connect the volume to the group — this avoids connecting to each host individually.

Q06

What is SafeMode and why is it important?

▾

SafeMode creates immutable, time-locked snapshots that cannot be deleted or modified by anyone — including a full array administrator — until the retention window expires.

🔒
Ransomware scenario: An attacker compromises admin credentials and attempts to delete all snapshots before encrypting production data. With SafeMode enabled, the snapshot copies are untouchable. The array itself enforces the retention period — no credentials override it.

Enabling or disabling SafeMode requires involvement from Pure Storage Support (out-of-band verification), making it a genuine air gap for the snapshot chain even when the array is fully compromised.

Q07

Walk through the CLI commands to create a Protection Group, add volumes to it, and configure a snapshot schedule.

▾ 
# Create the protection group
purepgroup create pg-proddb

# Add volumes
purepgroup add pg-proddb --vollist vol-db01,vol-db02

# Set snapshot schedule: every 4 hours, keep 24 snapshots locally
purepgroup schedule pg-proddb \
  --snap-enabled true \
  --snap-frequency 14400 \
  --snap-at 0 \
  --snap-per 24

# Add a replication target and enable replication schedule
purepgroup add pg-proddb --targetlist remote-array
purepgroup schedule pg-proddb \
  --replicate-enabled true \
  --replicate-frequency 3600

Networking & Connectivity

MTU, jumbo frames, FC zoning, protocol differences
Q08

Why is end-to-end jumbo frame (MTU 9000) verification critical for NVMe/TCP or iSCSI, and how do you test it?

▾

iSCSI and NVMe/TCP set the Don't Fragment (DF) bit on large packets. If any hop in the path has a lower MTU, packets are dropped silently — causing connection failures or severe throughput degradation, not an obvious error.

  Host NIC       Switch Port A     Switch Port B     Array Port
  MTU 9000  ──►  MTU 1500  ──►   MTU 9000  ──►   MTU 9000
                      ↑
              Drop point — large frames discarded here

Test with a large ICMP packet and the DF flag set. The payload size is MTU minus headers (9000 − 28 = 8972 bytes):

# Linux — test each hop: switch management IP, array data IP
ping -M do -s 8972 10.10.1.50

# Windows
ping -f -l 8972 10.10.1.50
⚠️
Check every hop: host NIC settings, each switch port (trunk and access), and the array's network interface. A single misconfigured switch port kills large-frame throughput across the entire path.

Q10

How do you verify that FC zoning is configured correctly for a new host connection?

▾ 
# On a Brocade switch — show zones containing a WWPN
zoneshow

# Verify the host HBA WWPN is in the same zone as the array port WWPN
# Each zone should contain: 1 initiator (host HBA) + 1 or more targets (array ports)

# On the host (Linux) — confirm HBA sees the target
systool -c fc_host -v | grep port_name
cat /sys/class/fc_host/host*/port_state

# Check if the array target is visible as a remote port
ls /sys/class/fc_remote_ports/
💡
Best practice: use single-initiator zoning — one host HBA WWPN per zone, paired with all required array target ports. This prevents fabric storms if a HBA misbehaves.

Troubleshooting

Volume visibility, latency spikes, performance investigation
Q11

A host can't see a newly mapped volume even though the connection shows correctly in Purity. What do you check?

▾

Follow this checklist in order:

# 1. Rescan the host bus (Linux)
rescan-scsi-bus.sh -a

# 2. Or trigger a manual rescan per HBA
echo "- - -" > /sys/class/scsi_host/host0/scan

# 3. VMware — rescan via CLI
esxcli storage core adapter rescan --all

# 4. Verify multipath sees the new device
multipath -ll

If a rescan doesn't reveal the device:

🔍
Confirm the volume was connected to the correct Host object in Purity — not a different host with a similar name. Also check FC zoning includes the right array port WWPNs for this host's HBA.

Q13

The array is showing a drive failure. What does the recovery process look like?

▾

FlashArray handles most drive recovery automatically. Here's what happens and what you do:

# Check hardware status
puredrive list

# View current RAID-HA rebuild status
purearray list --space

# If a drive is marked 'failed', verify it's physical and not a transient error
puredrive list --verbose
🔄
Purity begins rebuilding RAID-HA parity automatically when a DFM fails — no admin action needed to initiate rebuild. Your job is to monitor rebuild progress, open a Pure support case, and schedule the field-replaceable unit (FRU) swap. Pure proactively monitors via Pure1 and may open the case before you notice.

Q14

How would you identify which volume or host is causing high overall array latency (noisy neighbor)?

▾ 
# List volumes sorted by IOPS (snapshot of current state)
purevol list --performance --sort iops

# List volumes sorted by throughput
purevol list --performance --sort output_per_sec

# Check per-host performance
purehost list --performance

In Pure1, use the Workload Analyzer to view IOPS and bandwidth heat maps across all volumes over time — this quickly surfaces any volume that spiked in the relevant window.

⚡
Once you identify the noisy volume, check whether QoS bandwidth limits are configured (purevol set --bandwidth-limit) — this is FlashArray's primary tool for throttling a runaway workload without impacting other tenants.

NetApp Cloud Volumes ONTAP on Azure

A working reference covering CVO's licensing, storage, networking, and CIFS/AD model on Azure — built around how each concept maps to (or diverges from) on-prem ONTAP.

NetApp ONTAP Microsoft Azure Interview / Concept Prep
01 — Foundations

What is Cloud Volumes ONTAP?

CVO is NetApp's software-defined ONTAP, packaged to run as a virtual machine instance in a public cloud — AWS, Azure, or GCP. It delivers the same data management capabilities you'd get on-prem (Snapshot, SnapMirror, FlexClone, deduplication, compression, multiprotocol NAS/SAN), running on cloud compute and disk resources instead of physical controllers and shelves.

Deployment and lifecycle management happen through BlueXP (formerly Cloud Manager) — a SaaS console that orchestrates CVO instances across cloud accounts.

01 — Foundations

Working Environment & Working Environment ID

A Working Environment in BlueXP represents a single CVO deployment (or a discovered on-prem ONTAP cluster). Each one gets a unique Working Environment ID — the identifier BlueXP and its APIs use for all subsequent operations on that system.

  • Calling the BlueXP REST API for automation
  • Setting up SnapMirror between two CVO instances — each side has its own ID
  • Referencing a specific deployment in support cases or troubleshooting
01 — Foundations

Licensing models: PAYGO vs BYOL

Three independent "meters" run at the same time for any CVO deployment. It's easy to conflate them — keep them separate:

WhatWho bills itBased on
Azure VM + Managed Disks Azure (standard infrastructure billing) Compute size & disk capacity, hourly
ONTAP software entitlement NetApp (BYOL) or Azure Marketplace (PAYGO) Total provisioned capacity (TiB)
SnapMirror copy traffic (on-prem → CVO) Not billed by NetApp licensing; possible Azure egress Data volume crossing regions

PAYGO

An hourly pay-as-you-go subscription via Azure Marketplace. The ONTAP software entitlement is billed alongside — but separate from — your VM and disk charges. No upfront purchase from NetApp.

BYOL

A capacity license (e.g. 20 TiB) purchased directly from NetApp/reseller ahead of time, installed on the CVO system. It covers your provisioned capacity instead of marketplace charging for it — Azure VM/disk costs still apply regardless. Usage is charged against the license first; any overage spills into PAYGO rates.

Current restriction

NetApp has restricted the purchase, extension, and renewal of BYOL licensing — new deployments are increasingly steered toward marketplace subscription models (PAYGO / Annual) rather than traditional BYOL.

Does creating a volume itself cost extra?

No single "create volume" event is billed. A new volume simply consumes against your total provisioned capacity, which is what the ONTAP license (PAYGO or BYOL) is metered against. SnapMirror destination (DP) volumes count the same way — they consume capacity on whichever CVO system hosts them, often at a different package rate (e.g. "Essentials Secondary HA" for DP/read-only volumes on an HA-capable system).

Important: replication I/O itself (SnapMirror, SnapVault, NDMP) is not separately charged by NetApp's licensing — only the provisioned capacity of the volumes involved matters.

02 — Storage

Managed disks, aggregates & volumes

You pay Azure for the VM instance and its attached Managed Disks. ONTAP then carves those disks into aggregates, and volumes are created inside those aggregates — same hierarchy as on-prem (disk → aggregate → volume).

DetailSingle-nodeHA pair
Disk type Azure Managed Disks (typically Premium SSD) Premium SSD Shared Managed Disks (pre-9.12.1: Premium page blobs)
Disks per aggregate Up to 12 disks, all the same type and size
Disk visibility Attached only to its own node Shared — each HA node has access to the other node's storage

After deployment, CVO automatically creates a small root aggregate (aggr0) per node for system volumes/configuration (not for user data), plus a data aggregate (aggr1) where NFS/SMB volumes are provisioned.

Disk size affects more than capacity: it determines the maximum volume size in that aggregate, total CVO capacity, and performance — Azure Premium Storage throughput scales with disk size.

02 — Storage

RAID0 — and why no RAID-DP

The RAID type for every CVO aggregate, on every cloud, is RAID0 (striping). No RAID-DP, no RAID-TEC, no parity disks, no hot spares — RAID0 doesn't support hot spares because there's nothing to reconstruct.

On-prem ONTAP

  • Physical disks can fail outright
  • RAID-DP / RAID-TEC provide parity protection
  • ONTAP owns disk-failure recovery (reconstruction)

CVO on Azure

  • A "Managed Disk" is a durable, replicated service — not one physical drive
  • RAID0 stripes across disks for capacity/performance only
  • Azure's storage layer owns durability beneath the disk abstraction

Doing RAID-DP on top of Azure's own replication would mean paying for redundancy twice — Azure's replication and ONTAP's parity overhead — for no real benefit. So ONTAP's role shifts from disk-failure protection to pure aggregation/striping.

02 — Storage

If a disk fails, what actually happens?

On-prem, a "disk" is one physical drive — a single point of failure. In Azure, a Managed Disk is a virtual disk backed by Azure's storage service, which already keeps multiple copies of your data across different physical hardware (and sometimes different datacenters, depending on the redundancy option — LRS vs ZRS).

Physical drive holding a data copy dies
→
Azure detects it (it already has 2–3 copies)
→
Azure serves I/O from surviving copies
→
Azure rebuilds a fresh copy in the background

Your VM/CVO never sees any of this — the Managed Disk, from ONTAP's point of view, never goes offline or degrades. The "copy to another disk, replace the failed one" behavior absolutely happens — just one layer below anything ONTAP can observe.

Where redundancy planning actually shifts to

Since disk-level failure is essentially Azure's problem, the resilience questions that matter for CVO move up a layer: the VM/node crashing (→ HA pairs + Mediator), and an entire Availability Zone or region going down (→ multi-AZ HA, or DR via SnapMirror to another region).

03 — Resilience

VM crashes, HA pairs & the Mediator

Single-node CVO

If the underlying Azure VM crashes, Azure's infrastructure typically restarts/migrates the VM onto healthy hardware — Managed Disks remain attached throughout, since they're network-attached and decoupled from the VM lifecycle. But there's downtime during the restart, and no automatic failover to a different VM — it's the same VM coming back, not a new one taking over.

HA CVO — the real failover story

  • Two CVO nodes run in different Availability Zones, each with its own set of Managed Disks, mirrored between nodes (SyncMirror-style)
  • A small Mediator VM monitors both nodes' health and arbitrates failover decisions — analogous to "epsilon" / quorum tie-breaking on-prem
  • If Node A crashes, the Mediator detects it and triggers automatic failover — Node B takes over using its own already-synced copy of the data; LIFs/IPs migrate to Node B
  • When Node A recovers, it rejoins the HA pair and resyncs
What HA does not do

It is not "Azure detects a crash, spins up a brand-new VM, then attaches the old disks to it." That would involve redeployment and significant downtime. Failover goes to an already-running second node with its own storage — not disk reattachment to a fresh instance.

Note for interviews: Azure's general VM-recovery behavior for single-node systems is a platform characteristic, not a CVO feature or guaranteed SLA — don't present it as one.

04 — Networking

ExpressRoute & hybrid connectivity

Azure ExpressRoute is a dedicated, private connection between an on-prem datacenter and Azure, bypassing the public internet. For CVO, it's commonly used to:

  • Provide low-latency, high-bandwidth connectivity for SnapMirror replication from on-prem ONTAP to CVO (hybrid DR)
  • Give on-prem clients/applications access to CVO NFS/CIFS shares without traversing the public internet
  • Meet compliance requirements that prohibit storage traffic over public networks

ExpressRoute circuits come in bandwidth tiers (50 Mbps – 10 Gbps+); CVO's network interfaces need to sit in a VNet peered/connected to the ExpressRoute gateway. A site-to-site VPN is the alternative when ExpressRoute isn't available — both ultimately serve the same purpose: giving CVO a private path back to on-prem AD, DNS, and replication sources.

04 — Networking

Deployment flow — where to start

  1. Azure subscription Your billing/account boundary — everything else lives inside it.
  2. Resource Group A logical container for all CVO-related resources: VM, disks, NICs, load balancer, etc. Makes cleanup and management easier.
  3. VNet + Subnet(s) The network where CVO's NICs live. Needs connectivity (peering, VPN, or ExpressRoute) to wherever your on-prem AD, DNS, and clients are.
  4. BlueXP Connector A small VM deployed first into your subscription/VNet — the orchestration agent BlueXP SaaS uses to provision CVO into your environment.
  5. Deploy CVO (via BlueXP or Azure Marketplace) Choose single-node or HA, VM size, disk type/size for the initial aggregate, and licensing model (PAYGO/BYOL).
  6. Networking auto-provisioning BlueXP creates required NICs and private IPs, a Network Security Group, and — for HA — an Internal Load Balancer.
  7. Post-deployment configuration Create/extend the data SVM, set up CIFS/NFS, create volumes, configure AD/DNS, and add any missing SVM routes.
Bundled at deploy time

The marketplace deployment installs licenses for NFS, CIFS (SMB), iSCSI, Autonomous Ransomware Protection (ARP), SnapLock, and SnapMirror automatically — protocol licensing isn't a separate manual step.

04 — Networking

IP addressing, LIFs & the load balancer

Azure CVO doesn't use ifgrp (that's a physical-port bundling concept). The closer analogy is LIFs — logical interfaces still assigned to roles, just mapped onto Azure NICs and Load Balancer frontends instead of physical ports.

What gets allocated

  • The Console automatically allocates the required private IPs — the count depends on single-node vs HA
  • IPs are assigned across 4 NICs per node during deployment
  • An SVM management LIF is created automatically on HA pairs, but not on single-node systems

HA floating IPs

Cluster management1 floating IP
NFS/CIFS data — Node 11 floating IP
NFS/CIFS data — Node 21 floating IP
SVM management (optional)1 floating IP

4 frontend IPs (FIPs) live in the Azure Internal Load Balancer and are configured to migrate on failover events. The per-NIC IPs themselves do not migrate on failover.

Manual step often missed

For each new LIF you create after initial deployment, you must manually add a frontend IP configuration, health probe, and load-balancing rule to the ILB. BlueXP automates this for the LIFs created at deployment time — not for ones added afterward.

What the IPs look like

Standard private IPs from your VNet's subnet range (e.g. 10.x.x.x) — no different in format from any other Azure private IP. The "type" (mgmt / data / SVM-mgmt / cluster) is a role ONTAP and BlueXP assign, not a different IP format.

Speed / throughput factors

  • The VM SKU — larger instance types have higher network bandwidth caps
  • The disk type/size backing the aggregate — Premium Storage performance scales with disk size
  • Whether traffic flows through the ILB (HA) — negligible overhead, but an architectural dependency

There's no on-prem-style "set MTU to 9000" tuning exposed the same way — Azure VNets typically run at 1500 MTU by default; this is mostly an Azure platform setting, not something configured inside ONTAP.

04 — Networking

Routes & routing

A route tells an SVM "to reach network X, send traffic via gateway Y" — same concept as on-prem. Each SVM has its own routing table; without a route, its LIFs can only reach hosts in their own subnet.

The Azure twist: the gateway is the fabric itself

In Azure, the first usable IP of every subnet is always reserved as the default gateway, implemented by Azure's software-defined network (SDN). This gateway is not a VM, not visible, and not configurable — it's Azure's internal routing fabric.

172.18.10.0network address
172.18.10.1Azure default gateway (SDN)
172.18.10.2 – .3reserved, Azure internal use
172.18.10.4+usable for VMs / NICs

Why you need this for a new SVM

After creating a management LIF on an SVM, it's not reachable automatically — not even from within the same subnet, and the same is true for LIFs in general on CVO nodes. Before going further, verify whether a default route exists for the new SVM; if missing, create one so the SVM can reach external services like DNS and Active Directory — required for CIFS/SMB. 

network route show -vserver svm_matrix_cifs

network route create -vserver svm_matrix_cifs \
    -destination 0.0.0.0/0 \
    -gateway 172.18.10.1

Same network route syntax as on-prem — the only conceptual shift is that the "next hop" is an abstraction Azure guarantees exists at a fixed address, rather than physical infrastructure someone provisioned.

On-prem

  • Gateway = a real router's IP on the VLAN
  • Route configured once, applies broadly
  • Router is hardware your network team manages

CVO on Azure

  • Gateway = Azure's invisible SDN router at the subnet's .1
  • Each SVM needs its own default route explicitly created
  • "Router" is the platform — always present, never configured
Consequence of a missing route

Without it, an SVM can serve clients in its own subnet fine, but cannot reach AD/DNS/NTP in a different subnet, peered VNet, or on-prem (via VPN/ExpressRoute) — so CIFS server creation and Kerberos authentication fail, and SnapMirror to peers elsewhere won't work either.

04 — Networking

CIFS & Active Directory — AD doesn't have to live in Azure

CIFS still requires AD, but the domain controllers don't need to physically run in Azure. Three common options:

  1. Extend on-prem AD to Azure Deploy AD domain controller VMs in your Azure VNet (or a peered hub VNet), replicating from your existing on-prem domain.
  2. Connect to on-prem AD over VPN/ExpressRoute CVO's SVM just needs network reachability and DNS resolution to your existing on-prem domain controllers — no AD infrastructure needs to exist in Azure at all.
  3. Azure AD Domain Services A managed, AD-compatible domain Microsoft hosts — useful if you don't want to manage DC VMs yourself (verify current availability/roadmap, as Microsoft has been evolving this offering).

Whichever path you choose, you must set up DNS and Active Directory reachable from Azure — the DNS server must provide name resolution for the AD environment. And as covered above, if no default route exists for the SVM, one must be created so it can reach DNS servers and domain controllers.

Once connectivity exists

CIFS server creation and share access on CVO work exactly like on-prem — same commands, same model: 

vserver cifs create -vserver svm1 -cifs-server FILESRV01 -domain example.com

vserver cifs share access-control create -vserver svm1 -share apps \
    -user-or-group "EXAMPLE\Domain Users" -permission Change

ONTAP doesn't know or care that the domain controllers are remote — as long as DNS, Kerberos, and LDAP traffic can reach them, share-level ACEs and underlying NTFS permissions combine the same way they do on a physical cluster.

<section class="section" id="s1">
  <div class="section-header">
    <div class="section-icon icon-teal">🌐</div>
    <div class="section-title-wrap">
      <div class="section-num">Section 01 · Networking Foundation</div>
      <h2>IP Subnet Management</h2>
    </div>
  </div>

  <p>Managing network interfaces in ONTAP usually involves manually specifying IP addresses, subnet masks, and gateways each time. A smarter approach is to define an <strong>IP Subnet</strong> — a named pool of reserved addresses tied to a specific broadcast domain. Once a subnet exists, ONTAP can automatically assign addresses from the pool when creating LIFs, eliminating repetitive manual entry.</p>

  <div id="s1-sub1"></div>
  <h3>Creating a Named Subnet</h3>
  <p>A subnet is associated with exactly one broadcast domain. You cannot assign the same subnet to multiple broadcast domains. The subnet holds a range of IPs (e.g. 10.50.10.101–10.50.10.109) along with the prefix length and default gateway.</p>

  <div class="code-block">
    <div class="code-header">
      <div class="code-dots"><span class="dot-red"></span><span class="dot-ylw"></span><span class="dot-grn"></span></div>
      <span class="code-lang">ONTAP CLI — Create Subnet</span>
    </div>
    <div class="code-body"><span class="comment"># Create a named subnet for data LIFs</span>
<span class="prompt">::></span> <span class="cmd">network subnet create</span> \
      -subnet-name <span class="value">prod-data-pool</span> \
      -broadcast-domain <span class="value">Default</span> \
      -subnet <span class="value">10.50.10.0/24</span> \
      -ip-ranges <span class="value">10.50.10.101-10.50.10.109</span> \
      -gateway <span class="value">10.50.10.1</span>

<span class="comment"># Verify subnet was created</span>
<span class="prompt">::></span> <span class="cmd">network subnet show</span>
<span class="output">Subnet Name     Broadcast Domain   Subnet              Gateway         Available  IP Range
prod-data-pool  Default            10.50.10.0/24       10.50.10.1      9          10.50.10.101-10.50.10.109</span>

<span class="comment"># Delete an existing subnet if needed</span>
<span class="prompt">::></span> <span class="cmd">network subnet delete</span> -subnet-name <span class="value">old-subnet</span></div>
  </div>

  <div class="callout teal">
    <span class="callout-icon">💡</span>
    <div class="callout-content">
      <div class="callout-title">Design Tip</div>
      <div class="callout-text">Name your subnets by purpose (e.g., <code>prod-data-pool</code>, <code>mgmt-pool</code>, <code>ic-pool</code>). This makes it immediately obvious which IP range serves which function — especially useful in large environments with multiple VLANs and broadcast domains.</div>
    </div>
  </div>

  <div id="s1-sub2"></div>
  <h3>Auto-Assigning IPs via Subnet</h3>
  <p>When creating a LIF, instead of specifying an explicit IP address, simply reference the subnet name. ONTAP picks the next available IP from the pool and assigns it automatically. This reduces configuration errors and speeds up provisioning.</p>

  <div class="code-block">
    <div class="code-header">
      <div class="code-dots"><span class="dot-red"></span><span class="dot-ylw"></span><span class="dot-grn"></span></div>
      <span class="code-lang">ONTAP CLI — Create LIF Using Subnet</span>
    </div>
    <div class="code-body"><span class="comment"># Create a data LIF — IP auto-assigned from subnet pool</span>
<span class="prompt">::></span> <span class="cmd">network interface create</span> \
      -vserver <span class="value">svm-prod</span> \
      -lif <span class="value">data-lif-01</span> \
      -service-policy <span class="value">default-data-files</span> \
      -role <span class="value">data</span> \
      -home-node <span class="value">cluster-node1</span> \
      -home-port <span class="value">e0d</span> \
      -subnet-name <span class="value">prod-data-pool</span> \
      -status-admin <span class="value">up</span>

<span class="comment"># Verify — ONTAP assigned 10.50.10.101 automatically</span>
<span class="prompt">::></span> <span class="cmd">network interface show</span> -vserver svm-prod
<span class="output">LIF           Role    Status   Address           Subnet           Home Port
data-lif-01   data    up/up    10.50.10.101/24   prod-data-pool   e0d</span>

<span class="comment"># Check remaining pool — 8 IPs left after first LIF allocated</span>
<span class="prompt">::></span> <span class="cmd">network subnet show</span> -subnet-name prod-data-pool
<span class="output">Available: 8</span></div>
  </div>

<section class="section" id="s2">
  <div class="section-header">
    <div class="section-icon icon-blue">🔗</div>
    <div class="section-title-wrap">
      <div class="section-num">Section 02 · Replication Prerequisites</div>
      <h2>Cluster &amp; SVM Peering</h2>
    </div>
  </div>

  <p>Before establishing any replication relationship (including SVM DR), both clusters must have dedicated intercluster LIFs and a verified cluster peering relationship. SVM peering then defines which source and destination SVMs are allowed to exchange data via SnapMirror.</p>

  <div id="s2-sub1"></div>
  <h3>Intercluster LIF Configuration</h3>
  <p>Intercluster LIFs carry replication traffic between two ONTAP clusters. Each node participating in replication should have at least one intercluster LIF. These LIFs can also draw addresses from an existing subnet pool if the subnet's broadcast domain is shared with the intercluster ports.</p>

  <div class="code-block">
    <div class="code-header">
      <div class="code-dots"><span class="dot-red"></span><span class="dot-ylw"></span><span class="dot-grn"></span></div>
      <span class="code-lang">ONTAP CLI — Intercluster LIFs</span>
    </div>
    <div class="code-body"><span class="comment"># Create intercluster LIF on Cluster A, Node 1</span>
<span class="prompt">::></span> <span class="cmd">network interface create</span> \
      -vserver <span class="value">cluster-alpha</span> \
      -lif <span class="value">ic-lif-a1</span> \
      -service-policy <span class="value">default-intercluster</span> \
      -home-node <span class="value">node-alpha-01</span> \
      -home-port <span class="value">e0e</span> \
      -subnet-name <span class="value">prod-data-pool</span>

<span class="comment"># Create intercluster LIF on Cluster A, Node 2</span>
<span class="prompt">::></span> <span class="cmd">network interface create</span> \
      -vserver <span class="value">cluster-alpha</span> \
      -lif <span class="value">ic-lif-a2</span> \
      -service-policy <span class="value">default-intercluster</span> \
      -home-node <span class="value">node-alpha-02</span> \
      -home-port <span class="value">e0e</span> \
      -subnet-name <span class="value">prod-data-pool</span>

<span class="comment"># Both IPs assigned from pool: 10.50.10.103 and .104</span></div>
  </div>

  <div id="s2-sub2"></div>
  <h3>Establishing Cluster Peering</h3>

  <div class="code-block">
    <div class="code-header">
      <div class="code-dots"><span class="dot-red"></span><span class="dot-ylw"></span><span class="dot-grn"></span></div>
      <span class="code-lang">ONTAP CLI — Cluster Peer Create &amp; SVM Peer</span>
    </div>
    <div class="code-body"><span class="comment">## On Cluster A — generate a passphrase and initiate peering</span>
<span class="prompt">cluster-alpha::></span> <span class="cmd">cluster peer create</span> \
      -peer-addrs <span class="value">10.60.10.201,10.60.10.202</span> \
      -ipspace <span class="value">Default</span> \
      -generate-passphrase <span class="value">true</span>
<span class="output">Passphrase: NTAP-XK7P2-QR4M9</span>

<span class="comment">## On Cluster B — accept peering with the passphrase</span>
<span class="prompt">cluster-beta::></span> <span class="cmd">cluster peer create</span> \
      -peer-addrs <span class="value">10.50.10.103,10.50.10.104</span> \
      -passphrase <span class="value">NTAP-XK7P2-QR4M9</span>

<span class="comment">## Verify</span>
<span class="prompt">::></span> <span class="cmd">cluster peer show</span>
<span class="output">Peer Cluster Name   Availability   Auth Status
cluster-beta        Available      ok</span>

<span class="comment">## Create SVM Peering (Cluster A side)</span>
<span class="prompt">cluster-alpha::></span> <span class="cmd">vserver peer create</span> \
      -vserver <span class="value">svm-prod</span> \
      -peer-vserver <span class="value">svm-prod-dr</span> \
      -peer-cluster <span class="value">cluster-beta</span> \
      -applications <span class="value">snapmirror</span>

<span class="comment">## Accept on Cluster B</span>
<span class="prompt">cluster-beta::></span> <span class="cmd">vserver peer accept</span> \
      -vserver <span class="value">svm-prod-dr</span>

<span class="prompt">::></span> <span class="cmd">vserver peer show</span>
<span class="output">Peer SVM       Peer Cluster   Peering State
svm-prod-dr    cluster-alpha  peered</span></div>
  </div>

  <div class="callout warning">
    <span class="callout-icon">⚠️</span>
    <div class="callout-content">
      <div class="callout-title">SVM Type Must Be "DP Destination"</div>
      <div class="callout-text">When creating the destination SVM for SVM DR on the target cluster, you must explicitly set its <strong>subtype to <code>dp-destination</code></strong> during vserver creation. A regular SVM (default subtype) cannot serve as an SVM DR destination. This is a common setup mistake — the SnapMirror relationship will refuse to initialize if the destination SVM was not created with the correct subtype.</div>
    </div>
  </div>

  <div class="code-block">
    <div class="code-header">
      <div class="code-dots"><span class="dot-red"></span><span class="dot-ylw"></span><span class="dot-grn"></span></div>
      <span class="code-lang">ONTAP CLI — Create DP-Destination SVM</span>
    </div>
    <div class="code-body"><span class="comment"># On Cluster B — create the DR destination SVM with correct subtype</span>
<span class="prompt">cluster-beta::></span> <span class="cmd">vserver create</span> \
      -vserver <span class="value">svm-prod-dr</span> \
      -subtype <span class="value">dp-destination</span> \
      -allowed-protocols <span class="value">cifs,nfs</span> \
      -rootvolume-security-style <span class="value">ntfs</span>

<span class="comment"># Confirm subtype is dp-destination</span>
<span class="prompt">::></span> <span class="cmd">vserver show</span> -vserver svm-prod-dr -fields subtype
<span class="output">vserver         subtype
svm-prod-dr     dp-destination</span></div>
  </div>

  <div id="s3-sub2"></div>
  <h3>Identity Preserve: The Critical Parameter</h3>
  <p>The <code>-identity-preserve true</code> flag is what distinguishes SVM DR from plain volume SnapMirror. When enabled, ONTAP replicates not just volume data but also the SVM's full configuration — CIFS setup, NFS exports, subnets, DNS mappings, and all LIFs (with their original IP addresses). This is what makes transparent client failover possible.</p>

  <div class="grid-2">
    <div class="card">
      <div class="card-title"><span style="color:var(--accent-green)">✅</span> Identity Preserve: TRUE</div>
      <ul class="bullet-list">
        <li>Copies CIFS/SMB configuration to DR site</li>
        <li>Replicates NFS export policies</li>
        <li>Creates same LIFs (same IPs) on destination</li>
        <li>Copies subnet associations and DNS config</li>
        <li>Clients reconnect to same IP — no changes needed</li>
      </ul>
    </div>
    <div class="card">
      <div class="card-title"><span style="color:var(--accent-orange)">⚠️</span> Identity Preserve: FALSE</div>
      <ul class="bullet-list">
        <li>Only volume data is replicated</li>
        <li>No protocol configuration copied</li>
        <li>No LIFs created at DR site</li>
        <li>Manual reconfiguration required post-failover</li>
        <li>Clients must update DNS/IP after failover</li>
      </ul>
    </div>
  </div>

  <div class="code-block">
    <div class="code-header">
      <div class="code-dots"><span class="dot-red"></span><span class="dot-ylw"></span><span class="dot-grn"></span></div>

      <h2>ONTAP Upgrade: Overview &amp; Version Selection</h2>
    </div>
  </div>

  <p>ONTAP upgrades are operationally straightforward, but the <strong>pre-upgrade preparation</strong> is where most of the complexity lies. Getting the version selection, compatibility checks, and risk assessment right before touching a single controller is what separates a smooth upgrade from a production incident.</p>

  <div class="grid-3">
    <div class="stat-card">
      <div class="stat-num" style="color:var(--accent-teal)">3</div>
      <div class="stat-label">Upgrade Types</div>
    </div>
    <div class="stat-card">
      <div class="stat-num" style="color:var(--accent-blue)">5+</div>
      <div class="stat-label">Pre-checks Required</div>
    </div>
    <div class="stat-card">
      <div class="stat-num" style="color:var(--accent-green)">~30m</div>
      <div class="stat-label">Per Controller (min)</div>
    </div>
  </div>

  <div id="s5-sub1"></div>
  <h3>Step 1: Selecting the Target Version</h3>
  <p>Version selection is not arbitrary. The goal is to move to a release that resolves current security risks and bugs in your environment while also satisfying hardware limits across all your platforms. Most organizations follow an <strong>N-1 strategy</strong> — upgrading to one release behind the latest, which offers the most thorough patch history and stability.</p>

  <div class="card">
    <div class="card-title">📌 Version Selection Decision Framework</div>
    <ul class="step-list">
      <li><span class="step-num">1</span><div><strong>Identify existing risks</strong> — Use ActiveIQ to find high/medium/low risks in your environment and note which ONTAP versions resolve them.</div></li>
      <li><span class="step-num">2</span><div><strong>Apply N-1 strategy as baseline</strong> — Most enterprises target one version behind latest (e.g. if 9.13 is current, target 9.12.1P5).</div></li>
      <li><span class="step-num">3</span><div><strong>Always pick highest patch</strong> — Within your chosen release line, always pick the latest patch (e.g. 9.10.1P7 over 9.10.1P3). Later patches fix bugs found in earlier ones.</div></li>
      <li><span class="step-num">4</span><div><strong>Check hardware ceiling</strong> — Confirm all your hardware models support the chosen version (see Section 6).</div></li>
      <li><span class="step-num">5</span><div><strong>Validate upgrade path</strong> — Ensure there is a direct upgrade path. Some older versions require an intermediate hop.</div></li>
    </ul>
  </div>

  <div id="s5-sub2"></div>
  <h3>Step 2: Release Notes &amp; Risk Assessment</h3>
  <p>Before selecting your target version, review the official ONTAP release notes. They document every enhancement, bug fix, known issue, and resolved security risk for that release — and crucially, which versions introduced or resolved each item. Search for <em>NetApp ONTAP release notes</em> and open the HTML version for the target release family.</p>

  <div class="callout info">
    <span class="callout-icon">📖</span>
    <div class="callout-content">
      <div class="callout-title">Where to Find Release Notes</div>
      <div class="callout-text">Navigate to <strong>docs.netapp.com → ONTAP → Release Notes</strong> and select your target version line. The release notes list enhancements by category (SAN, NAS, Security, Replication, etc.) and include a dedicated resolved issues section with bug IDs. This document should be reviewed thoroughly before any upgrade project begins.</div>
    </div>
  </div>

</section>

This document typically becomes part of the Change Control submission for the upgrade project.</div>

<section class="section" id="s9">
  <div class="section-header">
    <div class="section-icon icon-purple">🎓</div>
    <div class="section-title-wrap">
      <div class="section-num">Section 09 · Knowledge Check</div>
      <h2>Scenario-Based Interview Q&amp;A</h2>
    </div>
  </div>

  <div class="quiz-card">
    <div class="quiz-q">Q1: What is the key difference between volume-level SnapMirror and SVM DR failover from an end-client perspective?</div>
    <button class="quiz-btn" onclick="this.nextElementSibling.style.display='block';this.style.display='none'">Reveal Answer</button>
    <div class="quiz-a">With <strong>volume-level SnapMirror</strong> failover, the destination volume gets a new IP / LIF at the DR site. End clients must update their mount points or DNS entries to reconnect — causing visible disruption.<br><br>With <strong>SVM DR (identity-preserve true)</strong>, the destination SVM is configured with the same LIF IP addresses as the source. During failover, destination LIFs activate with the identical IPs, and source LIFs go down. Clients reconnect automatically without any configuration changes — transparent failover.</div>
  </div>

  <div class="quiz-card">
    <div class="quiz-q">Q2: You are upgrading a mixed environment with FAS 8020, FAS 9000, and AFF A400 controllers. Your target is ONTAP 9.9. Is this feasible?</div>
    <button class="quiz-btn" onclick="this.nextElementSibling.style.display='block';this.style.display='none'">Reveal Answer</button>
    <div class="quiz-a"><strong>Partially feasible — but the FAS 8020 is a blocker.</strong><br><br>FAS 9000 and AFF A400 support ONTAP 9.9. However, FAS 8020 has a maximum supported version of 9.8.x — it cannot be upgraded to 9.9 (the version will appear grayed out in HWU). This means:<ul style="margin:10px 0 0 16px"><li>Either downgrade your target to 9.8.x (which all three platforms support)</li><li>Or plan a hardware refresh/replacement of FAS 8020 before attempting a 9.9 upgrade</li></ul>The correct tool for this analysis is <strong>hwu.netapp.com → select all platforms → check ONTAP version support</strong>.</div>
  </div>

  <div class="quiz-card">
    <div class="quiz-q">Q3: What does "identity-preserve" do in an SVM DR relationship, and when would you set it to false?</div>
    <button class="quiz-btn" onclick="this.nextElementSibling.style.display='block';this.style.display='none'">Reveal Answer</button>
    <div class="quiz-a"><code>identity-preserve true</code> (default) replicates the complete SVM identity: protocol configurations (CIFS/NFS), DNS settings, subnet associations, and all LIF definitions with their original IP addresses. This enables transparent client failover.<br><br><code>identity-preserve false</code> replicates only volume data. No protocol config or LIFs are created on the destination. Use this when:<ul style="margin:10px 0 0 16px"><li>The DR site uses a completely different IP addressing scheme</li><li>You plan to manually configure the destination SVM for a different client population</li><li>You only need data protection without transparent failover capability</li></ul></div>
  </div>

  <div class="quiz-card">
    <div class="quiz-q">Q4: Your 12-node ONTAP cluster upgrade is scheduled for an 8-hour maintenance window. A colleague suggests using rolling upgrade to "reduce risk." Is this the right approach?</div>
    <button class="quiz-btn" onclick="this.nextElementSibling.style.display='block';this.style.display='none'">Reveal Answer</button>
    <div class="quiz-a"><strong>No — for a 12-node cluster, batch upgrade is the correct procedure.</strong><br><br>Rolling upgrade for 12 nodes at ~30 minutes per node would require approximately 6 hours of pure upgrade time — and that excludes any troubleshooting time if issues arise. This easily exceeds the change window and puts the entire project at risk.<br><br>ONTAP will automatically select batch upgrade for clusters &gt; 8 nodes. In batch mode, ONTAP upgrades one node from each HA pair simultaneously (i.e., 6 nodes in parallel for a 12-node cluster), reducing total time to approximately 1.5–2 hours. The partner nodes remain active and serve data throughout.<br><br>"Reducing risk" through rolling upgrade in a large cluster is counterproductive — the extended maintenance window itself creates risk to the change control process.</div>
  </div>

  <div class="quiz-card">
    <div class="quiz-q">Q5: When checking software compatibility for an ONTAP upgrade, what happens if you miss a server HBA driver alert in the IMT?</div>
    <button class="quiz-btn" onclick="this.nextElementSibling.style.display='block';this.style.display='none'">Reveal Answer</button>
    <div class="quiz-a"><strong>LUNs may become inaccessible after the upgrade.</strong><br><br>Incompatible HBA drivers are the most common cause of post-upgrade SAN connectivity issues. Symptoms include:<ul style="margin:10px 0 0 16px"><li>LUNs disappearing from hosts after ONTAP upgrade</li><li>Multipath paths going offline</li><li>Application failures due to storage connectivity loss</li></ul>IMT <em>Alert</em> items are mandatory — they must be reviewed and addressed before the upgrade maintenance window. The recommended approach: export IMT results with all applicable KB articles, share with the server team well in advance, and confirm all patches/driver updates are applied before the change window opens.</div>
  </div>

  <div class="quiz-card">
    <div class="quiz-q">Q6: What is the vserver-dr-protection parameter and how does it affect SVM DR replication?</div>
    <button class="quiz-btn" onclick="this.nextElementSibling.style.display='block';this.style.display='none'">Reveal Answer</button>
    <div class="quiz-a">The <code>-vserver-dr-protection</code> parameter on a volume controls whether that volume participates in SVM DR replication.<br><br><strong>protected</strong> (default): The volume is included in SVM DR replication. It will be created on the destination SVM during the next scheduled or manual transfer.<br><br><strong>unprotected</strong>: The volume is excluded from SVM DR. Even if it exists on the source SVM, it will not appear on the DR destination. Use this for temporary or scratch volumes that don't need disaster recovery coverage — this also reduces replication overhead and baseline transfer time.</div>
  </div>

</section>

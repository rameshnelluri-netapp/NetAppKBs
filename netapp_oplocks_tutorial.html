Section 01

What are Oplocks?

Opportunistic locks (oplocks) are a client-side caching mechanism in the SMB/CIFS protocol. They allow a Windows (or any SMB-capable) client to cache file data, read-aheads, write-behinds, and lock information locally — without constantly communicating with the ONTAP storage server.

🧠 The Core Idea

When a client opens a file and no other client is accessing it, ONTAP grants the client an oplock — essentially a promise: "You are the only one using this file right now. You may cache reads and buffer writes locally. I'll notify you if anyone else wants it."

This eliminates the need for the client to flush writes to the server after every operation, reducing network round-trips and dramatically improving performance for single-user file access patterns.

✅ Without Oplocks — Expensive

  • Every read goes to the storage server
  • Every write is synchronously flushed
  • Every byte-range lock is server-round-tripped
  • Applications like Office, ERP, IDEs behave sluggishly

⚡ With Oplocks — Efficient

  • Reads are served from local client cache
  • Writes are buffered, flushed in batches
  • Lock state is managed locally by the client
  • Network traffic is significantly reduced
⚠️
The trade-off: cached data is at risk
If a CIFS session is unexpectedly lost while a client holds an oplock (e.g. network failure, client crash), any writes buffered locally but not yet flushed to ONTAP may be lost. This is an acceptable trade-off for most workloads but is critical to understand before disabling oplocks on live systems.
Section 02

Oplock Types in ONTAP

ONTAP supports four oplock levels, negotiated dynamically between the client and server based on SMB dialect version and current access patterns.

Exclusive / Level 1

Exclusive Oplock

Granted when a single client opens a file that no other client is using. The client gets exclusive read + write caching. Only available in SMB2+.

✓ Read cache ✓ Write cache ✗ Handle cache
🔌 SMB 2.0+
Batch

Batch Oplock

Designed for programs that repeatedly open, read/write, and close the same file (like compilers or batch scripts). The client can open the file once and keep the handle open server-side while performing many local open/close cycles.

✓ Read cache ✓ Write cache ✓ Handle cache
🔌 SMB 1.0, 2.x
Level II / Shared

Level II Oplock

Granted when multiple clients open the same file. Allows shared read caching across all clients, but no write buffering — any write by any client immediately invalidates all level-2 oplocks.

✓ Read cache ✗ Write cache ✗ Handle cache
🔌 SMB 1.0, 2.x, 3.x
Lease Oplock

Lease Oplock (SMB 2.1+)

A more flexible caching model introduced in SMB 2.1. A lease is tied to a client-file pair (not just an open handle), so caching persists across multiple open/close cycles on the same client. Essential for non-disruptive ONTAP operations.

✓ Read cache ✓ Write cache ✓ Handle cache
🔌 SMB 2.1+ (Windows 7+)
💡
Lease oplocks are essential for non-disruptive operations
ONTAP uses lease oplocks internally to enable non-disruptive LIF migration and storage failover for SMB clients. Disabling lease oplocks at the CIFS server level will break NDO (Non-Disruptive Operations). Disabling at the share level only affects opportunistic — not lease — oplocks when configured via vserver cifs options.
Section 03

How Oplocks Work — The Break Sequence

The oplock lifecycle has two phases: the grant (straightforward) and the break (the critical part). Understanding the break sequence is essential for diagnosing oplock-related performance and stability issues.

Oplock Grant → Break → Re-grant lifecycle
1
Client 1 opens file, oplock granted
Client 1 opens report.xlsx. ONTAP grants a Batch or Exclusive oplock. Client 1 begins caching reads and buffering writes locally.
ONTAP → Client 1
2
Client 2 requests access to the same file
Client 2 sends an open request for report.xlsx. ONTAP cannot grant conflicting access while Client 1 holds an exclusive oplock.
Client 2 → ONTAP
3
ONTAP sends an Oplock Break Request to Client 1
ONTAP holds off on Client 2's open request and sends a break to Client 1, asking it to flush its write cache and release or downgrade its oplock. The timer starts: Client 1 has 35 seconds to respond.
ONTAP → Client 1
4
Client 1 flushes cache and acknowledges
Client 1 writes its buffered data to ONTAP and sends an Oplock Break Acknowledgement, either releasing the oplock fully or downgrading to a Level II shared oplock.
Client 1 → ONTAP
5
ONTAP grants Client 2 access
With Client 1's cached data now safely on disk, ONTAP processes Client 2's open request and grants the appropriate lock level (often Level II for both clients).
ONTAP → Client 2
🚨
Oplock break timeout — 35 seconds
If Client 1 does not respond to the break request within 35 seconds, ONTAP forcefully cleans up all locks for Client 1 and grants Client 2 access. Any data buffered by Client 1 that was not flushed is lost. Prior to ONTAP 9.0, a "delayed break" EMS message was also logged with the offending client's IP address. This 35-second delay is also what causes application timeouts when clients are slow to respond to break requests.
Section 04

Configuration — Three Levels

Oplocks in ONTAP are controlled at three levels: the CIFS server (SVM), the volume, and the share. A disable at any level takes precedence — the most restrictive setting wins.

📋
Precedence rule
Share-level disable > Volume-level disable > SVM-level setting. Disabling oplocks on a share overrides any volume-level enablement. Disabling on a share disables both opportunistic and lease oplocks for that share.
ontap-cli :: SVM / CIFS Server level 
# Show current CIFS server oplock settings
vserver cifs options show -vserver svm_prod

# Enable oplocks at SVM level (default)
vserver cifs options modify -vserver svm_prod \
  -is-oplocks-enabled true

# Disable oplocks at SVM level (affects all shares on this SVM)
vserver cifs options modify -vserver svm_prod \
  -is-oplocks-enabled false

# Lease oplocks — separate control (do not disable for NDO)
vserver cifs options modify -vserver svm_prod \
  -smb2-enabled-for-dc-connections true
ontap-cli :: Volume level 
# Show oplock setting on a specific volume
volume show -vserver svm_prod \
  -volume vol_cifs_data \
  -fields oplocks

# Enable oplocks on a volume (default is enabled)
volume modify -vserver svm_prod \
  -volume vol_cifs_data \
  -oplocks enabled

# Disable oplocks on a volume (e.g. database or critical data)
volume modify -vserver svm_prod \
  -volume vol_db_files \
  -oplocks disabled

# Output example:
Volume Modified: vol_db_files
ontap-cli :: Share level (existing share) 
# Show all share properties (check if oplocks is listed)
vserver cifs share properties show \
  -vserver svm_prod \
  -share-name Engineering

# Add oplocks property to an existing share (enables it)
vserver cifs share properties add \
  -vserver svm_prod \
  -share-name Engineering \
  -share-properties oplocks

# Remove oplocks property from an existing share (disables it)
vserver cifs share properties remove \
  -vserver svm_prod \
  -share-name DB_Share \
  -share-properties oplocks

# Verify:
vserver cifs share properties show
Vserver   Share          Properties
--------- -------------- -------------------
vs1       Engineering    oplocks browsable changenotify showsnapshot
vs1       DB_Share       browsable changenotify showsnapshot
ontap-cli :: Creating a new share with explicit oplock setting 
# Create share WITH oplocks (default — include oplocks in properties)
vserver cifs share create \
  -vserver svm_prod \
  -share-name UserData \
  -path /vol_cifs_data/userdata \
  -share-properties oplocks,browsable,changenotify,showsnapshot

# Create share WITHOUT oplocks (for databases, Hyper-V, SQL over SMB)
vserver cifs share create \
  -vserver svm_prod \
  -share-name SQL_Data \
  -path /vol_sql/data \
  -share-properties browsable,changenotify

# Note: omitting 'oplocks' from -share-properties disables it
Section 05

Monitoring Oplocks

Use these commands to inspect active oplocks, identify stale locks, and diagnose oplock break delays.

ontap-cli :: List all active oplocks 
# Show all lock types including oplocks (summary)
vserver locks show

# Filter for oplock type only
vserver locks show -lock-type op-lock

# Filter by SVM and volume
vserver locks show \
  -vserver svm_prod \
  -volume vol_cifs_data

# Filter by specific file path
vserver locks show \
  -path /vol_cifs_data/docs/report.xlsx

# Example output:
Vserver  Volume       Object Path              LIF      Protocol Lock Type  Client
-------- ------------ ------------------------ -------- -------- ---------- ------
svm_prod vol_cifs_data /docs/report.xlsx       lif_cifs cifs     op-lock    10.1.1.50
svm_prod vol_cifs_data /docs/budget.xlsx       lif_cifs cifs     op-lock    10.1.1.52
ontap-cli :: Detailed oplock information 
# Show full oplock details for a specific path
vserver locks show -instance \
  -path /vol_cifs_data/docs/report.xlsx

# Full output reveals lock UUID, oplock level, client IP, SMB open type:
Vserver:                  svm_prod
Volume:                   vol_cifs_data
Logical Interface:        lif_cifs_01
Object Path:              /vol_cifs_data/docs/report.xlsx
Lock UUID:                553cf484-7030-4998-88d3-1125adbba0b7
Lock Protocol:            cifs
Lock Type:                op-lock
Node Holding Lock State:  node1
Lock State:               granted
Oplock Level:             batch
Client Address:           10.1.1.50
SMB Open Type:            durable
SMB Connect State:        connected
ontap-cli :: Breaking a stale oplock manually 
# Step 1: Find the UUID of the oplock to break
vserver locks show -instance \
  -path /vol_cifs_data/docs/report.xlsx

# Note the Lock UUID from output:
Lock UUID: 553cf484-7030-4998-88d3-1125adbba0b7

# Step 2: Break the lock using its UUID
vserver locks break \
  -vserver svm_prod \
  -volume vol_cifs_data \
  -lock-id 553cf484-7030-4998-88d3-1125adbba0b7

# Break ALL locks on a volume (use with caution)
vserver locks break \
  -vserver svm_prod \
  -volume vol_cifs_data

# ⚠ WARNING: Breaking an active oplock forcibly discards the client's
#   buffered write cache. Data not yet flushed to ONTAP will be LOST.
#   Always verify the client is offline/disconnected first.
ontap-cli :: EMS events related to oplocks 
# Search EMS for oplock break timeout events
event log show \
  -messagename CIFS.oplock.break.timeout

# Search for any CIFS lock related events (last 24h)
event log show \
  -time-range 24h \
  -messagename CIFS.*

# Show all EMS severity error events
event log show \
  -severity error \
  -time-range 1h

# Example oplock break timeout EMS message:
Time                Node    Severity     Event
------------------- ------- ------------ -----------------------------------
2025-06-13 09:12:30 node1   ERROR        CIFS.oplock.break.timeout:
                                         Oplock break timed out for client
                                         10.1.1.77, file: /vol_cifs/data.xlsx
Section 06

Real-World Scenarios

Expand each scenario to see the problem, root cause, and resolution with ONTAP CLI commands.

1
Office users experience slow file open / "File in use" errors
SMB/CIFS · Microsoft Office · Oplock break delay
▶

Symptom

Users report that opening Excel files takes 30–40 seconds intermittently. Some users see "File is locked by another user" even when no one else has the file open.

Root Cause

A previous user's client crashed while holding a batch oplock. ONTAP sent a break request, waited 35 seconds, then forcefully cleaned up — but the stale lock entry was still visible briefly. Additionally, a network switch issue caused break request packets to be delayed, making healthy clients appear to be timed out.

Diagnosis Commands

vserver locks show -vserver svm_files -lock-type op-lock
# Look for locks with disconnected SMB connect state:
vserver locks show -instance -vserver svm_files
# Check EMS for timeout events:
event log show -messagename CIFS.* -time-range 1h

Resolution

Break the stale oplock using the UUID. Investigate the network switch for packet loss between that client subnet and the storage LIF. No oplock configuration change was required — this was a network fault, not an oplock misconfiguration.

2
Database application reports data inconsistency / recommends disabling oplocks
SQL over SMB · DB vendor requirement · Volume-level disable
▶

Symptom

A third-party database (e.g. older ERP system) vendor documentation explicitly states: "Disable opportunistic locking on any network share hosting database files." After enabling oplocks, the application shows occasional commit failures.

Why Databases Reject Oplocks

Database engines manage their own internal locking and consistency mechanisms. They assume all writes are immediately durable on disk. Oplocks allow the client OS to delay writes, which can desynchronize the database's internal state from what is actually on storage — causing corruption if the client crashes between a buffered write and a flush.

Resolution

# Disable oplocks at the volume level for the DB volume
volume modify -vserver svm_db \
  -volume vol_erp_data \
  -oplocks disabled

# Also remove from share if set there
vserver cifs share properties remove \
  -vserver svm_db \
  -share-name ERP_Data \
  -share-properties oplocks

# Verify
volume show -volume vol_erp_data -fields oplocks
Oplocks: disabled
3
ONTAP LIF migration breaks SMB sessions (NDO failure)
Non-Disruptive Operations · Lease Oplocks · SMB 2.1
▶

Symptom

During a planned ONTAP upgrade or LIF migration, Windows clients lose their SMB sessions and need to reconnect manually. Applications running against CIFS shares throw "network path not found" errors.

Root Cause

Lease oplocks were disabled at the CIFS server level — either intentionally (misguided policy) or accidentally. Lease oplocks are what enable SMB clients to survive LIF migrations and storage failover events without disconnecting. Without them, every failover event appears as a disconnect to the client.

Diagnosis

vserver cifs options show -vserver svm_prod \
  -fields is-oplocks-enabled

# Check if SMB2 is enabled (required for lease oplocks)
vserver cifs options show -vserver svm_prod \
  -fields smb2-enabled

Resolution

# Re-enable oplocks at SVM level
vserver cifs options modify \
  -vserver svm_prod \
  -is-oplocks-enabled true

# Ensure SMB 2.1+ is enabled (for lease oplock support)
vserver cifs options modify \
  -vserver svm_prod \
  -smb2-enabled true
4
Second client sees stale data from first client's cache
Multi-client access · Level II downgrade · Read cache staleness
▶

Symptom

In a shared folder, User B opens a file that User A recently edited. User B sees the old version of the file even though User A saved it 10 seconds ago.

What Happened

User A held a batch oplock and wrote changes to the local write cache. When User B opened the same file, ONTAP sent a break request to User A. The break request requires User A to flush the cache — but if User A's client is briefly unresponsive or the break packet is delayed on the network, User B momentarily reads from what appears to be a stale view until the flush completes.

Diagnosis

# Check oplock state during the issue window
vserver locks show -vserver svm_files \
  -path /vol_cifs/shared/document.docx \
  -instance
# Look for: Lock State: break-sent (means break was issued, pending ack)

Resolution Options

1. Investigate and fix client-side network latency causing slow break acknowledgements.

2. For ultra-critical shared files where multiple concurrent editors are expected, disable oplocks at the share level to force synchronous writes — accepting the performance trade-off.

3. Consider application-level locking (e.g. SharePoint, file collaboration tools) for heavy multi-editor workflows instead of raw CIFS shares.

Section 07

When to Enable vs. Disable

The default in ONTAP is to leave oplocks enabled. Disabling should be a deliberate, targeted decision — never a blanket policy across all SVMs.

Workload / Use Case Oplock Setting Level to Configure Reasoning
General file sharing (Office, PDFs, images) Enable Default (no change) Maximum performance benefit. Single-user file access is the common case.
Home directory shares Enable Default (no change) Each user owns their own files. Oplock conflicts are rare.
Database files (SQL, Oracle, ERP) Disable Volume + Share Databases manage their own caching and consistency. Oplocks interfere.
Hyper-V / VMs over SMB Selective Share level Hyper-V 2012+ uses its own SMB3 persistent handles. Follow Microsoft guidance per share type.
SQL Server over SMB Disable Share level SQL Server documentation explicitly requires oplocks to be disabled on data/log shares.
Software build / compile shares Enable Default (no change) Batch oplocks are designed for this. Build tools open/read/close many files rapidly.
Backup staging areas Enable Default Backups are typically single-stream writes. Oplocks help throughput.
Critical shared config files (multi-writer) Disable Share or Volume When consistency matters more than performance and multiple clients write the same file.

Best Practice Checklist

  • ✓Never disable oplocks globally at the SVM level unless every single share on that SVM requires it. Use volume or share-level control for targeted workloads.
  • ✓Never disable lease oplocks at the CIFS server level. Lease oplocks are required for SMB non-disruptive operations during LIF migrations and failover.
  • ✓Monitor EMS for CIFS.oplock.break.timeout events. Repeated timeouts from the same client IP indicate a network problem or misbehaving application — not an oplock misconfiguration.
  • ✓Use vserver locks show -instance to identify which oplock level (batch, exclusive, level2) is being held before breaking a lock manually.
  • ✓When disabling for a database workload, disable at both the volume and share level — one without the other is insufficient since share-level takes precedence over volume-level.
  • ✓Ensure all Windows clients accessing SMB shares support SMB 2.1+ (Windows 7 / Server 2008 R2 or later) to benefit from the more resilient lease oplock model.
  • ✓After any oplock configuration change, test with the target application and verify with vserver cifs share properties show and volume show -fields oplocks.
✅
Golden rule
Leave oplocks enabled everywhere, then selectively disable only for workloads where the application vendor explicitly requires it or where you have diagnosed actual data consistency problems caused by oplock caching. The performance cost of disabling is real and measurable — every file operation becomes a synchronous round-trip to storage.

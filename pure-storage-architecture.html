Every core ONTAP concept (WAFL, inodes, aggregates, disk shelves, qtrees, upgrades) is mapped to its Pure Storage equivalent so you can translate your existing mental model directly.

Section 01 · Architecture

FlashArray Overview

A Pure Storage FlashArray is an all-flash storage system built around two key principles that differ fundamentally from traditional RAID-based arrays like ONTAP: DirectFlash (Pure manages raw NAND directly instead of treating SSDs as black-box drives behind a RAID controller) and RAID-HA (a Pure-proprietary erasure-coding scheme, not traditional RAID). Every FlashArray ships as an HA pair of controllers sharing access to a common pool of DirectFlash Modules (DFMs) or NVMe SSDs.

Purity Operating Environment — the equivalent of ONTAP

Purity//FA (Purity for FlashArray) is the operating system that runs on every FlashArray controller — directly analogous to ONTAP running on a FAS/AFF controller. Purity handles everything: the storage stack (DirectFlash management, RAID-HA erasure coding, metadata), protocol services (FC, iSCSI, NVMe-oF, NFS, SMB, S3), replication (ActiveCluster, ActiveDR, async replication), and the management plane (CLI, REST API, web GUI, Pure1 cloud telemetry).

Unlike ONTAP's WAFL filesystem which is the foundational data structure for everything (volumes, LUNs, files all live inside WAFL), Purity is built around a global metadata-driven content-addressable architecture. Data blocks are stored once, referenced by content hash, and every volume/LUN/file is essentially a map of pointers into this shared global pool — deduplication and pattern removal are inherent to how data is stored, not a bolt-on efficiency feature run periodically.

Controller modules and active-active design

Both controllers (CT0 and CT1) are active-active at all times — both can service I/O for any volume simultaneously (unlike ONTAP's HA pair where each node owns specific aggregates and the partner is a standby path via ALUA). Purity's NVRAM on each controller mirrors incoming writes to the partner controller before acknowledging the host — conceptually similar to ONTAP's NVRAM mirroring over the HA interconnect, but because both controllers are always active, there's no "takeover/giveback" cycle in the ONTAP sense. If one controller fails, the surviving controller simply continues servicing all I/O — Pure calls this "no perceptible failover".

💡
Key mental shift from ONTAP In ONTAP, capacity is organised into aggregates that belong to specific nodes — a volume "lives" on a node and moving it is an explicit operation (volume move). In Purity, there is one global pool per array. Every volume can be served by either controller with no concept of "which node owns this." There is no equivalent of moving a volume between aggregates because there is only one pool.
Section 02 · Hardware Models

FlashArray Model Lineup

FlashArray//X and //XL — performance tier

Mainstream

FlashArray//X R4 / R5

The flagship general-purpose all-NVMe platform. Latest generation (R5) succeeded R4. Supports block (FC, iSCSI, NVMe/FC, NVMe/TCP, NVMe/RoCE) and unified File services (NFS, SMB) on the same array via the Unified Data Plane introduced in Purity//FA 6.6.

High-end Scale-out

FlashArray//XL R1 / R5

Larger chassis with more CPU, memory, and DFM slots than //X — built for the highest-density, highest-performance workloads. The //XL 190 model became generally available in early 2026 (Q4 FY26). Same Purity OS as //X — fully unified management and replication across the family.

FlashArray//C, //E, and //ST — capacity & specialised tiers

Capacity-optimised

FlashArray//C R3 / R4 / R5

Uses higher-capacity, lower-cost QLC DirectFlash Modules instead of the TLC modules in //X. Targets capacity-heavy workloads (backup targets, large file repositories, secondary storage) where raw IOPS density matters less than $/TB. Same Purity//FA OS and feature set as //X.

Backup-optimised

FlashArray//E

A newer capacity/economy tier positioned explicitly for backup and archive use cases — Pure has specifically highlighted //E as a target for FlashArray File Services given its cost profile, announced alongside Purity//FA 6.6.

Block-only, extreme density

FlashArray//ST

A block-only platform built on off-the-shelf commercial PCIe Gen5 SSDs (e.g. 6.4/12.8 TB drives) rather than Pure's proprietary DFMs, delivering over 10 million IOPS from a single system with up to 400 TB usable. The newer generation uses an Emerald Rapids CPU platform.

Cloud

Pure Storage Cloud / Cloud Block Store

Purity//FA running as a software-defined instance on AWS or Azure (also referred to as "Pure Storage Cloud Dedicated" / PSCD), providing the same Purity feature set and replication compatibility with on-prem FlashArrays for hybrid-cloud DR scenarios.

Section 03 · Data Flow

Write Path & Data Flow

FlashArray's write path is built around three core ideas: always-on inline reduction (deduplication, compression, and pattern removal happen before data ever touches flash — never as a background job), NVRAM-first acknowledgement (similar in spirit to ONTAP's NVRAM write-ahead log), and global metadata addressing (every unique data pattern is stored exactly once across the entire array, referenced by content hash from any volume).

Step-by-step write flow

Comparison to ONTAP write path

StageONTAP / WAFLPurity / FlashArray
Initial commit Write logged to NVRAM (write-ahead log), ACK sent to client Write hashed, committed to NVRAM, ACK sent to host (~150µs typical latency)
HA protection NVRAM mirrored to HA partner over dedicated HA interconnect NVRAM mirrored to partner controller; both controllers active for all volumes
Deduplication timing Background or inline post-process job, scoped per volume/aggregate Always-on, inline, global across the entire array — happens before the write ever lands on flash
Final write to media Consistency Point — buffered writes flushed to disk as large sequential I/O every ~10s Asynchronous flush from NVRAM to DFMs with RAID-HA erasure coding — continuous, not interval-based
Redundancy scheme RAID-DP / RAID-TEC (double/triple parity across spinning disks or SSDs in a RAID group) RAID-HA — Pure's proprietary erasure coding designed specifically for flash failure characteristics, operating across DFMs in the shared pool
Section 04 · Networking

Network Concepts

Block protocols — FC, iSCSI, NVMe-oF

FlashArray presents block storage as volumes mapped to hosts (or host groups) — the equivalent of ONTAP's LUN + igroup model. Each controller has multiple front-end ports that can be configured for:

  • Fibre Channel (FC): Each port has a WWPN. Hosts are zoned to array ports via FC switches — same fabric-zoning concept as ONTAP. ALUA applies identically — paths to the controller that "owns" a volume's primary copy are optimised, though because both controllers are active-active, path optimisation differences are typically smaller than in ONTAP.
  • iSCSI: Each port has an IP address and IQN. Host iSCSI initiators discover targets and log in, same workflow as ONTAP iSCSI.
  • NVMe-oF (NVMe/FC, NVMe/TCP, NVMe/RoCE): Lower-latency block access using the NVMe protocol over fabrics instead of SCSI. NVMe/TCP requires no special fabric hardware — runs over standard Ethernet, introduced for FlashArray in the Purity 6.4 release line.

Pure equivalents of ONTAP SAN constructs

LUN-to-host associationlun mapConnect volume to host

File Services — NFS and SMB

Since Purity//FA 6.1, FlashArray has supported native File Services — NFS and SMB shares served directly from the same Purity OS and global storage pool used for block volumes, via the Unified Data Plane. This is architecturally different from ONTAP, where NAS (NFS/CIFS via SVMs and LIFs) and SAN (LUNs via igroups) have always been first-class, deeply integrated citizens of the same WAFL filesystem since the C-Mode era.

  • File System — the Pure construct roughly analogous to an SVM's namespace or a FlexVol used for NAS. A File System is created on the array and exported via NFS and/or SMB.
  • Directories & Exports — within a File System, individual directories can have their own NFS export policies and SMB share configurations — conceptually similar to qtrees having independent export policies within a FlexVol.
  • Network Interfaces (data VIPs) — IP addresses assigned to controllers for serving NFS/SMB traffic, the rough equivalent of NAS LIFs.
Pure CLI — create a file system and NFS export 
# Create a File System (analogous to a FlexVol for NAS)
purefs create fs-app01

# Create a directory within the file system
puredir create fs-app01:dir1

# Create an NFS export policy and export the directory
purenfs create --rules "*(rw,no_root_squash)" nfs-policy-app01
puredir setattr --export nfs-policy-app01 fs-app01:dir1

# Create an SMB share on the same directory
puredir setattr --share share-app01 fs-app01:dir1
⚠️
Maturity difference vs ONTAP NAS ONTAP's NAS stack has 20+ years of multiprotocol depth — NFSv3/v4.x with delegations, Kerberos, FlexCache, FlexGroup, FPolicy, SVM-DR for entire NAS environments. Pure's File Services (GA since 6.1, with continuous enhancement through 6.6) covers core NFSv3/v4.1 and SMB with continuous availability, but administrators evaluating Pure for heavy NAS workloads should verify specific feature parity (e.g. NFSv4 delegations, Kerberos krb5p, FPolicy-equivalent ransomware detection) against current release notes for their target Purity version.

Section 05 · Concept Mapping

NetApp ONTAP vs Pure Storage — Concept by Concept

This section walks through the core ONTAP storage concepts you already know and explains how (or whether) each one maps onto Pure Storage's architecture. Some concepts translate cleanly; others don't exist on Pure because Pure's architecture eliminates the problem they were designed to solve.

WAFL vs Purity's storage engine

WAFL (Write Anywhere File Layout) is ONTAP's filesystem — a write-anywhere, copy-on-write design where every volume is a WAFL container with its own inode table, block maps, and Snapshot mechanism. WAFL is fundamentally a per-volume filesystem structure; aggregates are pools of RAID groups that WAFL volumes are carved out of.

Purity has no direct equivalent of "a filesystem per volume." Instead, Purity maintains one global content-addressable metadata layer for the entire array. Every unique 512-byte-to-variable-length data segment is stored once, hashed, and referenced by pointer from however many volumes/files need it. A "volume" in Purity is essentially a namespace of pointers into this shared data store, plus its own snapshot/metadata history — there's no separate "filesystem" being formatted onto carved-out RAID capacity the way a FlexVol sits on an aggregate.

AspectWAFL (ONTAP)Purity metadata engine
Scope Per-volume filesystem instance, multiple volumes share an aggregate's RAID groups Single global metadata/data layer for the entire array — no per-volume filesystem boundary
Write placement Write-anywhere — new blocks written to free space, old blocks retained for snapshots Content-addressed — unique data written once anywhere in the pool; volumes reference it by pointer
Deduplication scope Per-volume by default; cross-volume dedup available within an aggregate (9.2+) Global, array-wide, always-on, inline — every volume/file system shares one dedup domain
Snapshot mechanism Metadata pointer freeze at a point in time — only changed blocks consume new space Conceptually similar — a snapshot is a frozen reference to the metadata map at that instant; changed data creates new references, unchanged data stays shared

Inode & metadata model

In WAFL, an inode is the per-file metadata record — ownership, permissions, timestamps, and pointers to the file's 4KB data blocks. Inode count is a finite resource per volume, sized at creation and occasionally needing manual increase (see maxfiles/volume modify -files).

Purity's metadata model is built around content hashes and a global address map rather than per-volume inode tables. For block volumes, there's no "file" concept at all — a volume is a flat block address space, and Purity's metadata maps logical block addresses to physical locations on DFMs via content hash, identical in structure regardless of how many volumes exist. For File Services, Purity does maintain file-level metadata (POSIX attributes, ACLs) per File System, but it is not exposed as a tunable "inode count" the way ONTAP's per-volume inode limit is — there is no published "maxfiles" parameter administrators routinely hit and increase the way they do with WAFL maxdir/inode limits.

ℹ️
Practical implication An ONTAP admin who has hit "out of inodes, plenty of free space" errors on a volume full of millions of small files won't encounter that specific failure mode on Pure File Services in the same way — Purity's metadata scales dynamically with the global pool rather than being pre-allocated per volume at creation time. Always verify current documented limits for File Services file counts per File System for your Purity version, as published limits do still exist at the platform level.

Aggregate vs Pool

An ONTAP aggregate is a collection of RAID groups (RAID-DP/RAID-TEC) owned by a specific node, providing the physical capacity that FlexVol volumes are carved from. Aggregates are node-scoped — moving a volume between aggregates (volume move) may also mean moving it between nodes.

Pure has one array-wide pool — there is no aggregate concept, no RAID group sizing decisions, no per-node capacity silos. Every DFM in the chassis contributes to one shared pool protected by RAID-HA. All volumes draw from this single pool regardless of which controller is currently servicing their I/O. This eliminates an entire category of ONTAP capacity-planning work: there's no "which aggregate should this volume go on," no aggregate-level hotspot analysis, and no volume-move operations for rebalancing.

Disk Shelves — DS shelves vs DirectFlash shelves

ONTAP disk shelves (DS224C, DS2246, NS224, etc.) house individual SAS/SATA/NVMe drives that are assembled into RAID groups inside aggregates. Adding a shelf means adding more spindles/SSDs that get incorporated into existing or new RAID groups and aggregates — an explicit storage aggregate add-disks operation.

Pure's expansion shelves house additional DirectFlash Modules — when you add a shelf, the new DFMs are automatically incorporated into the array's single global pool and protected by RAID-HA without any administrator RAID-group configuration. There's no equivalent of choosing RAID group size or deciding which aggregate new capacity joins — it simply expands the one pool.

Upgrades — ANDU vs Pure non-disruptive upgrade

ONTAP's ANDU (Automated Non-Disruptive Upgrade) upgrades one node at a time: takeover by the HA partner, upgrade the offline node, reboot, giveback, then repeat for the partner. Each node experiences a brief takeover/giveback cycle (~2-3 minutes of the partner serving double load).

Pure's upgrade process is conceptually similar but framed differently because both controllers are always active: Purity upgrades one controller at a time while the other continues servicing 100% of I/O, then the upgraded controller rejoins and the second controller upgrades. Pure markets this as "non-disruptive upgrades" with proven 99.9999% availability, and the Self-Service Upgrade (SSU) feature in Pure1 automates pre-checks and execution — administrators trigger upgrades from the Pure1 cloud console with minimal manual CLI work, compared to ONTAP's more manual cluster image update workflow.

Qtree & Volume hierarchy

In ONTAP, the hierarchy is: Aggregate → FlexVol Volume → Qtree → Files. Qtrees provide a sub-volume boundary for independent export policies, security styles, and quotas without the overhead of a full volume.

Pure's File Services hierarchy is: Pool (array-wide) → File System → Directory → Files. A Pure File System is the rough peer of a FlexVol used for NAS — it has its own snapshot history and can be replicated as a unit. A Directory within a File System can have its own NFS export and SMB share configuration — this is the closest functional analog to an ONTAP qtree, though Pure directories don't carry the independent security style concept (Unix/NTFS/Mixed) in the same explicit, configurable way qtrees do, since Purity's File Services security model is more unified across protocols.

💡
The biggest structural difference: SVMs have no Pure equivalent ONTAP's SVM is a complete virtualised storage server — its own namespace, its own CIFS server/AD identity, its own NFS configuration, its own LIFs, replicable as a unit via SVM-DR. Pure has no construct that virtualises "an entire storage server" this way. If your ONTAP design relies heavily on per-tenant SVMs with separate AD identities and SVM-DR for whole-tenant failover, the Pure equivalent requires rethinking around Pods (for replication grouping) plus array-level or File-System-level access controls — there isn't a one-to-one swap.
Section 06 · Interview Q&A

Interview & Exam Questions

  • Q01 Why doesn't Pure FlashArray have an "aggregate" concept like ONTAP? ▾
    ONTAP aggregates exist because WAFL volumes need a node-scoped pool of RAID-protected capacity to be carved from, and because ONTAP's HA model assigns aggregate ownership to specific nodes. Pure's architecture uses a single global content-addressable metadata layer across the entire array, with both controllers active-active for all volumes simultaneously. Since there's no per-node ownership of capacity and no per-volume filesystem instance that needs its own RAID-protected container, there's nothing for an "aggregate" abstraction to describe — every DFM contributes to one pool, period.
  • Q02 How does Pure achieve global, always-on deduplication while ONTAP dedup is typically per-volume or per-aggregate? ▾
    Purity's metadata engine hashes every incoming data block and checks it against an array-wide index of all previously-stored unique blocks before writing anything to flash. Because this check happens inline, in the write path, for the entire array's data — not scoped to a volume or aggregate's local fingerprint database — any duplicate data anywhere on the array (across all volumes, all hosts, all File Systems) is detected and deduplicated immediately. ONTAP's cross-volume dedup (9.2+) extended dedup scope to the aggregate level, but Pure's scope has always been the entire array from day one, because the underlying storage model has no volume/aggregate boundaries for data placement to begin with.
  • Q03 An ONTAP admin is used to running volume move to fix an aggregate hotspot. What's the Pure equivalent? ▾
    There isn't a direct equivalent because the underlying problem doesn't exist on Pure in the same form. Aggregate hotspots occur in ONTAP because capacity is partitioned into node-scoped aggregates with finite RAID-group I/O capacity, and an imbalanced workload distribution across aggregates concentrates load. Pure has one pool spanning all DFMs with both controllers actively serving all volumes — there's no aggregate-level partition for load to concentrate within. If a Pure array as a whole is approaching its performance ceiling, the resolution is typically array-level capacity planning (adding a shelf, or in extreme cases migrating volumes to a different/larger array using ActiveWorkload) rather than an internal rebalancing operation.
  • Q04 What is RAID-HA and how does it differ from RAID-DP? ▾
    RAID-DP (double parity) is a traditional RAID scheme using two dedicated parity disks per RAID group, tolerating two simultaneous disk failures within that group — designed around spinning-disk failure characteristics (long, predictable rebuild times). RAID-HA is Pure's proprietary erasure-coding scheme designed specifically for flash media — it distributes data and parity information across DFMs in the pool in a way optimised for flash's different failure modes (wear-leveling, predictable endurance curves rather than mechanical failure) and for very fast rebuild times given flash's high throughput. RAID-HA is not configured by the administrator (no RAID group size or parity-disk-count decisions) — it is an inherent property of how Purity manages the DFM pool.
  • Q05 How would you replicate an entire multi-tenant environment for DR — SVM-DR on ONTAP vs Pure? ▾
    On ONTAP, SVM-DR replicates an entire SVM — every volume, the CIFS server's AD identity, export policies, LIFs — as a single unit, and `snapmirror activate-destination` promotes the DR SVM to take over the tenant's identity wholesale. Pure has no SVM-equivalent virtualisation layer, so there's no single "promote this tenant's entire storage server" operation. Instead, you'd group the tenant's volumes and File Systems into a Pod and use ActiveDR or async Protection Group replication for that pod to a target array. Failover means promoting the pod's replicated copy to read-write on the target array, but network identity (IPs for File Services, host FC/iSCSI connections) must be reconfigured separately — there's no automatic "identity-preserve" of an entire virtualised storage server the way SVM-DR provides.
  • Q06 What's the difference between Pure's Feature Release (FR) and Long-Term Maintenance (LLR) tracks, and how does it compare to ONTAP's release model? ▾
    Feature Release lines (e.g. Purity//FA 6.6.x, 6.10.x) receive continuous new capabilities as they're developed — customers who want the newest features as soon as possible track FR lines. LLR lines (e.g. 6.9.x, planned through June 2028) are derived from a stable feature set and receive only bug fixes and security patches for an extended multi-year support window — chosen by customers prioritising stability. This is conceptually similar to ONTAP's split between standard releases and long-term-support (LTS) release trains, though Pure's Self-Service Upgrade (SSU) via Pure1 makes moving between Purity lines a largely automated, cloud-orchestrated process compared to ONTAP's more CLI-driven `cluster image update` workflow.
  • Q07 How does FlashArray File Services compare to ONTAP's NAS capabilities for a heavy multiprotocol NFS/SMB environment? ▾
    FlashArray File Services (GA since Purity 6.1, with continuous enhancement through 6.6 including SMB Continuous Availability and native vSphere NFS datastore management) provides NFSv3/NFSv4.1 and SMB on the same Unified Data Plane as block volumes, with all of Purity's always-on data reduction and replication features. However, ONTAP's NAS stack has over two decades of depth in areas like NFSv4 delegations, Kerberos krb5p, FlexGroup (multi-PB scale-out NAS volumes), FlexCache (edge caching), FPolicy (file-access policy enforcement for ransomware detection), and SVM-DR for whole-NAS-environment failover. For environments with deep dependencies on these specific ONTAP NAS features, a careful feature-parity check against the target Purity release is essential before migration — Pure's File Services covers core multiprotocol NAS well but is a newer, narrower stack than ONTAP's.
  • Q08 A customer wants to migrate a volume from an older FlashArray//X R3 to a new //X R5 with zero downtime. What feature enables this, and is there an ONTAP equivalent? ▾
    ActiveWorkload (introduced in the Purity 6.4 line) enables non-disruptive workload migration of a volume between different FlashArray models/generations — the volume's data is migrated to the new array while remaining online and accessible, with host connectivity transitioning transparently. ONTAP doesn't have a direct one-to-one equivalent for moving a volume between two entirely separate clusters/controllers non-disruptively in this way — the closest analogs would be SnapMirror-based migration (which involves a cutover window) or, within a single cluster, volume move between nodes (which doesn't help when migrating to genuinely different hardware/clusters). ActiveWorkload is specifically designed for cross-generation hardware refresh scenarios.

Rubrik — From Scratch

A hands-on reference: how a Rubrik cluster is built (on-prem appliance, HA, own storage), how onboarding actually works step by step, how a backup runs end to end, and how data extends out to the cloud via Cloud Cluster and archival.

Rubrik CDM / Security Cloud AWS / Azure / GCP Hands-on Onboarding
01 — Architecture

What is Rubrik, conceptually?

Rubrik is a converged backup appliance — it combines backup software, a scale-out storage filesystem, and compute into a single clustered system. Instead of the traditional model (separate backup software + media servers + a SAN/NAS target for backup data), Rubrik is one product that does all three.

Two deployment philosophies exist side by side:

Rubrik CDM (Cloud Data Management)

  • The software running on each node — does the actual backup, storage, and recovery work
  • Runs on physical appliances ("Briks"), as a virtual cluster, or as a Cloud Cluster

Rubrik Security Cloud (RSC)

  • A SaaS management plane that all CDM clusters register/report into
  • Where you create SLA Domains, monitor jobs, manage archival locations, and do most day-to-day admin (current direction — replacing per-cluster local UI for most tasks)

Mental model: CDM is the engine in each cluster; RSC is the single dashboard you drive all your engines from.

01 — Architecture

Cluster anatomy: Brik & nodes

A Brik is a physical chassis containing multiple Rubrik nodes. Nodes work together to form a Rubrik cluster.

  • Each node is made up of storage, network, and compute resources — it's a self-contained mini-server, not a passive disk shelf
  • A Brik chassis commonly ships with 3 or 4 nodes — a fourth node can be added later to a 3-node chassis
  • A production on-prem cluster requires a minimum of 4 physical nodes
  • The cluster is linearly scalable — adding capacity and I/O performance is just a matter of adding more nodes (or more Briks)

This is the "own storage" part of your question — Rubrik's storage isn't external; each node contributes its local disks to a cluster-wide distributed filesystem (covered next).

01 — Architecture

Atlas, Cerebro & how data is actually stored

Two internal components are worth knowing by name — they come up constantly in Rubrik docs and in interviews:

Atlas — the distributed filesystem

Atlas is Rubrik's scale-out filesystem. When CDM ingests data from your environment, Atlas writes both the data and its metadata using erasure coding, spread across disks, nodes, and clusters. This is the storage layer underneath every backup.

Cerebro — the orchestration brain

Cerebro is the distributed, intelligent system that controls scheduling and data movement across the cluster — it's what decides when backups run, where data lands, and coordinates replication/archival jobs.

Self-healing & resilience features

  • Erasure coding across disks, nodes, and clusters means individual disk or node failures don't cause data loss — Atlas rebuilds automatically
  • Self-heal time is typically under an hour for a failed disk/node
  • Copysets (introduced in CDM 5.1) intelligently distribute metadata copies across nodes — roughly a 50x reduction in the risk of data loss if two nodes fail simultaneously, compared to clusters without Copysets
  • Zero-byte clones — Rubrik creates multiple logical copies from a single "golden image" without consuming additional storage, enabling instant recovery / Live Mount without a storage penalty
Why this matters for "own storage"

There's no separate SAN/NAS to provision for backup targets. The cluster's nodes are the storage — capacity and resilience scale together as you add nodes, and Atlas/Cerebro handle placement and protection transparently.

01 — Architecture

HA & resilience model

"HA" in Rubrik isn't a separate bolt-on feature like an on-prem ONTAP HA pair — it's built into the cluster's distributed design:

  • With a minimum of 4 nodes, the cluster tolerates node failures without downtime — erasure coding ensures data remains accessible from surviving nodes
  • Each node carries storage + compute + network, so losing one node doesn't take a "controller" offline the way it would in an active/passive array
  • The cluster continues accepting backup jobs and serving restores during a single-node (or, in larger clusters, multi-node) failure, while Atlas rebuilds redundancy in the background

Network-wise, each node typically has two bonded network interfaces (commonly referred to as bond0/bond1), each supporting active-passive failover — one network is often used for management traffic and the other for backup data traffic, which can be segregated onto separate VLANs after the cluster is bootstrapped and registered.

Encryption note

If your cluster supports encryption of data at rest, CDM enables it by default during initial configuration. Disabling it afterward requires a full cluster reset (which deletes all stored data) — so this is a decision to make correctly at bootstrap time, not later.

02 — Onboarding

Deployment options

Before bootstrapping, you choose the form factor. This maps directly to "own storage" vs. cloud:

Form factorWhere it runsStorage model
Physical appliance (Brik) Your datacenter — purpose-built hardware (e.g. Cisco UCS C-Series) Local disks per node, pooled via Atlas — this is "own storage"
Virtual Cluster VMware/Hyper-V in your datacenter VMDK/VHD-backed virtual disks per node
Cloud Cluster AWS, Azure, or GCP — runs CDM software on cloud VM instances in a VPC/VNet Cloud block storage (EBS/Managed Disk/Persistent Disk) per node
Cloud Cluster Elastic Storage (ES) Same clouds, dense node configuration only Decouples compute scaling from storage scaling — storage backed by object storage

Cloud Cluster supports standard or dense node configurations (not mixed); ES only supports dense nodes. Dense nodes have higher resource requirements but are recommended for full feature support.

02 — Onboarding

Bootstrap & cluster setup — the actual first steps

"Bootstrap" is the process that takes a set of freshly-installed/powered-on nodes and turns them into a working cluster. This is genuinely step 1 of onboarding.

  1. Physical/network prerequisites Cable and power on the nodes. Each node needs an IP from the same subnet. Verify IPv6 connectivity is enabled at the switch — the initial onboarding UI is accessed via a node's IPv6 address.
  2. Access the onboarding UI From a browser, access a Rubrik cluster node using its IPv6 address (found on a sticker on the node, or via ifconfig from the Rubrik CLI). The onboarding page appears — click Start Setup.
  3. Node discovery The guided configuration wizard starts at a "Nodes Discover" step, automatically discovering all nodes that belong to this cluster.
  4. Cluster bootstrap (CLI alternative) Bootstrap can also be triggered via CLI: cluster bootstrap (optionally --skipIpmiSetup if you don't want CDM managing IPMI/out-of-band management — virtual platforms don't support IPMI configuration anyway).
  5. User & cluster configuration During bootstrap you provide: an admin email (for notifications/alerts), an admin password (minimum 8 characters), and a cluster name (how this cluster identifies itself in the UI and to other Rubrik clusters).
  6. NTP configuration Time sync is configured during bootstrap — verify NTP servers are reachable (ping test) before proceeding; replace the default pool with your own NTP server IPs if the defaults aren't reachable.
  7. Bootstrap progress The CLI/UI displays progress as the cluster initializes its filesystem (Atlas) and internal services. When complete, the cluster is ready to register. 
cluster bootstrap
# (or, to skip IPMI configuration on physical nodes)
cluster bootstrap --skipIpmiSetup

02 — Onboarding

Registering with Rubrik Security Cloud

Once bootstrap completes, the cluster is functional but unregistered. Registration connects the cluster to Rubrik Security Cloud (RSC) — the SaaS management plane.

  1. Sign in to the bootstrap-complete page Enter the admin username/password you set during bootstrap.
  2. Provide your RSC domain URL This is your organization's RSC account URL — found in the Rubrik Welcome email, or via the Rubrik Support portal (My Products tab) if you don't have it handy.
  3. Log in to RSC The Online Cluster Registration dialog appears.
  4. Select service type
    • Rubrik Security Cloud (SaaS) — manage all data protection through RSC (the current recommended/default direction)
    • CDM (Software) — manage this cluster locally instead
Offline registration option

If the cluster has no internet access for online registration, use offline registration: retrieve a registration key (as a JSON file) from the cluster, then open the Welcome email's registration link (or navigate to rubrik_account_name.<domain>/offlineregistration) in a separate browser session to complete pairing.

02 — Onboarding

Adding hosts & the Rubrik Backup Service (RBS)

With the cluster registered, the next onboarding step is connecting it to the things you want to protect. There are two broad protection styles:

Agentless (hypervisor-integrated)

  • VMware vSphere VMs via VADP (vSphere Storage APIs for Data Protection)
  • Rubrik requests/retrieves VM data directly from ESXi — no agent inside the guest OS
  • Backup processing is offloaded to ESXi hosts + the Rubrik cluster

Agent-based (Rubrik Backup Service / RBS)

  • For physical/virtual Windows, Linux, Unix, AIX, Solaris hosts, filesets, and applications (SQL Server, Oracle via RMAN, etc.)
  • RBS installed as a service/daemon on the host
  • Enables app-aware backups, volume-level protection, and granular SLA assignment

Adding a fileset host (typical flow)

  1. Install RBS on the target host The RBS connector is downloadable from RSC or the cluster UI — RSC also offers to download and install RBS automatically while adding a host that doesn't have it yet. RBS runs as a Windows service (or daemon on Linux/Unix) and requires no reboot to install.
  2. Add the host to the cluster In RSC: Inventory → select the fileset card matching the host's OS type → Add Hosts.
  3. Discovery Once RBS is installed and the host is added, application instances are auto-discovered — e.g. for SQL Server, instances and databases appear automatically without further configuration.
  4. Create a fileset (for file-level protection) Pair a fileset definition (include/exclude rules for folders and file types) with the host or NAS share.
  5. Assign an SLA Domain Apply protection by assigning an SLA Domain to the host, fileset, instance, database, or VM — covered next.

For VMware environments, connecting CDM to vCenter (or a Hyper-V Manager for Hyper-V hosts) surfaces the full VM inventory directly — no RBS install needed per-VM unless you also want application-aware (e.g. in-guest SQL) protection.

02 — Onboarding

SLA Domains — the policy model

This is the core conceptual shift from traditional backup software: Rubrik replaces the "imperative" model (manually define backup jobs, schedules, media agents, retention rules per job) with a declarative policy model.

An SLA Domain defines, in one object:

  • Snapshot frequency — how often backups are taken (e.g. hourly, daily)
  • Retention — how long each snapshot is kept
  • Archival policy (optional, up to two per SLA Domain) — where and when snapshots get tiered to an archival location (cloud, NFS, tape, etc.)
  • Replication policy (optional) — whether/where snapshots replicate to another Rubrik cluster

Assigning protection

You assign an SLA Domain to an object — and that's the whole "job definition." No backup windows to schedule, no media agents to pick.

Auto-protection (inheritance)

SLA Domains can auto-assign protection hierarchically:

  • Assign an SLA Domain to a top-level object (e.g. a vCenter Server, a Database Server, a folder)
  • Any new VM/database/object created under that parent automatically inherits the SLA Domain
  • Moving an object into a protected hierarchy also applies inherited protection
Why this matters for onboarding at scale

Instead of remembering to add every new VM to a backup job, you assign SLA Domains at the vCenter/folder/cluster level once — new VMs are protected automatically from the moment they're created, with zero additional admin action.

Remote Domains (multi-cluster visibility)

If you're using replication between clusters, SLA Domains used for replication are visible on the destination cluster's Remote Domains page — the only setting modifiable remotely is the archival policy, to avoid configuration drift across sites.

02 — Onboarding

How a backup actually runs, end to end

SLA Domain triggers schedule
→
Cerebro schedules the job
→
Snapshot taken (VADP / RBS / RMAN)
→
Data ingested & deduped
→
Atlas writes data + metadata (erasure coded)
→
Retention/archival rules applied per SLA

For a VMware VM

Rubrik uses VADP to request a snapshot from ESXi — no agent inside the VM. The changed-block data since the last snapshot is read and ingested into the cluster, deduplicated against existing data, and written via Atlas. The VM's own snapshot is then released on the ESXi side (Rubrik doesn't leave its own snapshots accumulating on the datastore).

For a SQL Server database (via RBS)

RBS coordinates with the SQL VSS writer (or operates in crash-consistent mode if configured) to produce an application-consistent backup, plus periodic log backups per the SLA Domain — enabling point-in-time recovery between full snapshots.

For a fileset

RBS walks the configured include/exclude paths, identifies changed files since the last backup, and ingests only the deltas.

In all cases, the result lands in the same place conceptually: a new, space-efficient, erasure-coded snapshot in Atlas, tracked by Cerebro, governed by the retention/archival rules in the assigned SLA Domain.

02 — Onboarding

Restore & Live Mount

Because of zero-byte clones and zero-copy recovery, Rubrik's restore model is fast by design:

  • Live Mount — instantly mount a snapshot (a VM, a database, a fileset/share) as a usable, writable clone without copying data first. Useful for test/dev, ransomware investigation, or quick file-level recovery without a full restore
  • Export/Restore — write the snapshot data back to the original or a new location (e.g. restore a VM to a different datastore, restore files to an original path)
  • Granular recovery — for SQL/Oracle, recover individual databases to a specific point in time using full snapshot + log backups, without restoring the entire VM
  • Recovery from archive — for snapshots already tiered to cloud archival, Rubrik can recover at the snapshot or file level selectively, without downloading the entire archived dataset (reducing egress costs)
03 — To the Cloud

Cloud Cluster — running CDM in AWS / Azure / GCP

This is the most literal answer to "moved the data to cloud": Rubrik CDM itself can run as a cluster of VM instances inside a cloud VPC/VNet — same software, same Atlas/ Cerebro architecture, just on cloud compute and cloud block storage instead of physical Briks.

AspectOn-prem BrikCloud Cluster
Compute Physical node (UCS C-Series etc.) Cloud VM instance (EC2 / Azure VM / GCE)
Storage Local disks per node, Atlas across them Cloud block storage (EBS / Managed Disk / Persistent Disk) per node, Atlas across them
Minimum nodes 4 (production) Same minimums apply for standard clusters
Network Physical switches, VLANs VPC/VNet security groups — inbound 22/443, outbound allowed

Setup flow (conceptually identical to on-prem)

  1. Network prep Create a security group/NSG allowing all outbound traffic and inbound on ports 22 and 443.
  2. Instantiate nodes Launch the Cloud Cluster instances from the cloud marketplace image (AMI for AWS, equivalent for Azure/GCP) — minimum node count per the standard/dense node requirements for that cloud.
  3. System setup / bootstrap Open an SSH (or console) connection to one of the nodes and run the system setup script — same cluster bootstrap flow as on-prem: NTP check, admin email/password, cluster name.
  4. Log in to CDM web UI First login uses the admin account created during bootstrap.
  5. Node entitlement Online entitlement via Rubrik CLI ties the cloud nodes to your Rubrik account/license; offline entitlement is available for nodes without internet access.
  6. Prepare hosts for protection Same as on-prem: obtain and install the RBS connector on hosts you want to protect — which can include workloads running in the cloud (cloud-native VMs) alongside on-prem hosts reachable via VPN/peering.

Standard vs Dense nodes

Standard and dense node configurations have different minimum resource requirements per cloud. Standard nodes have lower requirements but don't support all CDM features; dense nodes have higher requirements but are recommended for full feature parity with on-prem. Cloud Cluster Elastic Storage (ES) supports dense nodes only, and decouples storage scaling from compute by backing capacity with object storage.

"Own storage" → cloud, the direct path

A Cloud Cluster is functionally a second Rubrik cluster — your on-prem cluster can replicate SLA-protected snapshots to it, just like replicating to a second on-prem cluster at a DR site. This is the most common "moved to cloud" pattern: on-prem cluster stays the primary backup target, Cloud Cluster becomes the DR/secondary copy.

03 — To the Cloud

Archival & Rubrik Cloud Vault

The other "moved to cloud" path — and the more common one for cost-driven long-term retention — is archival: tiering older snapshots from the Rubrik cluster's local storage out to object storage, governed entirely by the SLA Domain's archival policy.

Supported archival locations

  • Rubrik Cloud Vault — Rubrik-managed cloud storage, hosted in AWS or Azure
  • Microsoft Azure (Blob storage, customer-owned)
  • Amazon S3
  • Google Cloud Platform
  • NFS Shares
  • Object Store (S3-compatible) — for on-prem object storage like StorageGRID
  • Tape

How it's configured

  1. Create an archival location In RSC, define the target (e.g. a Rubrik Cloud Vault location): select the cluster(s) it applies to, give it a name, choose a storage tier (Backup or Archive), and select a region.
  2. Add it to an SLA Domain An SLA Domain can include up to two archival policies, each specifying which archival location to use and the minimum time after backup before data is archived.
  3. Assign the SLA Domain Once assigned to a workload, RSC begins archiving snapshots to the archival location based on the SLA's frequency/timing rules — fully automatic from here.

Archival behaviors worth knowing

TermWhat it means
Instant ArchiveArchive a snapshot to the archival location immediately, outside the normal SLA schedule
Instant TieringSupported for Rubrik Cloud Vault Archive-tier locations — tier data as soon as it's eligible
Tier existing snapshotsRapidly tier already-existing local snapshots to the Archive tier when configuring the policy
Immutable (container-level)Optional setting for Rubrik Cloud Vault Backup-tier locations — protects archived data from deletion/modification
Bypass proxyRoute archival traffic directly, skipping the system proxy, if configured
Rubrik Cloud Vault feature limits

Rubrik Cloud Vault (Archive-tier) locations do not support Consolidation, Smart Tiering, or CloudOn — these features are available with customer-owned Azure/AWS/GCP archival locations instead. Check feature support against your chosen archival target before designing retention policies around it.

Recovery from archive

Archived data doesn't need to be fully rehydrated to restore from — Rubrik supports recovering at the snapshot or file level selectively from archive, avoiding the cost and time of downloading an entire archived dataset just to pull one file.

03 — To the Cloud

Putting it together — a realistic rollout

A typical end-to-end onboarding that matches "HA + own storage, then moved data to cloud" looks like this:

  1. Deploy the on-prem cluster Rack 4+ nodes (1+ Briks), cable bond0/bond1 networks, verify IPv6 at the switch.
  2. Bootstrap cluster bootstrap — set admin credentials, cluster name, NTP, encryption-at-rest decision.
  3. Register with RSC Online or offline registration, choose RSC (SaaS) management.
  4. Onboard workloads Connect vCenter/Hyper-V for VM protection (VADP, agentless); install RBS on physical/VM hosts needing app-aware or fileset protection.
  5. Define SLA Domains Create policies (e.g. "Gold: hourly snapshots, 30-day retention"); assign at the vCenter/folder level for auto-protection of new VMs.
  6. Validate backups & test restores Confirm jobs run per SLA; test Live Mount for a sample VM and a sample database.
  7. Extend to cloud — pick one or both:
    • Replication to a Cloud Cluster (AWS/Azure/GCP) for DR — a second full Rubrik cluster running in the cloud
    • Archival to Rubrik Cloud Vault or customer-owned object storage for long-term, cost-efficient retention of older snapshots
  8. Add the archival/replication policy to existing SLA Domains No new jobs to create — the existing protected workloads automatically start sending data to the cloud target per the updated policy.
The throughline

Every stage — local backup, replication to a cloud cluster, archival to object storage — is driven by the same SLA Domain object. Onboarding a new workload to "cloud protection" is just assigning it the SLA Domain that already has the cloud policy attached, not building a parallel cloud backup pipeline.

ONTAP Export Policy
Deep Dive

A complete walkthrough of export policies — how rorule and rwrule control access, what each auth method actually means on the wire, real ONTAP CLI examples, client-side mount commands, and troubleshooting steps.

ONTAP 9.x NFSv3 / NFSv4 AUTH_SYS · Kerberos Security
Section 01 · Concepts

What is an Export Policy?

An export policy is a named container of one or more rules that controls which NFS (and CIFS) clients are allowed to access a FlexVol volume or qtree, and with what level of access. Every SVM volume must have an export policy assigned — ONTAP will not serve NFS data without one.

The policy itself is just a name. The work is done by the rules inside it. Each rule matches a client (by IP, subnet, hostname, or netgroup) and specifies three things: which protocol versions are allowed, which auth methods give read-only access (rorule), and which auth methods give read-write access (rwrule).

Architecture — how the layers stack

💡
Key Design Principle One policy can be shared across many volumes. Changing a rule in the policy affects every volume it is assigned to instantly. Use this to manage access at scale — one policy per application tier, not per volume.

Rule evaluation order

Rules are evaluated top-down by rule index (1, 2, 3 …). The first rule whose clientmatch field matches the client's IP is the only rule applied — ONTAP stops evaluating. If no rule matches, access is denied. This means rule order matters significantly.

Section 02 · Access Control

rorule & rwrule — exactly what they control

This is the most misunderstood part of export policy. rorule and rwrule do not control filesystem permissions. They control which authentication methods are accepted before granting read-only or read-write access. Think of them as two security checkpoints that each ask: "what credential did you arrive with, and is that credential type on our approved list?"

Decision flow

⚠️
Critical distinction Export policy grants or restricts access at the network/auth level. Even after rwrule grants read-write, the client is still subject to Unix mode bits and ACLs on individual files. Both layers work in series — export policy is the network gate, file permissions are the per-file gate.

Auth method values — what each one means on the wire

No encryption

sys

AUTH_SYS (UNIX auth). The client sends a plain UID + GID list in the RPC header. No cryptographic verification — the server trusts whatever the client claims. Default for most NFSv3 environments. Works everywhere but provides zero identity assurance.

Identity verified

krb5

Kerberos authentication. The client proves its identity with a Kerberos ticket from a KDC. ONTAP verifies the ticket — so you know who the client is. Data on the wire is still unencrypted and unsigned. Identity verified, data unprotected.

Integrity checked

krb5i

Kerberos + integrity. Every RPC message is signed with a checksum. If anyone tampers with data in transit, the checksum fails and the operation is rejected. Data is readable on the wire but tampering is detected. Good middle ground for internal networks.

Encrypted

krb5p

Kerberos + privacy. Full encryption of every RPC payload. Strongest option — identity verified, data signed, data encrypted. Highest CPU overhead on both client and storage. Required for compliance mandates or highly sensitive data.

Wildcard

any

Accepts sys, krb5, krb5i, krb5p, and even unauthenticated requests. Essentially disables auth checking for that rule. Commonly used for rorule in open lab environments, then tighten rwrule to sys or krb5.

No auth

none / never

none — allows unauthenticated access. Requests are mapped to the anon UID (default 65534). Used deliberately for anonymous read shares.
never — absolute block, no exceptions. Use in rwrule to make a share permanently read-only.

Section 04 · ONTAP CLI

ONTAP CLI — Create, Assign & Verify

Create an export policy and add rules

Step 1 — Create the policy

ONTAP CLI 
# Create a new export policy on the SVM
vserver export-policy create \
  -vserver svm1 \
  -policyname nfs-prod-policy

# Verify it was created
vserver export-policy show -vserver svm1

Step 2 — Add rules

ONTAP CLI — Rule: full RW for subnet 10.10.10.0/24 
vserver export-policy rule create \
  -vserver        svm1 \
  -policyname     nfs-prod-policy \
  -ruleindex      1 \
  -clientmatch    10.10.10.0/24 \
  -protocol       nfs3,nfs4 \
  -rorule         sys \
  -rwrule         sys \
  -superuser      sys \
  -allow-suid     true
ONTAP CLI — Rule: read-only for DR subnet 10.20.0.0/16 
vserver export-policy rule create \
  -vserver        svm1 \
  -policyname     nfs-prod-policy \
  -ruleindex      2 \
  -clientmatch    10.20.0.0/16 \
  -protocol       nfs3 \
  -rorule         sys \
  -rwrule         never \
  -superuser      none
ONTAP CLI — Rule: Kerberos-only RW for secure subnet 
vserver export-policy rule create \
  -vserver        svm1 \
  -policyname     nfs-prod-policy \
  -ruleindex      3 \
  -clientmatch    10.30.0.0/24 \
  -protocol       nfs4 \
  -rorule         krb5 \
  -rwrule         krb5p \
  -superuser      krb5

Assign policy to a volume

ONTAP CLI 
# Assign during volume creation
volume create \
  -vserver        svm1 \
  -volume         vol-app01 \
  -aggregate      aggr1_node1 \
  -size           100g \
  -junction-path  /vol-app01 \
  -export-policy  nfs-prod-policy

# Change policy on an existing volume
volume modify \
  -vserver        svm1 \
  -volume         vol-app01 \
  -export-policy  nfs-prod-policy

Verify and inspect

ONTAP CLI — Show all rules in a policy 
vserver export-policy rule show \
  -vserver    svm1 \
  -policyname nfs-prod-policy

Pro tip vserver export-policy check-access is your best friend when troubleshooting. It simulates the full rule evaluation for any client IP + auth method + protocol combination without needing a live client. Use it before raising a support case.
Section 05 · Linux Client

Client-Side Commands

AUTH_SYS (sys) — standard NFS mount

Linux Client — NFSv3 mount with sys auth 
# Mount NFSv3 with AUTH_SYS (default, no extra options needed)
mount -t nfs \
  -o rw,nfsvers=3,sec=sys \
  10.10.10.20:/vol-app01 \
  /mnt/app01

# Verify mount is active
mount | grep app01
df -h /mnt/app01

# Test write access
touch /mnt/app01/test-write.txt && echo "RW working" || echo "DENIED"

# Test read
ls -la /mnt/app01/
Linux Client — NFSv4 mount with sys auth 
mount -t nfs4 \
  -o rw,nfsvers=4.1,sec=sys,proto=tcp \
  10.10.10.20:/vol-app01 \
  /mnt/app01
Linux Client — /etc/fstab persistent mount 
# Add to /etc/fstab for persistent mounts
10.10.10.20:/vol-app01  /mnt/app01  nfs  rw,nfsvers=3,sec=sys,_netdev  0  0
10.10.10.20:/vol-app02  /mnt/app02  nfs  rw,nfsvers=4.1,sec=sys,_netdev  0  0

# Mount all fstab entries
mount -a

Kerberos client — krb5 / krb5i / krb5p

Linux Client — Kerberos prerequisites 
# Install required packages (RHEL/Rocky)
dnf install -y nfs-utils krb5-workstation krb5-libs

# Obtain a Kerberos ticket first
kinit ramesh@EXAMPLE.COM

# Verify ticket is present
klist
# Credentials cache: API:...
# Default principal: ramesh@EXAMPLE.COM
#   Valid starting       Expires
#   06/11/2026 10:00:00  06/11/2026 20:00:00
Linux Client — Mount with krb5 / krb5i / krb5p 
# krb5 — Kerberos auth, unencrypted data
mount -t nfs4 \
  -o rw,nfsvers=4.1,sec=krb5 \
  10.10.10.20:/vol-secure \
  /mnt/secure

# krb5i — Kerberos auth + data integrity (checksum)
mount -t nfs4 \
  -o rw,nfsvers=4.1,sec=krb5i \
  10.10.10.20:/vol-secure \
  /mnt/secure

# krb5p — Kerberos auth + integrity + encryption (strongest)
mount -t nfs4 \
  -o rw,nfsvers=4.1,sec=krb5p \
  10.10.10.20:/vol-secure \
  /mnt/secure
Linux Client — Verify the sec= option active on mount 
# Show active mount options including security flavour
cat /proc/mounts | grep app01
# 10.10.10.20:/vol-app01 /mnt/app01 nfs4 rw,relatime,vers=4.1,rsize=65536,
#   wsize=65536,sec=sys ...

# Check negotiated security using nfsstat
nfsstat -m /mnt/app01

Test access scenarios from client

Linux Client — Reproduce a permission denied and diagnose 
# Attempt to write — will show EACCES if rwrule denies
touch /mnt/app01/testfile
# touch: cannot touch '/mnt/app01/testfile': Permission denied

# Check if it is export-policy denial (EACCES) vs file permission (EPERM)
# Export policy denial: write fails even to a 777 directory
# File permission denial: write fails only on specific files/dirs

# Create a 777 test dir to isolate the layer
mkdir /mnt/app01/testdir 2>&1
# If mkdir fails → export policy issue (rwrule denying)
# If mkdir succeeds but touch in existing dir fails → file ACL issue
Section 06 · Troubleshooting

Troubleshooting Export Policy Issues

Step 1 — Run check-access on ONTAP first

ONTAP CLI 
# Simulate exactly what the client sees
vserver export-policy check-access \
  -vserver               svm1 \
  -client-ip             10.10.10.55 \
  -volume                vol-app01 \
  -authentication-method sys \
  -protocol              nfs3 \
  -access-type           write

# Run for read too
vserver export-policy check-access \
  -vserver               svm1 \
  -client-ip             10.10.10.55 \
  -volume                vol-app01 \
  -authentication-method sys \
  -protocol              nfs3 \
  -access-type           read

Step 2 — Check clientmatch is correct

ONTAP CLI 
# Show all rules with full detail
vserver export-policy rule show \
  -vserver    svm1 \
  -policyname nfs-prod-policy \
  -instance

# Common mistake: client IP 10.10.10.55 but clientmatch is 10.10.0.0/16
# → wrong subnet, rule won't match

Step 3 — Check the volume has the right policy

ONTAP CLI 
volume show -vserver svm1 -volume vol-app01 -fields export-policy,junction-path

# Also check qtrees if exports at qtree level
qtree show -vserver svm1 -fields export-policy

Step 4 — Enable NFS trace on the SVM (advanced)

ONTAP CLI — Advanced privilege 
set -privilege advanced

# Enable NFS access trace for a specific client IP
vserver nfs access-cache config modify \
  -vserver  svm1 \
  -is-nfs-access-logging-enabled  true

# Watch event log while client tries to mount
event log show -severity ERROR -message-name nfs.export*

set -privilege admin

Common issues and fixes

SymptomLikely causeFix
Client gets Permission denied on mount rorule does not match client's auth method, or clientmatch doesn't cover client IP Run check-access, verify clientmatch subnet and rorule value
Client can read but not write rwrule is never, or rwrule requires krb5 but client uses sys Check rwrule value; change to sys or match client's auth method
Root user gets Nobody mapping -superuser is set to none — root is squashed to anon UID Set -superuser sys or -superuser krb5 to match the environment
Kerberos mount fails immediately Client has no valid Kerberos ticket (klist shows no credentials) Run kinit user@REALM on the client first
New volume is not accessible Volume was created with the default export policy which has no rules Assign the correct policy: volume modify -export-policy <name>
Rule exists but client still denied A higher-index rule is matching first due to wrong rule order Check rule indexes; reorder with vserver export-policy rule setindex
🚨
Default policy has no rules Every SVM is created with a policy called default. This policy has no rules — meaning no client can access any volume assigned to it. Newly created volumes are assigned to default by default. Always explicitly assign your policy when creating a volume.
Section 07 · Interview Q&A

Interview & Exam Questions

  • Q01 What is the difference between rorule and rwrule in ONTAP export policy? ▾
    rorule controls which authentication methods are accepted to grant any access at all (read-only as a minimum). If the client's auth method doesn't match rorule, the mount is denied with EACCES. rwrule is only evaluated after rorule passes — it controls which auth methods are permitted to write. If the client passes rorule but fails rwrule, it gets read-only access. rorule is the front door; rwrule is the inner door.
  • Q02 A client can read but cannot write even though rwrule is set to sys. What do you check? ▾
    First run vserver export-policy check-access with access-type write to confirm it is an export policy issue and not a file permission issue. Check that the client IP matches the correct clientmatch — a higher-index rule with rwrule=never might be matching first. Verify the volume has the correct policy assigned with volume show -fields export-policy. Also check the -superuser parameter — if the client is mounting as root and superuser is set to none, root is squashed to nobody and denied writes.
  • Q03 What does setting rwrule to "never" achieve? How is that different from not having a rule? ▾
    Setting rwrule to never explicitly means no client, regardless of auth method, will ever be granted write access through this rule. The share is permanently read-only. Not having a rule at all means no clientmatch exists — the client is denied entirely (both read and write) because no rule matched. never is a deliberate read-only setting; no rule is a default deny.
  • Q04 What is the difference between krb5, krb5i and krb5p? ▾
    krb5: Kerberos authentication only. Identity is verified via a Kerberos ticket. Data on the wire is unencrypted and unsigned. krb5i: Adds integrity — every RPC message carries a checksum. Tampering in transit is detected and the operation rejected. Data is still readable but modification is detected. krb5p: Adds full privacy — every RPC payload is encrypted. Highest security, highest CPU overhead. Required for compliance mandates. The suffix i = integrity, p = privacy.
  • Q05 What is the difference between auth method "none" and "never"? ▾
    none means accept requests with no authentication — unauthenticated clients are mapped to the anon UID (default 65534 / nobody). It is an explicit grant to anonymous clients. never means no client regardless of what credential they present will be granted access through this rule. none in rorule = anonymous read allowed. never in rwrule = no client can ever write.
  • Q06 How does ONTAP evaluate multiple rules in an export policy? ▾
    Rules are evaluated in order by their rule index (1, 2, 3…). The first rule whose clientmatch field matches the client's IP address is the rule applied — ONTAP stops evaluating immediately after the first match. If no rule matches the client's IP, access is denied. This means rule order is critical: more specific rules (individual IPs or smaller subnets) should have lower index numbers than broad rules (large subnets or 0.0.0.0/0 catch-alls).
  • Q07 A newly created volume is inaccessible to all NFS clients. What is the most likely cause? ▾
    The most common cause is that the volume was created with the default export policy, which exists on every SVM but contains no rules. No rules means no clientmatch ever succeeds, so all clients are denied. Fix: run volume modify -vserver <svm> -volume <vol> -export-policy <policy-with-rules> to assign a policy that has appropriate rules for your clients.
  • Q08 How do you test export policy access without a live NFS client? ▾
    Use vserver export-policy check-access with parameters: -client-ip, -volume, -authentication-method, -protocol, and -access-type. This simulates the full rule evaluation and tells you exactly which rule index matched and what access level was granted. It is the fastest diagnostic tool and should always be the first step before any other troubleshooting.

Hotspot

A hotspot occurs when I/O load concentrates on a single disk, aggregate, or volume — creating a performance bottleneck while surrounding resources sit idle. This tutorial covers what causes hotspots, how ONTAP measures and reports them, CLI commands to detect and confirm them, and every resolution method available.

Section 01 · Concepts

What is a Hotspot?

A hotspot in NetApp ONTAP is an I/O imbalance condition where one storage component — a disk, a RAID group, an aggregate, or a volume — receives a disproportionately high workload relative to its neighbours. The result is that the hot component becomes a bottleneck: latency climbs, queue depth grows, and application performance degrades — while the surrounding components are underutilised.

Hotspots are particularly common in spinning disk (HDD) environments because each disk has a fixed rotational speed and a single read/write head. An SSD can service many more IOPS simultaneously, making disk-level hotspots less common on all-flash arrays — but aggregate, volume, and LUN-level hotspots still occur on AFF platforms when workloads are unevenly distributed.

Types of hotspots

💽

Disk-level hotspot

One physical disk — usually a data disk in a RAID group — receives the majority of I/O. Common when a single heavily accessed file or LUN maps to blocks on that disk. The disk busy% counter spikes while peers sit idle. Most impactful on HDDs; less so on SSDs.

🗄️

Aggregate-level hotspot

One aggregate on a node receives far more workload than other aggregates on the same or peer nodes. Usually caused by placing too many high-I/O volumes on a single aggregate. Results in high aggregate utilisation and elevated latency for all volumes on that aggregate.

📁

Volume-level hotspot

A single FlexVol receives disproportionate I/O. Can happen with application databases, NFS home directories for power users, or shared volumes with many concurrent writers. Competes for WAFL buffer cache and CPU cycles on the owning node.

🔲

LUN-level hotspot

Specific in SAN environments. A single LUN within a volume carries the bulk of IOPS — often a database log LUN or a VM swap disk. Even if the volume is balanced, this LUN's workload can dominate the node's processing queue.

⚡

Node CPU hotspot

One node in an HA pair or cluster handles the majority of client connections and I/O processing, while its partner node runs light. Results in high CPU utilisation, increased WAFL scheduling latency, and potential NFSv4 lease renewal issues.

Root causes

CauseWhy it creates a hotspotTypical scenario
Uneven volume placement All high-IOPS volumes placed on same aggregate by default or manually DBA puts all Oracle data, redo log, and temp volumes on aggr1
Sequential I/O pattern on HDD Sequential writes concentrate on the last-written disk in a RAID stripe Large file copy or backup job hitting a single data volume
Small file / metadata storm Millions of small file operations flood WAFL metadata blocks, which map to specific disks NFS home directory server with thousands of users at login time
No QoS limits A runaway workload monopolises node resources with no ceiling Dev/test VM doing unthrottled random 4K writes on a shared cluster
Unbalanced HA pair All aggregates are on one node; partner has minimal load Node failover never reversed — all workload stays on one node indefinitely
Flash Pool cache misses SSD cache repeatedly misses, driving all I/O to spinning disks Workload with low temporal locality — each I/O hits a different LBA
Section 02 · Measurement

How ONTAP Measures Hotspots

Key counters to watch

CounterWhat it measuresWhere to find it
disk.busy Percentage of time the disk is busy servicing I/O (0–100%). The primary disk hotspot indicator. statistics show, sysstat -d
disk.avg_latency Average I/O latency on the disk in microseconds. Rises sharply when busy% is high on HDD. statistics show -object disk
aggregate.total_transfers Total I/O operations per second across the aggregate. Compare across aggregates on the same node. statistics show -object aggregate
volume.avg_latency Average client-visible latency for operations on the volume. Reflects combined disk + WAFL + network overhead. statistics show -object volume
node.cpu_busy Overall CPU utilisation on the node. Consistently above 80% points to a node hotspot. sysstat -x 1
wafl.total_cp_msecs Time spent in Consistency Point write cycles. Elevated CP time indicates WAFL is under pressure. statistics show -object wafl

Thresholds — when to act

CounterNormalWatchHotspotCritical
disk.busy (HDD) 0–40% 40–60% 60–80% >80%
disk.busy (SSD) 0–60% 60–75% 75–90% >90%
Aggregate utilisation <50% 50–70% 70–85% >85%
Node CPU busy <50% 50–70% 70–85% >85%
Volume latency (NFS) <2 ms 2–5 ms 5–20 ms >20 ms
ℹ️
Relative comparison matters more than absolutes A disk at 70% busy% is only a hotspot if its peers are at 20%. If all disks in the RAID group are at 65–70%, the aggregate is simply under high load — a different problem. Always compare the suspect component against its peers, not just against a fixed threshold.
Section 03 · CLI Detection

CLI Detection Commands

Disk-level hotspot detection

ONTAP CLI — Check disk busy% across all disks 
# Show disk busy% — run several times 5–10 seconds apart
statistics show \
  -object  disk \
  -counter disk_busy_pct,avg_latency \
  -raw

# Simpler view — disk performance summary
storage disk show -fields disk,type,rpm,avg-latency,iops

# Advanced privilege — per-disk I/O detail
set -privilege advanced
statistics show -object disk -instance 0a.00.1   # specific disk
set -privilege admin
Sample output — disk 0a.00.2 is a hotspot 
Object: disk
Instance               disk_busy_pct    avg_latency (us)
--------------------   -------------    ----------------
0a.00.0                       22              540
0a.00.1                       18              480
0a.00.2                       91             8240   ← HOTSPOT
0a.00.3                       25              560
0a.00.4                       19              510
0a.00.5                       27              590
P (parity)                    30              620
ONTAP CLI — Real-time disk I/O with sysstat 
# -d flag shows per-disk breakdown, 1-second intervals
sysstat -d 1

# -x flag for extended node stats including CP activity
sysstat -x 1
ONTAP CLI — Which volume/file is generating the I/O 
# Show top volumes by IOPS on the node right now
statistics show \
  -object  volume \
  -counter total_ops,avg_latency \
  -sort-key total_ops

# Show top files by access count (advanced)
set -privilege advanced
statistics show -object file -counter total_ops
set -privilege admin

Aggregate-level hotspot detection

ONTAP CLI — Compare aggregate utilisation across nodes 
# Show aggregate I/O stats — look for imbalance between aggregates
statistics show \
  -object  aggregate \
  -counter total_transfers,user_reads,user_writes,avg_latency

# Show aggregate space and utilisation
aggr show \
  -fields aggregate,node,availsize,usedsize,percent-used,state

# Show how many volumes live on each aggregate
volume show -fields volume,aggregate,node,size \
  -state online | sort aggregate

Volume-level hotspot detection

ONTAP CLI — Top volumes by latency and IOPS 
# Show volume performance — sort by latency descending
statistics show \
  -object   volume \
  -counter  avg_latency,read_ops,write_ops,total_ops \
  -sort-key avg_latency \
  -order    descending

# QoS workload activity — see which workloads are busiest
qos statistics workload show

# Volume move eligibility — useful before planning rebalance
volume show -fields aggregate,node,total-footprint,size \
  -vserver svm1
ONTAP CLI — Node-level CPU and load 
# Show per-node CPU utilisation
statistics show \
  -object  node \
  -counter cpu_busy,processor_busy

# Node summary including CPU, ops, latency
sysstat -x 2 5   # 2-second intervals, 5 iterations

# See if one node has all the LIFs (clients stuck on one node)
network interface show \
  -fields lif,vserver,curr-node,curr-port,address
Section 04 · System Manager

Detecting via System Manager & Active IQ

System Manager (ONTAP 9.9+)

Navigate to Storage → Volumes and click the Activity Tracking tab on any volume. This shows a live IOPS and latency graph for the volume and identifies the top client IPs generating the most I/O. No CLI knowledge required — it is the fastest way to point a finger at a specific workload.

For aggregate-level view: Storage → Tiers shows each aggregate with a live utilisation bar and IOPS histogram. Hovering over a tier shows peak and average throughput.

Active IQ Unified Manager

Active IQ Unified Manager provides automated hotspot detection via its Performance Event engine. It compares each component's performance to a learned baseline and raises a Dynamic Threshold event when the component's latency or IOPS deviates significantly above its expected range. Key event types to look for:

EventMeaning
Volume Latency above Performance Threshold Volume latency exceeded the defined SLO or dynamic baseline.
Aggregate IOPS above Performance Threshold Aggregate is receiving more IOPS than its historical baseline suggests.
Node CPU Utilisation above Threshold Node CPU is consistently above the configured warning threshold.
Disk Performance Threshold Breached One or more disks in an aggregate have busy% above the defined limit.
💡
Quick detection sequence In practice: start with sysstat -x 1 to spot a hot node. If CPU is high, run qos statistics workload show to find the offending workload. Then run statistics show -object disk to confirm which disk is physically hot. That three-step sequence covers 90% of production hotspot investigations.
Section 05 · Resolution

Resolution Methods

Method 1 — Volume Move (rebalance across aggregates)

The most common and effective fix for aggregate-level hotspots. Move high-I/O volumes from the overloaded aggregate to a less-loaded one — on the same node or to the partner node. The move is non-disruptive; clients continue to access data throughout.

ONTAP CLI — Non-disruptive volume move 
# Check destination aggregate has enough space
aggr show -aggregate aggr2_node1 -fields availsize

# Start the volume move — non-disruptive by default
volume move start \
  -vserver           svm1 \
  -volume            vol-db-redo \
  -destination-aggregate aggr2_node1

# Monitor progress
volume move show

# Output shows phase: replicating → cutover → done
# Clients see no interruption — cutover takes < 1 second

# Move to partner node aggregate (load balance across nodes)
volume move start \
  -vserver           svm1 \
  -volume            vol-db-data \
  -destination-aggregate aggr1_node2
⚠️
Check LIF home port after node move If you move a volume to an aggregate on the partner node, client connections continue via LIF redirection — but with added network hop latency. After the move, consider also migrating the volume's serving LIF to the new home node using network interface migrate to restore optimal I/O path.

Method 2 — Flash Pool (add SSD cache to HDD aggregate)

If the hotspot is caused by a read-heavy workload on an HDD aggregate, adding SSD cache via Flash Pool can absorb the read I/O without moving volumes. Frequently accessed blocks are promoted to the SSD tier automatically.

ONTAP CLI — Add SSD cache to existing HDD aggregate 
# Check if Flash Pool is supported on the aggregate
aggr show -aggregate aggr1_node1 -fields type,is-flash-pool-eligible

# Add SSDs to create a Flash Pool
aggr add-disks \
  -aggregate   aggr1_node1 \
  -disk-type   SSD \
  -disklist    1.1.0,1.1.1,1.1.2,1.1.3

# Confirm Flash Pool is active
aggr show -aggregate aggr1_node1 -fields type,hybrid-enabled

Method 3 — QoS Policy (throttle the hot workload)

If the hotspot is caused by one runaway workload (a dev job, a backup, a poorly written query), apply a QoS maximum limit to that volume or LUN. This limits the IOPS or throughput it can consume, protecting other workloads sharing the same aggregate.

ONTAP CLI — Create and apply a QoS max policy 
# Create a QoS policy with a maximum ceiling
qos policy-group create \
  -policy-group  pg-limit-backup \
  -vserver       svm1 \
  -max-throughput 500IOPS

# Assign the QoS policy to the hot volume
volume modify \
  -vserver          svm1 \
  -volume           vol-backup \
  -qos-policy-group pg-limit-backup

# Verify the policy is active and being enforced
qos statistics workload show
qos statistics performance show

# Adaptive QoS — scales IOPS limit with volume size
qos adaptive-policy-group create \
  -policy-group       pg-adaptive-prod \
  -vserver            svm1 \
  -expected-iops      1000IOPS/TB \
  -peak-iops          5000IOPS/TB

Method 4 — FlexCache (edge caching for read hotspots)

If the hotspot is a read-heavy volume accessed by clients across multiple sites or from one specific location, deploy a FlexCache volume at the edge. Read I/O is served from the cache, dramatically reducing traffic to the origin and relieving the hotspot.

ONTAP CLI — Create a FlexCache for a hot volume 
# Create a FlexCache volume on the edge cluster
volume flexcache create \
  -vserver           svm-edge \
  -volume            vol-app01-cache \
  -size              50g \
  -aggr-list         aggr1_edge \
  -origin-vserver    svm-origin \
  -origin-volume     vol-app01

# Check cache hit rate — should climb above 90% for good workloads
statistics show \
  -object  flexcache \
  -counter cache_hits,cache_misses,hit_percent

Method 5 — LIF migration (fix node CPU hotspot)

ONTAP CLI — Migrate LIFs to rebalance client load across nodes 
# See current LIF distribution across nodes
network interface show -fields lif,curr-node,home-node,address

# Migrate a specific LIF to the partner node
network interface migrate \
  -vserver     svm1 \
  -lif         svm1-nfs-lif1 \
  -dest-node   node2 \
  -dest-port   e0c

# Auto-rebalance all LIFs across nodes (ONTAP 9.11+)
network interface rebalance -vserver svm1
Section 06 · Prevention

Prevention Best Practices

  • 1

    Spread volumes across aggregates intentionally

    Never place all production volumes on a single aggregate. For database environments, separate data, redo log, temp, and archive volumes across different aggregates — ideally on different nodes. Even if it feels like over-engineering on day one, it buys headroom as workloads grow.

  • 2

    Use Adaptive QoS as a safety net on shared clusters

    Apply adaptive QoS minimum/maximum policies to all volumes on a shared cluster. This prevents any single workload from monopolising aggregate or node resources. A 5000 IOPS/TB ceiling is a reasonable starting point that stops runaway jobs without impacting normal workloads.

  • 3

    Balance LIF placement at SVM creation time

    When creating SVMs and LIFs, distribute LIF home ports evenly across both nodes in an HA pair. Do not let all client-facing LIFs home on node1 — this concentrates all NFS/CIFS processing on a single node regardless of where volumes actually live.

  • 4

    Revert LIFs after a takeover/giveback

    After an HA failover and giveback, LIFs do not always migrate back automatically. Run network interface revert * after a giveback to return LIFs to their home nodes. Failing to do this leaves one node doing double the client work indefinitely.

  • 5

    Set Active IQ Unified Manager performance thresholds

    Configure automated threshold alerts in Active IQ Unified Manager so you are notified when disk busy% exceeds 70%, aggregate utilisation exceeds 75%, or volume latency exceeds your SLO. Proactive alerting means you fix hotspots before users complain.

  • 6

    Consider all-flash (AFF) for mixed-workload environments

    Disk-level hotspots are a fundamentally HDD problem. SSDs have no rotational latency and can service random I/O from many logical locations simultaneously, making disk-specific hotspots essentially non-existent. If disk hotspots are a recurring problem, the architectural answer is transitioning the affected aggregate to SSD.

Section 07 · Interview Q&A

Interview & Exam Questions

  • Q01 What is a hotspot in NetApp ONTAP and at what levels can it occur? ▾
    A hotspot is an I/O imbalance where one storage component receives disproportionately more workload than its peers. It can occur at five levels: disk level (one physical disk in a RAID group receiving most I/O), aggregate level (one aggregate on a node receiving most of the cluster's I/O), volume level (one FlexVol dominating WAFL and CPU resources), LUN level (one LUN inside a volume monopolising SAN I/O), and node level (one node in an HA pair handling all client connections and I/O processing).
  • Q02 Which CLI command gives you the fastest overview of a disk hotspot? ▾
    The fastest initial command is sysstat -d 1 which gives a per-disk I/O breakdown refreshed every second. It immediately shows if one disk is significantly busier than its peers. Follow it with statistics show -object disk -counter disk_busy_pct,avg_latency for point-in-time values you can compare side by side. The combination of a high busy% on one disk and much lower values on its RAID group peers confirms a disk hotspot.
  • Q03 What is the difference between a disk hotspot and an aggregate hotspot? How do you distinguish them? ▾
    A disk hotspot means one physical disk is disproportionately busy within its RAID group — you see one disk at 90% busy% while peers are at 20%. An aggregate hotspot means an entire aggregate is overloaded relative to other aggregates on the same node — all disks in the aggregate may be moderately busy, but the aggregate is absorbing far more IOPS than other aggregates. To distinguish: check statistics show -object disk first — if all disks in the aggregate are equally elevated, it is aggregate-level. If only one disk is hot while its RAID group peers are idle, it is disk-level.
  • Q04 How do you resolve an aggregate hotspot non-disruptively? ▾
    Use volume move start to migrate high-I/O volumes from the overloaded aggregate to a less-loaded aggregate — either on the same node or on the partner node. Volume move is completely non-disruptive: clients continue reading and writing throughout the operation. ONTAP copies data in the background, then performs a sub-second cutover. After the move, the source aggregate's I/O load drops and the destination aggregate absorbs the workload. If moving to the partner node's aggregate, also consider migrating the associated LIF to avoid cross-node I/O path overhead.
  • Q05 A dev team's backup job is causing a storage hotspot affecting production workloads. What is the fastest fix? ▾
    Apply a QoS maximum policy to the backup volume immediately. Create a policy group with a maximum IOPS or throughput ceiling — for example qos policy-group create -policy-group pg-backup-limit -max-throughput 200MB/s — and assign it to the backup volume with volume modify -qos-policy-group pg-backup-limit. This takes effect within seconds. The backup job continues but is throttled, and production workloads regain their share of the aggregate's capacity. Long-term, the backup volume should be moved to a dedicated aggregate or scheduled to run during off-peak hours.
  • Q06 Why are disk hotspots more common on HDD aggregates than AFF/SSD aggregates? ▾
    A spinning hard disk has a single read/write head and fixed rotational speed (typically 10K or 15K RPM). It can only service one I/O request at a time physically, and its throughput is constrained by seek time and rotational latency. If a specific LBA range is accessed repeatedly (e.g. a hot file or hot block), that disk's head is constantly seeking to the same track — driving busy% to near 100% while other disks sit idle. An SSD has no moving parts, no seek latency, and can service many parallel I/O requests from any logical block address simultaneously. While SSD IOPS can still be saturated, the absence of mechanical constraints eliminates the disk-specific concentration effect that makes HDD hotspots so severe.
  • Q07 After an HA failover and giveback, one node has much higher CPU than the other. Why? ▾
    During a failover, all LIFs from the failed node migrate to the surviving partner. After a giveback, the volumes return to the original node but the LIFs do not automatically revert unless configured to do so or manually triggered. This leaves all client connections still terminating on the partner node, which then forwards I/O to the original node across the cluster interconnect — double the processing work on the partner. The fix is to run network interface revert * after confirming giveback is complete, which returns all LIFs to their home nodes and restores balanced load across both nodes.
  • Q08 What is the difference between QoS maximum and QoS minimum in the context of hotspot management? ▾
    A QoS maximum is a ceiling — it limits a workload to a defined IOPS or throughput cap regardless of available resources. Use it to prevent a workload from creating a hotspot by consuming too much. A QoS minimum is a floor — it guarantees a workload receives at least a defined level of resources even when the system is under contention. Use it to protect critical workloads from being starved when a hotspot forms around them. In a shared cluster, best practice is to apply both: a minimum on production volumes to protect their SLO, and a maximum on dev/test/backup workloads to prevent them from creating hotspots for production.
